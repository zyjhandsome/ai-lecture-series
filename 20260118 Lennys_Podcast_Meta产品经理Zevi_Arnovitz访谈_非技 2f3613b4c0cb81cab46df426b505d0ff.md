## 完整AI工作流程详解

### 工作流程概览

Zevi 开发了一套完整的 AI 工作流程，使用 `/commands`（斜杠命令）实现自动化：

```
1. /create issue → 在 Linear 中创建问题
2. /explore → 探索和理解问题
3. /create plan → 创建实施计划
4. /execute → 执行代码
5. /review → 代码审查
6. /peer review → 同行评审（多模型交叉审查）
7. /update documentation → 更新文档
```

### 步骤1：创建问题（/create issue）

当在开发过程中遇到 bug、功能建议或改进想法时，使用此命令快速记录到 Linear：

> Zevi：“我在开发大型项目时，突然遇到 bug 或有了不想立即开发的想法。Claude 的主要目标是快速捕获我的想法并在 Linear 中创建问题。”
> 

**关键特点**：
- 使用 MCP（Model Context Protocol）技术连接 Linear
- 自动生成问题摘要（TLDR）、当前状态、预期结果和上下文
- 适合在开发过程中快速记录想法

### 步骤2：探索阶段（/explore）

探索阶段让 AI 深入理解要解决的问题：

> Zevi：“这是让 CTO 深入理解我们要解决的问题、代码库的当前状态、需要影响哪些文件，以及如何最佳实现技术方案。”
> 

**流程**：
1. AI 分析源代码和生产环境
2. 理解当前应用配置
3. 了解数据结构
4. 提出澄清性问题（范围、数据模型、UX/UI、验证方式等）

### 步骤3：创建计划（/create plan）

基于探索阶段的对话，创建 Markdown 格式的实施计划：

**计划模板包含**：
- 摘要（TLDR）
- 关键决策
- 详细任务描述
- 每个任务的状态追踪

> Zevi：“我经常使用不同的模型来执行某些任务。Cursor 有一个叫 Composer 的模型，速度极快。Gemini 3 在 UI 方面表现惊人。所以很多时候，我会把计划分成后端和前端，然后让 Gemini 读取计划并执行前端部分。”
> 

### 步骤4：执行（/execute）

使用 Cursor Composer 执行计划：

> Zevi：“Composer 速度惊人。它让你保持在创意流中。完整功能只需几分钟。”
> 

**执行特点**：
- Composer 速度极快
- 保持创意流不中断
- 传统工程师可能需要数天的工作，几分钟即可完成

### 步骤5-6：审查与同行评审

详见下方”代码审查与同行评审技术”章节。

### 步骤7：更新文档

> Zevi：“当 Claude 无法执行某些操作或遇到严重 bug 时，我会问：‘是什么导致你犯这个错误？’ Claude 会反思并思考是什么导致了错误。然后我说：‘好的，让我们更新你的工具和文档，以确保这个错误永远不会再发生。’”
> 

**关键实践**：
- 持续进行项目后分析
- 更新 Claude.md 文件（系统提示）
- 更新工具和文档
- 确保 AI 不会重复犯错

## 多模型协作策略

### 不同模型的特点与用途

Zevi 将不同 AI 模型想象为具有不同性格的人：

| 模型 | 性格比喻 | 优势 | 用途 |
| --- | --- | --- | --- |
| **Claude** | 完美的 CTO——善于沟通、聪明、有主见但协作 | 规划、沟通、全面分析 | 主要开发和规划工具 |
| **ChatGPT/Codex** | 最强程序员——穿着运动裤和凉鞋，在暗室工作 | 解决最棘手的问题 | 处理最困难的 bug |
| **Gemini** | 疯狂科学家——极具艺术天赋 | UI 设计、前端工作 | 前端开发 |
| **Cursor Composer** | 快速执行者 | 速度极快 | 快速实现功能 |

### 模型选择策略

> Zevi：“使用所有这些模型，发挥它们的优势并用其他模型弥补它们的弱点，对我来说是一个改变游戏规则的方法。”
> 

**具体应用**：
- **Claude**：主要工作流程、规划、代码审查
- **Gemini**：UI/前端工作（在 Anti-Gravity 中使用）
- **Codex**：复杂 bug 修复
- **Composer**：快速执行简单任务

### Claude.md 文件的作用

Claude.md 是加载到每次对话上下文中的系统提示：

> Zevi：“在 Claude.md 文件中，我设置了一些基本内容，比如’这是我们的工作流程，这是我们的工作方式。在探索阶段，我希望你挑战我的推理’等各种内容。”

## 从低代码工具到Cursor的进阶之路

### 工具进化路径

```
GPT 项目（CTO 模拟）→ Bolt/Lovable → Cursor + Claude Code
```

### 为什么从 Bolt/Lovable 迁移到 Cursor

> Zevi：“我感觉 Bolt 对我应该如何做事情有非常强烈的意见。我的知识达到了一定程度，我可以毕业并拥有更多控制权。”
> 

**Bolt/Lovable 的局限**：
- 过于急于编写代码
- 在项目初期很有趣，但后期会造成复杂问题
- 缺乏规划会导致复杂的 bug
- 对实现方式有强烈意见，灵活性不足

**Cursor 的优势**：
- 更多控制权
- 可以自定义决策
- 直接将 Claude 接入代码系统
- 拥有所有工具，可以做任何想做的事

### 渐进式学习建议

> Zevi：“如果你不在技术领域，代码是很可怕的。我把它看作一种暴露疗法。”
> 

**推荐的学习路径**：
1. **GPT 项目**：美观界面，简单使用
2. **Bolt/Lovable**：开始接触代码生成
3. **Cursor 清晰模式**：逐渐深入
4. **终端 + 深色模式**：成为完整开发者

> Zevi：“慢慢开始，逐步进行，直到你打开终端，使用完全深色模式，成为一个完整的开发者。”
> 

### 关于 Vibe Coding 工具的公平评价

> Zevi：“Base44、Bolt、Lovable、Replit、v0 等这些产品做得非常出色，基本上消除了创建产品的所有复杂性和猜测，让你能够简单地跟随直觉去构建。它还帮你设置 Google 登录、数据库，你不需要决定使用哪个数据库。”
>

## 代码审查与同行评审技术

### 核心挑战

> Lenny：“人们面临的主要挑战是审查 AI 编写的代码。”
> 
> 
> **Zevi**：“对我来说，发现错误非常困难。”
> 

### 解决方案：多模型同行评审

Zevi 开发了一种创新的”同行评审”技术，让不同 AI 模型相互审查代码：

```
1. 手动 QA 测试
2. Claude 自我审查（/review）
3. Codex 审查
4. Cursor Composer 审查
5. 同行评审（/peer review）——让主要模型评估其他模型的反馈
```

### 同行评审命令详解

> Zevi：“基本上，这个命令是说：‘你对这个项目负有最终责任。公司内的其他技术负责人分析了你的代码，审查了它，发现了这些问题。’ 不要不加质疑地相信他们说的一切。原因是你比他们有更多的上下文，你领导了这个项目。你需要解释为什么发现的问题不是真正的问题是错误的，或者自己修复它们。”
> 

**同行评审的好处**：
- 不同模型会发现不同的问题
- Claude 有时会说：“这已经是第三次被提出了。我第三次说，这不是问题，是故意的。”
- 模型之间相互竞争和验证

### /learning opportunity 命令

> Zevi：“每当我遇到太难理解的事情时，我就使用 /learning opportunity，然后谈论我想学习的内容。这基本上设置了 Claude 的场景：‘我是一个未来的技术产品经理，我有中级工程知识。我理解架构。我希望你用 80/20 法则解释我们当前正在做的事情。’”
>

## 在大公司应用AI编程的建议

### 对产品经理的建议

> Zevi：“我认为产品经理现在不应该交付复杂的数据库迁移或任何大型项目，但对于独立的 UI 项目，特别是如果你只是构建、创建 PR，然后交给开发人员做最终调整——我认为这绝对是可能的。”
> 

**可行的应用场景**：
- 独立的 UI 项目
- 创建 PR 供开发人员审查
- 学习和协作机会

### 使代码库对 AI 友好

> Zevi：“让你的代码库对 AI 友好是非常重要的一步，我相信这需要由技术人员来完成。基本上，我的代码库包含大量的纯文本。会有多个 Markdown 文件解释如何在代码库的某些区域工作以及高级框架，以便 AI 代理可以更容易地导航代码库。”
>

## AI对产品经理技能的影响

### 对”AI削弱技能”论点的反驳

> Zevi：“我强烈反对这种观点，并且听过很多次这种看法。”
> 

**常见误解**：
- “你基本上是在外包你的思考”
- “过度依赖 AI 会削弱技能”
- “产出质量会下降”

### Zevi 的反驳

> Zevi：“产品经理的工作不是总是有正确答案或成为房间里最聪明的人。至少按照我受过的培训和我相信的产品经理角色，恰恰相反。基本上，它是利用所有可用资源，尽快为用户交付正确的解决方案。”
> 

**关键观点**：
- AI 就像一个永远可用的智能导师
- 不评判你，真正帮助你
- 关键是有意识地使用这些工具
- 如果用 AI 生成内容然后直接发布——那是人为错误，不是 AI 的问题

### 对初级 PM 的价值

> Zevi：“特别是对于更初级的产品经理，这让你能够在比通常更高的水平上发挥作用。我在 Wix 时不会考虑公司的营销策略是什么，或者整个产品的入职流程应该如何彻底改革。但在我的副业产品中，我可以做任何我想要的决策，思考策略、营销和信息传递。”
>

## 用AI准备Meta面试

### AI 驱动的面试准备方法

> Zevi：“每当我面临新的挑战或问题时，我首先想到的是用 AI 来帮助找到解决方案。”
> 

**准备流程**：
1. 在 Claude 中创建专门的面试准备项目
2. 在网上搜索最佳信息（如 Ben Erez 的结构化方法）
3. 创建一个作为教练的项目
4. 使用 AI 进行模拟面试
5. 获取反馈

### 创新方法

**Base44 问答游戏**：
> **Zevi**：“我在产品问题中真的很难进行正确的细分。所以我创建了一个问答游戏，生成问题和不同的细分，我必须选择。”

**Comet 浏览器分析**：
> **Zevi**：“我使用 Perplexity 的浏览器 Comet，让代理对最常被问到的问题进行各种分析。这样我就知道应该优先准备哪些问题。”

### 关键经验

> Zevi：“最改变游戏规则的因素是与真人进行模拟面试。直接在 LinkedIn 上联系人们，请他们为我进行真实的模拟。”
>

## 职业建议与学习心态

### Failure Corner：Wix 的教训

**背景**：Zevi 在 Wix 开始作为实习生，被分配到编辑器团队——Wix 的核心产品，与非常资深的产品经理一起工作。

> Zevi：“我记得刚进来时想：‘我的第一次产品评审，我要让这些人印象深刻。’ 我基本上没有分享我在想什么，独自加班工作，心想：‘我要在这次产品评审中大放异彩。他们会印象非常深刻。’”
> 

**结果**：惨败。格式不对、无法回答问题、感觉很糟糕。

**关键领悟**：
> **Zevi**：“他们不期望我成为 10 倍更好的产品经理，他们期望我成为 10 倍更好的学习者。当我理解这一点时，我的整个心态都改变了。”

### 利用团队成员作为导师

**具体做法**：
- **Neri**：产品感觉最好——产品感觉问题找他
- **Oya**：方法论专家——方法论问题找她
- **Yahra**：系统思维——理解二阶、三阶效应

> Zevi：“每当我在这些领域之一遇到问题时，我就会去找他们咨询。这有两个目的。首先，我学到了很多。第二件事是，下次产品评审时，我的成功对他们来说感觉就像是他们的成功。”
