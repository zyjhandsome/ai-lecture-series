## 软件工程的定义与本质

### “软件工程”一词的起源

**Margaret Hamilton** 是第一个创造”软件工程”这个术语的人：
- 当时她离开了载人轨道实验室（Man Orbiting Laboratory）项目
- 正在参与阿波罗计划（Apollo Program）
- 她是少数几个在一群硬件/结构工程师中工作的软件开发者之一
- 她想要一个能区分自己工作性质的术语，于是开始使用”软件工程师”这个称谓

> **Grady Booch**：“NATO软件工程会议是在Margaret工作几年后才组织的，当时组织者有意选择这个有争议的名称——就像’人工智能’这个术语在西海岸第一次会议上被有争议地命名一样。”
> 

### 软件工程的本质

软件工程是一门**平衡多种力量**的工程学科：

| 力量类型 | 具体约束 |
| --- | --- |
| **物理定律** | 信息传递不能超过光速 |
| **硬件限制** | 系统规模受底层硬件约束 |
| **算法约束** | 有些问题理论上可解但实现困难（如Viterbi算法、快速傅里叶变换） |
| **人力约束** | 能否招到足够的人？能否有效组织团队？ |
| **法律问题** | 数字版权管理等 |
| **伦理问题** | 我们知道如何构建某些东西，但应该构建吗？ |

> **Grady Booch**：“理想的软件团队规模是零——当然这不太实际。次优是一人团队，然后规模逐渐增长。某些项目因其规模和持久的经济社会重要性，无法由小团队完成。”
>

## 第一个黄金时代（1940年代末-1970年代末）

### 时代背景

- **软件与硬件完全耦合**：早期编程就是在插板上插拔插头（如ENIAC）
- **所有软件都是定制开发的**，与特定机器绑定
- **“数字”（Digital）一词**：直到1940年代末才被创造
- **“软件”（Software）一词**：直到1950年代才出现

> **Grady Booch**：“如果用Carl Sagan的宇宙日历来衡量，软件会处于最后几纳秒。这是一眨眼都不到的时间。”
> 

### 主要特征

| 特征 | 描述 |
| --- | --- |
| **主要抽象方式** | 算法抽象（Algorithmic Abstraction） |
| **主要应用领域** | 数学计算、业务流程自动化（会计、工资单） |
| **代表人物** | Ed Yourdon、Tom DeMarco、Larry Constantine、Grace Hopper |
| **重要发明** | 流程图、高级编程语言（FORTRAN）、实体关系模型 |
| **劳动分工** | 分析师、程序员、打孔员、计算机操作员 |

### 边缘创新

虽然主流集中在商业和数学应用，但真正的创新发生在边缘地带：

| 领域 | 创新 |
| --- | --- |
| **国防工业** | 分布式系统、实时处理（SAGE系统） |
| **天气预报** | 数值计算应用 |
| **医疗设备** | 嵌入式软件 |
| **人机交互** | “所有演示之母”（Mother of All Demos） |
| **形式化方法** | David Parnas、C.A.R. Hoare、Edsger Dijkstra |

### SAGE系统：第一个大规模软件项目

- **全称**：Semi-Automatic Ground Environment（半自动地面环境）
- **目的**：应对苏联轰炸机越过北极入侵美国的威胁
- **规模**：据报道消耗了当时美国**20-30%的软件开发者**
- **持续时间**：从1950-60年代开始，最后一个系统直到1990年代才退役
- **历史意义**：这是第一个图形用户界面（CRT显示管）的诞生地

> **Grady Booch**：“在计算机历史上有两大主要影响力：一是商业，二是战争。因此我认为，现代计算的很多成果实际上是在悲伤的织机上编织的（referring to Jacquard’s loom）。”
> 

### 第一个黄金时代的危机

到1970年代末，出现了所谓的**“软件危机”**：

- **NATO软件工程会议**首次公开讨论这个问题
- **问题核心**：
    - 软件明显有用
    - 有强烈的经济激励
    - 但行业无法**足够快地**生产**足够规模**的**高质量**软件
- **注意**：这与今天的危机（监控、系统崩溃）性质不同

## 第二个黄金时代（1970年代末-2000年代初）

### 推动因素

| 因素 | 说明 |
| --- | --- |
| **复杂性增长** | 系统越来越复杂，第一代方法难以应对 |
| **分布式需求** | 国防领域对分布式系统的需求 |
| **微型化革命** | 集成电路、晶体管带来的个人计算机 |
| **新抽象理论** | 面向对象思想从Simula语言萌芽 |

### 主要特征

| 特征 | 描述 |
| --- | --- |
| **主要抽象方式** | 对象抽象（Object Abstraction） |
| **核心思想** | 将数据和算法结合到一个地方 |
| **代表人物** | Grady Booch、Ivar Jacobson、Jim Rumbaugh（“三剑客”）、Peter Coad |
| **重要发明** | UML、面向对象设计、设计模式 |
| **语言演进** | C++（Bjarne Stroustrup）、Object Pascal、Ada |

### Plato的哲学遗产

> **Grady Booch**：“柏拉图在他的对话中观察到看待世界的两种方式的二分法：一种是通过过程（事物如何流动），另一种是通过事物（原子）。’原子’这个词本身就来自希腊语。顺便说一下，柏拉图的著作现在在某些美国大学被禁了，因为他太激进了。”
> 

### 函数式编程的角色

John Backus（FORTRAN发明者）后来转向函数式编程研究：

> **Grady Booch**：“我在他去世前几个月采访过他，问他为什么函数式编程从未成为主流。他的回答是：**函数式编程使困难的事情变得容易，但使简单的事情变得极其困难。**”
> 

### 面向对象的局限

- **过度强调继承**：被认为是”银弹”，但实际上是个错误
- **核心价值**：通过类和对象看待世界的思想是正确的

### 开源软件的兴起

| 时代 | 开源形式 |
| --- | --- |
| **第一个黄金时代** | IBM SHARE用户组——用户自发分享软件 |
| **第二个黄金时代** | 开源库、操作系统（Unix/Linux） |

> **Grady Booch**：“在软件经济的早期，软件基本上是免费赠送的。IBM直到1960-70年代才意识到可以从软件赚钱，于是将软件和硬件解耦并开始收费。”
> 

### 平台时代的萌芽

- **服务导向架构（SOA）**的兴起
- **SOAP协议**的诞生
- **HTML/HTTP**成为传递信息和流程的更高层抽象
- 为当今AWS、Salesforce等平台奠定基础

> **Grady Booch**：“平台就是那些经济上有趣的城堡，被护城河环绕。像Salesforce这样的组织通过API让你跨越护城河——当然要收费。”
> 

### AI的第一次和第二次浪潮

| 时期 | 特征 | 结局 |
| --- | --- | --- |
| **1940-50年代** | 符号方法、第一个真空管人工神经元（SNARC） | 发现神经网络无法规模化 |
| **1980年代** | 规则系统、专家系统（如MYCIN） | 超过几百条if-then规则就无法处理 |

## 第三个黄金时代（2000年代初至今）

### 开始时间

> **Grady Booch**：“这可能有点争议，但我认为我们正处于软件工程的第三个黄金时代——它实际上始于千禧年前后，而不是现在。”
> 

### 标志性事件

| 事件 | 影响 |
| --- | --- |
| **互联网泡沫崩溃（~2000）** | 经济重组，但创新继续 |
| **Y2K问题** | 大量英雄式工作防止了灾难 |
| **抽象层级再次提升** | 从单个程序组件到整个库和包 |

> **Grady Booch**：“Y2K问题——人们回顾时说’我们本不需要担心’。但身处其中你会意识到，有大量英雄式的工作。如果没有完成这些工作，很多问题就会发生。这是最好的技术——你根本看不到它。”
> 

### 当前问题

| 问题类型 | 具体内容 |
| --- | --- |
| **软件管理** | 软件数量太多，如何管理？ |
| **安全问题** | 软件供应链攻击（如Stuxnet） |
| **经济问题** | 大型公司”大到不能倒”——微软或Google倒闭会怎样？ |
| **伦理问题** | 我可以追踪你每天的位置。我能做到，但我应该做吗？ |

### AI辅助编码的定位

> **Grady Booch**：“AI辅助工具如Cursor、ChatGPT是对第三个黄金时代已有力量的**反应**。我们有这么多库，但知道它们的人不够多。我们想加速它们的使用。”
>

## 对Dario Amodei预测的回应

### Dario Amodei的观点

Anthropic CEO Dario Amodei在达沃斯论坛上表示：

> “软件工程将在12个月内可自动化。”
> 

### Grady Booch的直接回应

> **Grady Booch**：“让我用一个科学术语来描述Dario说的话：**这是彻底的胡说八道（utter bullshit）**。我认为他根本性地错误理解了什么是软件工程。”
> 

### 反驳论点

| 论点 | 详细说明 |
| --- | --- |
| **软件工程不只是写代码** | 软件工程师是平衡技术、经济、伦理等多种力量的工程师。代码只是机制之一 |
| **CEO的利益冲突** | Dario需要向利益相关者发表吸引眼球的声明 |
| **训练数据的局限** | 这些工具主要在重复出现的问题上训练，适合Web-centric的CRUD系统，但计算世界远大于此 |
| **模式自动化vs.真正创新** | 这些工具擅长自动化已见过成百上千次的模式，但软件远不止于此 |

> **Grady Booch**：“计算世界比Dario看到的要大得多。它不仅仅是Web-centric的大规模系统。还有很多东西尚未被自动化。我们不断把边界推得更远。”
>

## 开发者的存在危机与历史教训

### 当前的恐慌

> **主持人**：“很多开发者，尤其是在冬季假期之后，感到存在性恐慌。假期前AI还只是擅长自动补全，但现在它们生成的代码真的很好，好到我开始信任它们了。”
> 

### 历史上的类似危机

| 时期 | 危机 | 结果 |
| --- | --- | --- |
| **1950年代** | 编译器的发明——“你永远无法像手写汇编那样高效” | 高级语言胜出 |
| **1960-70年代** | FORTRAN等高级语言——“我们能写出比任何机器都好的汇编” | 高级语言再次胜出 |
| **1980-90年代** | 面向对象——同样的担忧 | 面向对象成为主流 |

> **Grady Booch**：“这不是开发者第一次面临存在危机。他们在第一个和第二个黄金时代都面临过同样的危机。所以当我看到这个时，我会说：‘这也会过去的。’”
> 

### 为什么恐慌更大了

> **Grady Booch**：“当时没那么多危机感，因为我们人数不多——可能只有几千人。现在我们有**数百万人**在合理地问：这对我意味着什么？”
> 

### Grady的建议

> **Grady Booch**：“我向年轻开发者保证，这实际上是进入软件行业的激动人心的时期。**专注于基础技能，因为这些技能永远不会消失。**”
>

## 技能演变：什么会过时，什么会更重要

### 将会过时的技能

| 领域 | 原因 |
| --- | --- |
| **软件交付流水线管理** | 太复杂、太混乱，自动化有明显的经济和安全价值 |
| **简单应用开发** | iOS/Web应用——普通人通过提示就能完成 |
| **基础设施代码** | 代理可以帮助”启动某个服务”而不需要手写复杂代码 |

### 将会更重要的技能

| 技能 | 原因 |
| --- | --- |
| **系统级思维** | 重心从程序和应用转向系统本身 |
| **大规模复杂性管理** | 如何处理系统级的复杂性 |
| **平衡多种力量** | 技术、经济、人文等多重因素的平衡能力 |
| **人文技能** | 这些技能稀缺而微妙，需求只会增加 |

> **Grady Booch**：“如果你有管理大规模复杂性的技能，如果你作为软件工程师知道如何处理这些技术和人文的多重力量，你的工作不会消失。如果有什么的话，对你的需求会更大，因为这些人文技能是如此稀缺和微妙。”
>

## 给开发者的建议

### 推荐阅读与学习

| 推荐内容 | 作者/来源 | 价值 |
| --- | --- | --- |
| **《人工科学》** | Herbert Simon & Allen Newell | 系统理论基础 |
| **复杂系统研究** | Santa Fe Institute | 理解复杂性 |
| **《心智社会》** | Marvin Minsky | 多代理系统架构思想 |
| **包容架构（Subsumption Architecture）** | Rodney Brooks | 受生物系统启发的架构 |
| **黑板架构（Blackboard Architecture）** | Hearsay系统 | 多代理协调 |
| **《维护》第一部分** | Stewart Brand（Stripe Press出版） | 系统维护问题 |

### 关于代理编程

> **Grady Booch**：“我们现在进入了代理编程时代，我认为人们才刚刚开始触及如何应用这些东西。去读Minsky的《心智社会》，你会看到一些指导你处理多代理的想法。”
> 

### 关于学习心态

> **Grady Booch**：“Feynman会说：‘自己去做，因为那是你真正理解的唯一方式。’我的反应是：’那很好，但世界上有太多我好奇的东西，我无法全部理解。让我决定我想做什么，然后让机器帮我做。’”
> 

### 关于机遇

> **Grady Booch**：“现在约束你的主要是你的想象力。一些摩擦、一些约束、一些开发成本正在消失。这意味着我现在可以把注意力放在我的想象力上，去构建以前根本不可能的东西。”
>

## 关键语录

### 关于软件工程的本质

> **Grady Booch**：“我们是工程师，因为就像其他类型的工程师一样，我们构建平衡各种力量的系统，而且我们在一个绝对美妙的介质中这样做。”

### 关于计算历史的两大驱动力

> **Grady Booch**：“在计算历史上有两大主要影响力：一是商业，二是战争。因此我认为，现代计算的很多成果实际上是在**悲伤的织机**上编织的。”
> 

### 关于技术的最高境界

> **Grady Booch**：“最好的技术会蒸发、消失，成为我们呼吸的空气的一部分。”
> 

### 关于Dario Amodei的预测
