## Harness 定义与编码代理新内循环

### Harness 是什么

**核心观点**：Harness（套件/控制层）是编码代理的「agent loop」——负责调用模型、采样输出、提供上下文与工具、执行工具调用并反馈结果，是让模型能在真实环境中可靠、安全运行的关键工程层。

**要点提炼**：
- Harness 即 agent loop：调用模型 → 获取「下一步做什么」→ 执行工具调用 → 将结果反馈给模型
- 工具可以是简单命令（执行可执行文件、返回 stdout 和退出码），也可以是更复杂的交互式终端、网页搜索等
- 对 Codex 而言，Harness 必须处理安全与沙箱，确保模型生成的 shell 命令在受控环境中执行

**详细内容**：

> **Michael Bolin**："Harness 就是我们所说的 agent loop，它负责调用模型、采样模型输出，给它上下文——'这是我要做的事，这是可用的工具，告诉我下一步做什么'——然后拿到模型的响应，通常是工具调用，比如'调用这个工具、带上这些参数，告诉我返回结果'。有时工具很简单，比如运行这个可执行文件、告诉我标准输出和退出码；我们也在做更复杂的实验，比如更交互式的终端会话、网页搜索等。对 Codex 来说，Harness 承担了大量工作，因为我们非常重视安全与沙箱。它接收模型发出的 shell 命令或计算机操作指令，确保它们在沙箱中、在用户设定的策略下执行。这一块有很多复杂性。"

### 模型与 Harness 的关系

**核心观点**：模型主导体验，但 Harness 让代理得以发挥；二者需协同开发，确保工具在训练中被使用、与分布一致，才能在产品中可靠工作。

> **Michael Bolin**："模型肯定会主导体验。Harness 仍有很大创新空间，这不是纯研究问题。对我们团队来说，关键是工程侧与研究侧的协作——共同开发代理，确保任何人都能提出代理应如何工作的想法，确保 Harness 让代理发光、做它能做的最好的事，并给代理正确的工具，确保这些工具在训练中被使用、与分布一致，这样我们发布给世界时才能可靠工作。"

## 安全与沙箱：让代理在用户机器上可靠运行

### 各操作系统的沙箱实现

**核心观点**：Codex 针对 macOS、Linux、Windows 分别实现沙箱；macOS 用 Seatbelt，Linux 用 bubblewrap、seccomp、Landlock，Windows 自研沙箱，代码开源可查。

**要点提炼**：
- macOS：Seatbelt（系统自带，调用方式在开源库外）
- Linux：bubblewrap、seccomp、Landlock 等库组合
- Windows：自研沙箱，代码在 Codex 开源仓库中
- 所有工具调用经沙箱编排，按不同工具类型走对应策略

**详细内容**：

> **Michael Bolin**："你可以看到全部实现，因为都在我们仓库里。每个操作系统我们做不同的事：macOS 有 Seatbelt；Linux 用 bubblewrap、seccomp、Landlock 等库；Windows 我们自建了沙箱。像 Seatbelt 是 macOS 自带的，所以开源库里只有调用方式；Windows 沙箱的代码在开源库里。我们 orchestrate 所有经过沙箱的调用，针对不同工具调用以适当方式执行。"

### 安全（Safety）与安全（Security）的区别

**核心观点**：在 AI 代理语境下，Security 指沙箱与权限控制（能读/写哪些目录）；Safety 指模型端判断工具调用是否安全、是否应执行。Harness 忠实执行工具调用，安全判断在模型侧。

**详细内容**：

> **Michael Bolin**："Safety 和 Security 在 AI 里常被混用，但有细微区别。我讲的主要是 Security——你可以运行这个工具，但只能读这些文件夹、写这些文件夹。行业里多数人会区分：Safety 更多在后端，确保模型建议的工具调用本身是安全的。从 Harness 角度看，它是在'服从命令'，忠实执行工具调用；但哪些工具调用安全、是否该执行，是模型决定的。所以如果你 fork Codex 且仍用我们的模型、依赖我们模型的安全能力，你会得到那层保护；如果你用别人的模型，那就另当别论了。"

## Codex 发布与增长轨迹

**核心观点**：Codex 自 2025 年 4 月发布以来反响积极，使用量约为年初的 5 倍；2025 年 8 月 GPT-5 发布后 CLI 刷新，增长明显加速；VS Code 扩展后来居上超过 CLI；2026 年初推出的 Codex App 增长最快。

**要点提炼**：
- 2025 年 4 月：随 o3/o4 mini 发布
- 2025 年 8 月：GPT-5 发布，CLI 刷新，使用量显著跃升
- 2025 年夏末/秋：VS Code 扩展发布，使用量超过 CLI
- 2026 年初：Codex App 发布，增长最快
- 早期模型在工具调用与指令遵循上不足，后续模型改进带来明显加速

**详细内容**：

> **Michael Bolin**："反响非常积极。使用量大概是年初的 5 倍。我们 2025 年 4 月发布，但真正起飞是 8 月 GPT-5 出来、我们刷新 CLI 之后。之后我们做了 VS Code 扩展，人们更倾向用那个，VS Code 使用量超过了 CLI。今年初我们发布了 App，增长非常快，在很多方面是首创，很受认可。"

## Codex App：面向开发者的新界面

### 为何需要独立 App

**核心观点**：开发者传统上花大量时间在 IDE，但 Codex App 开辟了「任务控制中心」式的新界面——管理多对话、浏览 diff、内置终端，打破「必须把所有代码放在眼前」的预期。

**要点提炼**：
- CLI 面向终端用户，VS Code/JetBrains/Xcode 面向 IDE 用户
- Codex App 是「任务控制」式界面：并行管理多对话
- 可浏览代理生成的 diff、用 Command+J 打开终端做临时操作
- 核心价值：组织与跨多代理工作，成为最高优先级

**详细内容**：

> **Michael Bolin**："开发者历史上花很多时间在 IDE，在用户所在之处见面很自然。有的用户在终端，所以我们有 CLI；很多用户在 IDE，所以我们进了 VS Code，现在也集成到 JetBrains 和 Xcode。Codex App 则建立了新界面，我把它想成任务控制中心——现在可以并行管理多场对话，同时保留传统 IDE 的关键能力：可以浏览代理做的 diff，可以用 Command+J 弹出终端做临时操作，不必切到别的窗口。它打破了'我必须把所有代码放在眼前'的预期；对很多人来说，能组织、跨多代理工作才是最高优先级，我们把它放在最显眼的位置。"

## 编码代理如何改变开发者工作流

### 吞吐量与并行工作流

**核心观点**：编码代理首先带来吞吐量提升；开发者可并行推进多个任务，伴随一定上下文切换，但掌握后能显著推进进度；Codex 团队内部有人维护多个 repo 克隆、在会议间隙推送消息让任务持续前进。

**要点提炼**：
- 吞吐量提升：注意力集中时可并行推进大量工作
- 上下文切换：有人喜欢、有人不喜欢，但掌握后收益大
- 实践：Michael 本人有 5 个 Codex repo 克隆，在任务间切换；有时是小修复，有时是全天对话做大型改动
- 会议间隙：5 分钟空档可发一条消息，让任务继续推进

**详细内容**：

> **Michael Bolin**："最大的变化之一是吞吐量。人们发现，如果认真投入，可以并行推进很多工作，这会带来一定上下文切换，不是每个人都喜欢，但如果你能掌握，就能推进很多事。我 personally 有 5 个 Codex repo 的克隆，在不同任务间切换——有时是别处看到的小问题想顺手修，有时是全天对话在做大改动。很多人会在会议间隙、比如 5 分钟空档，发一条消息推一下任务，让它继续前进。"

### 优化新内循环与代码审查

**核心观点**：编码代理引入「新内循环」，开发者花更多时间思考如何优化这一工作流（是否做成 skill、是否分享给团队）；代码审查量也显著增加，Codex 本身也在做大量 code review，如何最大化利用这一点仍是移动目标。

> **Michael Bolin**："另一个变化是，很多人花更多时间思考如何优化这个工作流。这一切对大家都很新，所以会投入时间想：这件事我老在做，要不要做成 skill？要不要分享给队友？好的开发者总在找好工具、优化内循环，但这是新的内循环，我们都在摸索如何优化。还有一个很自然的变化是代码审查的量——Codex 也在做大量 code review，省了我们很多时间，但如何最大化利用这一点，感觉还是移动目标。"

## 面向代理的代码库与文档设计

### 文档、测试与 agents.md 的重要性

**核心观点**：许多长期被视为最佳实践的东西（文档、测试驱动开发）过去常被权衡掉；在代理优先世界里，这些变得明显值得投入；agents.md 应写人类新成员也需要知道的内容，且与代码一致、避免重复。

**要点提炼**：
- 文档、测试驱动开发等最佳实践在代理时代变得「明显值得」
- agents.md 内容应适合人类新成员，也是最佳实践
- Codex 团队采用「AGI 优先」思路：代理应自己决定做什么，不喂过多指令
- 代理大部分时间在读代码（如 ripgrep），形成自己的理解；agents.md 放代码里难以快速获取的信息（如如何跑测试、哪些测试更重要）

**详细内容**：

> **Michael Bolin**："有趣的是，整个代理式编码旅程里，有很多实践早就被当作最佳实践，但我们没做。比如文档、测试驱动开发——不是说没人做，但大家会权衡'值不值得'。现在在代理优先世界里，这些变得明显值得，人们几乎在重新发现这些长期已知的东西，但真正关心它们。agents.md 里的内容，至少我们写的，适合人类新成员加入团队时需要知道的，也是最佳实践。好在 Codex 里你可以直接看 repo 里我们怎么做。我们的 agents.md 比较 modest。我们自认是 AGI 优先（AGI-first）——代理应该自己决定做什么，而不是我们喂它过多指令。与其写一份与源码平行、可能重复且可能错误的文档，不如让代理大部分时间读代码、形成自己的理解。我们尽量在 agents.md 里放它可能不会很快从代码里得到的东西，比如'应该这样跑测试'、'这些测试比那些更重要'，但不过度。"

### 是否由代理更新 agents.md

**核心观点**：行业里很多人在个人指令里让代理在完成后更新 agents.md；Codex 团队不强制这么做，但不少人会在个人指令里加「完成后请更新 agents.md 中不明显的部分」；团队倾向 modest 做法，不是几十页指令。

> **Michael Bolin**："很多人现在就这么做。我们团队没有作为通用实践，但你可以查历史确认。很多人会在个人指令里说'完成后请更新 agents.md 中不明显的部分'。行业里我看到不同论文、不同实验，取决于代理本身。我们采取 modest 做法，不是几十页指令，大概就几项。"

## 上下文工程与 Prompt 策略

### 是否存在「过多上下文」

**核心观点**：从经验看，复杂任务通常写一段话左右的 prompt，让 Codex 熟悉相关代码，必要时给明确文件指针；多数时候不刻意塞满上下文，让 Codex 自己搜索；文件与文件夹命名良好很重要，因为代理会搜索代码库。

**要点提炼**：
- 复杂任务：约一段话的 prompt，让代理熟悉相关代码，可选地给文件指针
- 多数内容来自 agents.md、用户 prompt、可能的一些文件引用
- 可给 GitHub 访问权限，让代理看类似 PR、对话历史
- 原则：告诉 Codex 它有哪些工具，不规定具体解法，让它自己发挥

**详细内容**：

> **Michael Bolin**："我不是研究员，没法给你更精确的答案，但从经验看，复杂任务时我大概写一段话，让它熟悉那部分代码，有时给明确文件指针。很多时候我不刻意塞满，它搜索代码库做得不错。我会注意文件、文件夹命名——这些本来就是好实践，但对代理搜索代码可能比我们意识到的更重要。大部分上下文会是 agents.md、我的 prompt、可能的一些文件引用。我也给它 GitHub 访问权限，比如'类似的事发生在这个 PR'、'这个 PR 的进展'，它能看到代码和 PR 上的对话。但更多是告诉 Codex 它有哪些选项、工具箱里有什么，而不是规定最佳解法，让它做它的工作。它是好模型，能做好。"

### 架构与模式遵循

**核心观点**：Codex 会遵循代码库中的模式；若架构本身良好，它会维持并强化这些不变性；变化速率更高时，有标准就能持续获益。

> **Michael Bolin**："Codex 会遵循代码库里的模式。如果你一开始就有好架构，它会遵循、维持、强化你设定的不变性，长期会处于好状态。这对人类开发者也一样，只是变化速率现在高得多，所以有这些标准就能持续获益。"

## 模型 vs Harness：何者更重要

### 模型主导，Harness 让代理发光

**核心观点**：模型主导体验；Harness 应尽可能小、紧，给代理少量但强力的工具；Codex 不提供显式的「读文件」工具，而是给终端控制权，让代理用 `cat`、`sed` 等；唯一妥协在安全与沙箱——这是让 Codex 能「放手运行」的重要兜底。

**要点提炼**：
- Harness 尽量小、紧：工具少而强
- Codex 无显式读文件工具，给终端控制权，让代理用命令行工具
- AGI 优先：给代理大空间，由它找最优前进点
- 安全与沙箱是唯一重要妥协，是兜底
- 避免「Harness 作者比 Codex 更懂」的倾向，如用特殊工具管理上下文窗口；若工具会输出 1GB 数据，偏向让 Codex 写文件再 grep，而非塞满上下文

**详细内容**：

> **Michael Bolin**："我们尽量让 Harness 小、紧。你会在 Codex 里看到，相比其他方案，我们给的工具很少。比如 Codex 没有显式的读文件工具，我们给它终端控制权，如果它用 `cat`、`sed` 或别的命令行工具读文件，我们鼓励它这么做。我 earlier 提到 AGI 优先——我们给它大空间，它应该找到空间里的最优点前进。某种意义上 Harness 在多个维度上变小，我们尽量保持小，只给少量非常强大的工具。唯一妥协是安全——沙箱、我们如何做安全，是重要的兜底，让我们能'放手让 Codex 跑'。有时人们会玩技巧，用特殊工具管理上下文窗口，感觉像 Harness 作者比 Codex 更懂、要 bias 它做某些事。我们尽量避开。如果它会跑一个吐出 1GB 数据的工具，我们不想把 1GB 塞进上下文，因为可能不相关。我们的看法是，bias Codex 让它写文件、再 grep 或做别的，它仍然可以自由选择怎么解。"

### 沙箱能否替代人类在环

**核心观点**：对编码任务，沙箱是主要的「人类在环」替代；在给定 prompt、问题、受沙箱约束的环境中，让代理探索空间能获得最佳解；若在多个并行任务上每隔几分钟就要人工介入，会从根本上限制吞吐量；训练时人类在环的纠错应发生在训练阶段，在推理时体现。

> **Michael Bolin**："对编码、至少我们关注的任务，沙箱是主要的替代。你有 prompt、问题、给 Codex 的环境，它受沙箱约束，让它探索空间能获得最佳解，尤其是在规模上。如果我 5 个 Codex 克隆都要每隔几分钟介入，那会从根本上限制吞吐量。训练时，我们希望人类在环的纠正在训练时发生，推理时体现。"

### Harness 的可靠性与性能

**核心观点**：Harness 可靠性很重要——若它崩溃，会话就结束，模型无能为力；因此性能与可靠性是核心原则；资源应尽可能用在模型上，Harness 做最小必要工作。

> **Michael Bolin**："Harness 的可靠性也很重要。Harness 崩溃，会话就结束，模型能做什么？所以性能与可靠性是我们实现 Harness 时的核心原则。我们不想搞一堆树莓派农场，每个跑 agent loop——要对机器资源 judicious，大部分能量花在模型需要做的事上，Harness 理想情况下做最小必要工作。"

## 多代理、工具集与未来方向

### 多代理与跨机器协作

**核心观点**：未来会走向多代理、子代理、跨机器通信；Harness 将不仅是单机进程，而是代理网络；Michael 认为短期内不会失业，但角色在变：从为开发者写工具，转向为代理写工具。

> **Michael Bolin**："我们 inevitably 会走向多代理、子代理、跨机器通信。那时 Harness 就不只是单机进程，而是代理网络。我预期那里还有很多有趣的工作，短期内不会失业。变化是，我职业生涯大部分时间在为开发者写工具，现在可能更多在为代理写工具。"

### 工具集：少而强

**核心观点**：倾向少量非常强大的工具，让代理用它们探索空间；代理也能自己写工具，但更希望有少量强大原语；具体原语集仍在实验中。

**要点提炼**：
- 终端工具：模型通过类人方式使用计算机终端，不只是执行单条命令；处理流式输出、高效使用
- 记忆：历史每次对话从零开始，所以有 agents.md 和上下文填充；repo 里有很多记忆相关实验
- 上下文连接器：从本地计算机任务扩展到发邮件、创建文档、在浏览器中操作等
- 上下文窗口：更大一般更好；如何压缩、触及限制时如何处理仍在积极探索

**详细内容**：

> **Michael Bolin**："代理显然可以自己写工具，但我们更希望有少量非常强大的工具让它探索空间。我们也会继续实验哪些原语值得聚焦。我觉得很多 pieces 已经看到了。我 earlier 用 shell 工具——更准确说是终端工具——模型通过终端与计算机交互，像人类一样，不只是执行单条命令，处理流式输出、高效使用。记忆是另一个大领域，有很多有趣工作。历史上每次对话从零开始，所以有 agents.md 和上下文填充；repo 里有很多记忆实验。不同上下文连接器——从本地计算机任务扩展到发邮件、创建文档、在浏览器操作——也有很多。还有标准 LLM 的东西：更大的上下文窗口一般更好，如何压缩、触及限制时怎么处理，仍在积极追求，都影响整体代理体验。"

## 关于 Slop 与代码质量

**核心观点**：Michael 未在 Codex 输出中明显看到「slop」；有时模型会倾向于写代码，答案可能是删代码；有时是结构问题（如在一个文件加 500 行而非新建文件），这类问题较易修复；更常见的是 Codex 用了讲者不熟悉的 idiom 或语言特性，反而带来学习。

> **Michael Bolin**："用 Codex 没怎么想到特别明显的问题。我不觉得有我会称为 slop 的东西。有时这些模型喜欢写代码，所以答案可能是删代码，有时需要更明确地这么说。或者它在一个文件加了 500 行，也许该新建文件——这类容易修。但很多情况下，更常见的是它知道某个 idiom 或语言特性，我还没遇到过，它加了，我学到新东西。这是 Codex 让我惊讶的常见方式，而不是反过来。"
