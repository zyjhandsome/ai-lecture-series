- 战略基金单笔体量较大，多数战略基金单笔仅 50 万–100 万或几百万美元
- 除资金外，提供微软生态与业务价值

**详细内容**：

**融资建议**：Alan 建议创业者「都聊」。AI 深科技创业者往往技术背景强，需要投资方也有一定技术门槛才能深入对话；若只和财务 VC 聊，深度可能有限。建议先确定领投的财务 VC，再找更懂该领域的战略投资者补齐本轮，这种组合能向市场释放强信号。

## 创业公司生存空间与融资建议

**核心观点**：创业公司被大厂功能「碾压」的担忧被放大；大厂在具体产品上团队小、资源有限，且自我革命有惯性；创业公司可更快迭代、试错成本低、可聚焦垂直场景，仍有显著优势。

**要点提炼**：
- 大厂在具体产品上团队规模有限，可调资源也有限
- 大厂既有产品已盈利、已售出，自我革命存在阻力与惯性
- 创业公司无包袱、可跑得更快、用最新技术、试错成本低
- 创业公司客户多为同类创业公司，更愿意试错
- 大平台做 80% 通用技术，创业公司可做 1%–2% 垂直软件，垂直可做得很大

**详细内容**：

Alan 认为这种担忧有些被放大。大厂如 Google、微软、亚马逊体量很大，但在具体产品上，往往是一个小团队在做，可调资源有限。更重要的一点是速度：大厂既有产品已很赚钱或已卖给大量客户，若要自我革命，存在阻力和惯性。创业公司没有包袱，可以跑得更快、用最新技术、反复试错，试错成本很低；而大厂一旦出错，客户很难原谅。创业公司卖给的对象往往也是创业公司，更愿意尝试和试错，因此可以快速迭代、快速适配产品，仍有很大优势。

此外，大平台更聚焦做通用技术（服务 80% 市场用户），创业公司可以先找一个相对利基或更窄的垂直领域，深耕该垂直，不做 80% 的通用软件，只做 1%–2% 的垂直技术，也可以做得很大。

## AI 软件护城河：易颠覆与难颠覆

**核心观点**：最易被颠覆的是「浅层」软件（如数据搬运、RPA、单功能软件、咨询/市场研究）；难颠覆的是企业信息流枢纽（如 Atlassian、ServiceNow），因其掌握工作流 know-how、企业专属数据，且 Agent 公司难以获取防火墙内的真实数据。

**要点提炼**：
- 易颠覆：数据搬运类、RPA、单功能软件、咨询/市场研究
- RPA 从「记录行为→重复执行」升级为 Agent 的「推理→执行」，是颠覆性技术
- 难颠覆：企业信息流枢纽（Atlassian、ServiceNow），掌握工作流 know-how 和企业数据
- 真正有用的数据在防火墙内，Agent 公司难以获取
- GPT-4 训练用约 1.5 PB 数据，而摩根大通、PayPal 等企业防火墙内有 15–30 PB 数据

**详细内容**：

**易被颠覆的软件**：

1. **类似 Porter 的浅层软件**：如把数据从这里搬到那里，非常浅。
2. **RPA（Robotic Process Automation，机器人流程自动化）**：最早只是记录行为、再用机器重复；现在用智能 Agent 替代，从「记录→重复」变为「推理→执行」，是颠覆性技术，会冲击绝大多数 RPA 服务商。
3. **单功能软件**：Alan 举例——朋友做 MRI/NMR 检查，医生给了一张 CD，需要专门软件才能读取，软件要花几百美元购买；朋友直接把文件扔给云端，云端自动生成软件读取。这类非常薄、单功能的软件很容易被颠覆。
4. **咨询、市场研究**：顾问大量阅读二手材料、做一手调研，但多数是二手材料；Agent 可在短时间内读完所有线上可得信息，人类无法在这一点上与之竞争，该领域正在快速被替代和颠覆。

**难被颠覆的软件**：

部署 Agent 时，Agent 需要大量上下文和真实数据（ground truth），数据来源在防火墙内，Agent 公司从外向内获取不到。像 **Atlassian**、**ServiceNow** 这类公司，天然成为企业信息流枢纽——无论工单、IT 支持、Slack、后端聊天软件，都会汇聚到某一点，这类公司非常持久。原因包括：

1. **不会被 Agent 替代，而是赋能 Agent**：它们给 Agent 提供真正需要的信息。
2. **自建 Agent 成本高**：企业若自建 AI Agent，需招募昂贵人才、测试合规、承担责任转移；出问题时可以找供应商做回滚、打补丁，自建则责任在己。
3. **Atlassian CEO 观点**：企业真正的护城河是几十年积累的流程经验；真正有用的数据都在防火墙内，Agent 公司难以轻易获取。

Alan 引用数据（略旧但仍有代表性）：GPT-4 训练约用 1.5 PB 数据，而摩根大通、PayPal 等企业防火墙内有 15、20、30 PB 数据。当企业能合理利用自身数据时，对性能和可用性的提升会非常显著。

## 传统软件 vs AI 原生：客户分层

**核心观点**：传统软件更适合卖给大企业（采购流程复杂、合规严格）；AI 原生工具更适合新兴行业和创业公司（试错成本低、团队已习惯新工具）。两种模式都有生存空间。

**要点提炼**：
- 大企业采购流程复杂、合规严格，更倾向买微软、Atlassian、ServiceNow 等成熟服务
- 大企业一旦出错，成本巨大甚至不可承受
- 新兴行业、创业公司更愿用 AI 原生工具，试错成本低、可跑得更快
- 创业公司要跑赢大厂，需敢于尝试新工具、新平台、新模型

**详细内容**：

需按目标客户区分。传统软件卖给大企业时，因其采购流程复杂、涉及严格合规，更倾向从微软、Atlassian、ServiceNow 等购买成熟服务体系；一旦出错，成本很高甚至不可承受。而服务新兴行业或创业公司的，创业公司可能更愿用 AI 原生工具——招的人可能在学校就用过新工具，试错成本低、可跑得更快。Alan 此前提到，创业公司要跑赢大厂，需敢于尝试新工具、新平台、新模型。不同市场各有空间，两种模式都活得不错。

- 很多公司因创始人背景强，产品未成熟、研发刚起步，就能拿数亿甚至数十亿种子轮
- AI 算力 + 早期公司投资已超或接近万亿美元
- 头部 AI 公司（OpenAI、云厂商等）年化收入合计可能不超过 100 亿美元
- 未来 3–5 年需产生约 8000 亿美元收入才能 justify 此前投资

**详细内容**：

「泡沫」一词很松散，需区分价值与价格。AI 是否有价值？绝对有，毋庸置疑。价格端是否存在跟风或 FOMO？肯定存在。很多公司因创始人背景强，产品未成熟、研发刚起步，就能拿数亿甚至数十亿种子轮，市场有些过热、存在形式主义表现。

从具体数字看：整个 AI 算力（含早期公司投资）已超或接近万亿美元；而若看最大 AI 公司（如 OpenAI、云厂商等），年化收入合计可能不超过 100 亿美元，中间缺口有人估算约 1 万亿美元或 8000 亿美元。换言之，未来 3–5 年，这些公司或新兴 AI 公司需产生约 8000 亿美元收入，才能 justify 此前万亿美元投资，缺口相当大。市场可能需要重新考虑：一是早期公司的投资定价；二是硬件与算力中心投资——不应只建新数据中心，而应更关注提升现有数据中心效率。

## 数据中心利用率与硬件创新

**核心观点**：GPU 集群利用率仅 30%–40% 甚至更低，主因包括训练/推理失败率高、over-provision 备份 GPU；通过 VM/容器化、光纤替代铜缆、毫米波替代铜/光纤等，可提升利用率至 60%–70%，并降低能耗。

**要点提炼**：
- GPU 集群利用率约 30%–40%，可能更低
- 失败率高：软件层错误、GPU 间/Blade 间连接线安装错误等
- 训练时 over-provision，大量备份 GPU 闲置，但有必要
- 方案：VM/容器化提升利用率；硬件上光纤替代铜缆（但光纤对温度、弯曲敏感）；毫米波（RF）替代铜或光纤
- 约 70% 失败与数据线连接端有关，约 10%–25% 涉及硬件数据线连接
- 目标：利用率从 30% 提升至 60%–70%，同时降低能耗

**详细内容**：

现有数据中心在 GPU 集群层面的利用率很低，有人说是 30%–40%，Alan 认为可能更低。花万亿美元建了这么多数据中心，利用率却这么低，是继续建新数据中心，还是聚焦提升现有数据中心利用率？这可能是市场未来 12–18 个月会做的事。

**利用率低的原因**：GPU 在训练或推理时失败率很高，软件层会出错，GPU 间、Blade（板卡）间连接线安装错误也可能导致整个训练失败。此外，训练时会 over-provision，放很多备份 GPU 闲置，一旦出错可立即把负载转移到另一块 GPU，这是有益做法。

**提升方向**：有人提出用类似早期 VM、Docker、Kubernetes 的方式提升 GPU 利用率；硬件上，约 70% 失败与数据线连接端有关，光纤是更贵的替代方案，但光纤对温度敏感（摄氏一度即影响性能）、对弯曲敏感，弯到一定程度会衰减信号。现在有新技术，如用毫米波（RF）替代铜或光纤，从不同角度解决 GPU 利用率低、错误率高的问题，目标是把利用率从 30% 甚至更低提升到 60%–70%，同时降低能耗。

## M12 投资组合画像

**核心观点**：M12 投资 100+ 公司、15 家独角兽、6 家已上市；重点赛道包括新硬件（光学替代传统芯片）、数据、AI 原生网络安全、垂直软件、游戏、区块链与 AI 交叉（Agent 身份与支付）。

**要点提炼**：
- 新硬件：光学替代传统芯片，代表公司 Neuroflux、d-Matrix
- 数据：Micro1 等，帮助企业获取专业、定向、高质量数据
- AI 原生网络安全：HiddenLayer、Protect AI 等，提供模型安全、模型评估
- 垂直软件：如 Didero，用 AI 替代传统供应链工作流
- 游戏：大世界模型、物理模型发展使游戏成为自然应用场景，行业在复苏
- 区块链与 AI：Agent 身份管理、Agent 支付等交叉应用

**详细内容**：

**新硬件**：用光学替代传统芯片，代表公司 Neuroflux、d-Matrix。

**数据层**：数据是终极护城河之一，投资了 Micro1 等，帮助企业获取非常定向、干净、高质量的数据。

**AI 原生网络安全**：提供网络安全、模型安全，代表公司 HiddenLayer、Protect AI；模型评估方面有 Alyssa 等。

**垂直软件**：如 Didero，用 AI 替代传统供应链中的部分工作流。

**游戏**：游戏行业在缓慢复苏，此前投资很低；大世界模型、物理模型发展使游戏成为自然应用场景，行业有新关注度。

**区块链与 AI**：区块链本身不新，但对 M12 是较新话题；关注区块链与 AI 的交叉点，尤其是 Agent 身份管理、Agent 支付，可能看到区块链技术的应用。

## Agent 支付：身份、合规与微支付

**核心观点**：Agent 支付与真人支付范式不同——付款方、收款方都变了；当前市场多停留在叙事或 demo 阶段；核心挑战包括 PCI 合规（Agent 不能持有信用卡信息）、身份（Agent 需原生身份）、一对多/多对多支付、微支付（传统通道无法支持）。

**要点提炼**：
- 法律上 Agent 不能获取个人信用卡信息，违反 PCI 合规
- 早期方案：虚拟卡（Stripe 等）限制额度，但速度慢、成本高、纠纷复杂
- 另一方案：Visa、Mastercard 的 tokenized credentials，跳过虚拟卡，用个人凭证直连银行，相对可行
- 但上述仍用真人身份，非「纯 Agent 支付」
- 真正挑战：Agent 需要原生身份、可验证可审计；一对多/多对多支付；微支付（可能小到 fraction of penny），传统通道无法支持
- 复杂场景（如 Agent 采购数据、算力、模型，多方向微支付）可能需要稳定币等新基础设施

**详细内容**：

**范式变化**：付款方、收款方都变了。当前市场多停留在叙事或实验/demo 阶段，真正的智能 Agent 做大额或「无形」支付尚未普及。

**合规挑战**：法律规定 Agent 不能获取个人信用卡信息，违反 PCI 合规；任何个人支付信息，尤其是信用卡，不能直接给 AI。

**早期方案**：
1. **虚拟卡**（如 Stripe）：一次性使用、预设额度（如 100 元），Agent 花完即止。Alan 认为这不是长期可持续方案——速度慢、仍用真人身份、成本高；且一旦 Agent 购物产生纠纷，从「人与商户」变成「人+Agent 与商户」，纠纷更复杂。
2. **Tokenized credentials**（Visa、Mastercard）：跳过虚拟卡，用个人凭证直连银行，设定额度（如 5 美元），通过 token 认证支付。Alan 认为这在近期可能是相对可行的方案，但仍非「纯 Agent 支付」，因为仍用真人身份。

**真正挑战**：
- **身份**：若企业采购员需要 Agent 代为采购数据、算力等，Agent 用谁的身份？采购员身份还是 Agent 原生身份？如何给 Agent 原生身份、且可验证可审计？
- **一对多/多对多**：过去多是一对一支付，现在可能一对多、甚至多层级一对多，传统通道难以解决。
- **微支付**：可能小到 fraction of penny，传统通道无法支持；如按使用量计费算力，每次使用即定价、即支付，这类场景可能需要稳定币等新基础设施。

**简单场景**（如 Agent 帮忙网购跑鞋）可能不需要稳定币，用传统通道更便宜、可靠、安全；复杂场景（一对多、多对多、微支付）则稳定币有潜力。

## Agent 安全：身份、数据与模型

**核心观点**：Agent 创造了全新攻击面，传统被动防御无法应对；需转向主动防御；热点包括 Agent 身份、数据安全、模型安全、Runtime Security；传统安全公司能力有限，新领域有收购案例（如 Vez、Protect AI）。

**要点提炼**：
- Agent 身份是当前最热话题之一：从「借用人+智能设备」到「Agent 原生身份」
- 传统安全是「开门」逻辑，进来后做什么多靠人自律；对 Agent 的信任未达此水平
- 需实时查看 Agent 在做什么、事后审计、出错时回滚与补救
- 数据安全：Agent 调用数据时的细粒度管理，涉及隐私的数据 Agent 不能看
- 模型安全：HiddenLayer、Protect AI 等，red teaming、pentesting，防 prompt injection 等
- 威胁主体变化：从外部威胁到内部威胁（Agent 被 prompt injection 或自主产生幻觉）
- Runtime Security：Agent 可访问多环境，实时监控其在做什么、是否合法，非常难，尚未有规模化方案

**详细内容**：

**Agent 身份**：从早期 Agent 借用人的身份，到如今 Agent 需有原生身份。传统网络安全相对简单——「开门」，进来后做什么多靠人自律；对 Agent 的信任未达此水平，需决定 Agent 可访问哪些资源（数据、工具），并实时查看 Agent 在做什么、事后审计、出错时回滚与补救。该领域有公司融了很多钱或被收购，如 Vez。

**数据安全**：Agent 实时调用数据时，防御需非常严格；涉及个人隐私的数据 Agent 不能看，需更细粒度管理。

**模型安全**：如 HiddenLayer、Protect AI（后者以约两倍价格被收购），做 red teaming、pentesting，确保模型上线后不会被 prompt injection 等攻击窃取秘密数据。

**传统安全公司的局限**：Alan 认为能解决这些新问题的传统公司不多。Agent 创造了全新攻击面，传统网络安全主要是被动防御，而 Agent 做事速度快、潜在风险大，被动防御无法预见和避免，需主动防御。模型安全是主动防御代表；传统防火墙、身份验证等无法解决 Agent 身份问题。威胁主体也变了——从外部威胁到内部威胁，Agent 可能受外部 prompt injection 或自主产生幻觉，带来不可预见的内部威胁。Runtime Security 非常难：Agent 可访问多个环境，实时监控其在做什么、是否合法，目前很多公司声称在做，但真正达到规模的尚未见到。

## AI 能否写底层代码

**核心观点**：AI 写底层代码仍有距离，应停留在应用层；底层代码需 100% 可靠，而 Agent 输出有概率性，且人们对其信任未达此水平；应用层出错风险有限，可容忍。

**要点提炼**：
- 底层代码复杂度远超现有 Agent 能力范围
- 底层代码与底层工具一旦出错，可能给公司毁灭性打击，必须 100% 可靠
- Agent 写代码有概率性，无法做到 99% 可靠
- 应用层出错，风险与损害有限，可容忍
- 对基础设施公司启示：代码生成、代码审查、设计阶段 AI 辅助是机会

**详细内容**：

Alan 认为 AI 写底层代码（如数据库引擎，对延迟、吞吐、稳定性要求极高的环境）仍有距离，应停留在应用层。底层代码的复杂度远超现有 Agent 能力；且 Agent 写代码有概率性，底层代码不能 99% 可靠，必须 100% 可靠，因为底层基础设施一旦出错，可能给公司毁灭性打击。此外，人们对 Agent 的信任尚未达到可交付底层的水平。应用层出错有一定概率可容忍，因为一旦出错，风险和损害有限。

## 代码生成护城河：Context is King

**核心观点**：代码生成领域「context is king」；长期护城河在于谁能掌握、积累、集中 context；设计阶段（可看到 Jira、Slack、Notion 等全流程数据）比代码审查（只看成品）更有 context；若 Codex 等封闭化，可能威胁开源平台生存。

**要点提炼**：
- 用前端模型（如 Opus、Codex）写代码已写得不错，真正护城河在 memory、context 的集中
- Codex 若某天完全封闭，可能威胁很多开源平台生存
- 代码审查是成品的最终审查，context 有限
- 设计阶段有更多 context：Jira ticket、Slack、Notion Board、公司偏好、行业要求等
- 该领域有 Clover Security、Prime Security、Clearly、CISO 等公司在做，处于早期
- 设计阶段 AI 辅助是 M12 更关注的领域

**详细内容**：

代码生成（codegen）领域，用前端模型（如 Opus、Codex）写代码已写得不错；真正长期的护城河在于确保 memory、context 集中在你的层面。若 Codex 某天完全封闭，可能威胁很多开源平台（如 Cursor、Lovable、Replit 等）的生存。代码审查是对成品的最终审查，context 有限；更多人认为真正有更多 context 的是设计阶段——可看 Jira ticket、Slack、Notion Board，以及公司偏好、个性化设置、行业要求等，全集中在设计过程。该领域有 Clover Security、Prime Security、Clearly、CISO 等公司，已有一定进展。Alan 总结：**context is king**，全流程谁能掌握 context 的积累与集中，谁就能最终胜出。

## 强监管行业 AI 落地

**核心观点**：强监管行业 AI 落地不是技术问题，而是 know-how 问题；关键在于能否跑通复杂合规流程；M12 投资了一家帮助网络安全软件卖给政府的平台，将合规负担从软件公司转移到中间商；FedRAMP 认证成本约 500 万–1500 万美元、周期约 3 年。

**要点提炼**：
- 强监管行业（医院、金融等）的阻力多在非技术层面
- 能跑通复杂合规流程才是关键
- FedRAMP（美国联邦政府审计体系）成本约 500 万–1500 万美元，周期约 3 年
- 多数公司不愿投入，MSSP（托管安全服务商）类中间商可承担合规负担
- M12 投资的平台帮助网络安全软件卖给政府，客户主动上门

**详细内容**：

Alan 认为这不是技术问题，而是 know-how 问题。这类公司往往卡在最后，阻力不在技术层面，而在于能否跑通整个复杂合规流程。

**案例**：M12 投资了一家软件平台（信息尚未公开），帮助其他软件（尤其是网络安全领域）把产品卖给政府。美国联邦政府有相对严格的审计体系 FedRAMP，除非公司很大、资金充足，可请大量律师和审计师完成，否则成本约 500 万–1500 万美元，周期约 3 年。多数公司不愿在此投入——现有客户都做不完，为何花这笔钱、等 3 年？网络安全领域有 MSSP（Managed Security Service Provider），类似承包商或中间商，把别人的软件放到自己平台，再转卖给医院系统、政府系统或大企业，把合规负担从软件公司转移到中间商；中间商经多年积累，对流程非常熟悉，也有现成渠道。对想进入强监管环境的公司，这是不错的解决方案。M12 投资该公司后，客户主要主动上门，不需花太多精力说服。

## 创业建议

**核心观点**：AI 普及降低了创业门槛，但门槛低意味着竞争更激烈；真正的差异化在于目标领域和自身 know-how；技术创业者需补足产品化、市场化能力；融资上重速度而非估值，可拆分为多轮小融资；团队需多样性，走出舒适区。

**要点提炼**：
- AI 普及（如 Lovable、Replit、Cursor、Codex）使创业门槛很低，不需自建数据中心、不需学编程
- 门槛低意味着所有人门槛都低，差异化在目标领域和 know-how
- 技术创业者常在产品化、市场化上有短板，需在生态中做研究、了解技术应用与目标市场
- 融资：早期重速度，合理估值、合理条款、真正能帮你的投资人更重要；可把大轮拆成 1–2 轮小融资，milestone 设小一点
- 基础模型训练首轮约 3000 万–5000 万美元，可先融 500 万做 POC/demo，再融资
- 产品不只卖给本领域，可能卖给其他垂直，团队需有懂其他行业、懂 go-to-market 的人
- 中国创业者倾向与中国人合伙，建议团队有多样性，走出舒适区，结识不同背景的人

**详细内容**：

**方向**：Alan 不敢指具体方向，但 AI 普及使创业门槛非常低——云使软件创业成本降低，AI 尤其替代写代码，不需学编程，用 Lovable、Replit、Cursor、Codex 等即可写出相对完整的小程序、小产品。门槛低意味着所有人门槛都低，更重要的差异化在于目标领域和自身 know-how；例如来自传统医疗行业，懂行业细节，即使用相同工具，也能更合理使用，在设计产品时知道哪些是行业独有的、只有业内人士才懂的东西，这才是真正的差异化。

**技术创业者**：很多新创业者技术很强，但在产品化、市场化上有短板；需在生态中做研究，了解技术的未来应用、目标市场、具体应用场景。

**融资**：早期更重速度，不必执着于某估值或必须从某大机构拿钱；合理估值、合理条款、真正能帮你的投资人更重要。融资额可更灵活；基础模型训练首轮约 3000 万–5000 万美元，可拆成先融 500 万做 POC、做出可 demo 的东西，再把大轮拆成 1–2 轮小融资，milestone 设小，可能最终跑得更快。

**团队**：很多创业者非常熟悉本领域、认识所有人，但产品不只卖给本领域，可能卖给其他垂直甚至完全不懂的领域；早期建团队时需考虑是否要有懂其他行业、懂 go-to-market 的人。中国创业者可能因性格或文化更愿与中国人合伙，建议团队有多样性，走出舒适区，结识不同背景的人，在技术之外有技能互补。

**个人**：Alan 喜欢与创业者交流，觉得对自己提升很大，也帮助过很多优秀创业者融资和业务发展，欢迎有兴趣的朋友联系。
