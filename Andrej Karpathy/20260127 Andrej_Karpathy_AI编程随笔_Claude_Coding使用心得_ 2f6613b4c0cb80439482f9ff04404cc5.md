## 编程工作流的剧变

### 从手动到Agent的转变

> “我确实现在主要用英语编程了，有点不好意思地告诉 LLM 要写什么代码……用语言描述。”
> 

**转变时间线**：
- **2025年11月**：80% 手动编码+自动补全，20% Agent
- **2025年12月**：80% Agent 编码，20% 编辑和微调

**转变速度**：仅用几周时间完成

### 对行业的影响判断

| 群体 | 感知程度 |
| --- | --- |
| 工程师群体 | **两位数百分比**（10%+）正在经历类似转变 |
| 普通大众 | **低个位数百分比**的人意识到这一变化 |

### 个人感受

> “这对我基本编程工作流来说是近20年来最大的变化，而且是在几周内发生的。”
> 
- 承认这种转变”有点伤自尊”
- 但用大规模”代码动作”操作软件的能力实在太有用了
- 需要适应、配置、学习，并理解它能做什么和不能做什么

## IDE与Agent的现实问题

### 对当前炒作的判断

Karpathy 认为以下两种炒作**目前都过度了**：

| 炒作类型 | Karpathy 的观点 |
| --- | --- |
| “不再需要 IDE” | 过度炒作 |
| “Agent 集群” | 过度炒作 |

### 模型仍然会犯错

> “如果你有任何真正在意的代码，我会像盯着老鹰一样盯着它们，在旁边开一个大的 IDE 窗口。”
> 

**错误性质的变化**：
- ❌ 不再是**简单的语法错误**
- ✅ 变成了**微妙的概念性错误**——就像一个有点粗心、匆忙的初级开发者会犯的那种

### 模型的具体问题清单

| 问题类型 | 具体表现 |
| --- | --- |
| **错误假设** | 最常见——代表你做出错误假设，然后一路狂奔不检查 |
| **不管理困惑** | 不会主动澄清自己的困惑 |
| **不寻求澄清** | 遇到不明确的地方不会问你 |
| **不暴露不一致** | 发现矛盾时不会提出来 |
| **不呈现权衡** | 不会展示不同方案的利弊 |
| **不该同意时同意** | 应该反驳时不反驳 |
| **过度谄媚** | 仍然有点太 sycophantic（讨好用户） |
| **过度复杂化** | 喜欢把代码和 API 搞复杂 |
| **抽象膨胀** | 抽象层越堆越多 |
| **不清理死代码** | 不会自己清理无用代码 |
| **低效实现** | 会用1000行代码写一个低效、臃肿、脆弱的实现，但其实100行就够 |
| **意外修改** | 会作为副作用修改/删除他们不喜欢或不太理解的注释和代码，即使与当前任务无关 |

### 关于Plan Mode

> “Plan mode 下情况会好一些，但还需要一个轻量级的 inline plan mode。”
> 

### 尝试过的解决方案

- 在 `CLAUDE.md` 中通过指令尝试修复上述问题
- 效果：尽管有这些简单尝试，问题仍然存在

### 尽管如此……

> “尽管有所有这些问题，它仍然是一个净正向的巨大改进，很难想象回到手动编码的日子。”
>

## Agent的韧性

### “感受AGI”的时刻

> “看着一个 agent 不知疲倦地工作在某件事上，真的很有趣。”
> 

**Agent vs 人类**：

| 特质 | Agent | 人类 |
| --- | --- | --- |
| 疲劳 | 永不疲劳 | 会累 |
| 士气 | 永不气馁 | 会沮丧 |
| 坚持 | 持续尝试 | 早就放弃”改天再战” |

### 典型场景

> “看着它为某件事挣扎很长时间，然后在30分钟后最终胜利——这是一个’感受AGI’的时刻。”
> 

### 核心洞察

> “你会意识到，耐力是工作的核心瓶颈，而有了 LLM，这个瓶颈被大幅提升了。”
>

## 加速还是扩展

### 难以量化的”加速”

Karpathy 认为很难衡量 LLM 辅助带来的”加速”效果：

> “当然，我确实感觉在原本要做的事情上快了很多。”
> 

### 主要效果：能力扩展

但**主要效果**不是加速，而是**做得更多**：

| 扩展类型 | 说明 |
| --- | --- |
| **值得做的事变多** | 以前不值得写的代码，现在可以写了 |
| **能做的事变多** | 以前因为知识/技能不足无法接触的代码，现在可以做了 |

### 结论

> “所以它肯定是加速，但可能更多的是一种扩展。”
>

## 杠杆效应

### LLM的核心优势

> “LLM 特别擅长循环直到达成特定目标，这就是大部分’感受AGI’魔力的来源。”
> 

### 编程范式转变

| 传统方式 | 推荐方式 |
| --- | --- |
| 告诉它**做什么** | 给它**成功标准**，然后看着它去做 |
| 命令式（Imperative） | 声明式（Declarative） |

### 获得杠杆效应的具体方法

1. **先写测试，再让它通过测试**
    - 让 Agent 先写测试
    - 然后让它写代码通过测试
2. **让它与浏览器 MCP 联动**
    - 把 Agent 放进浏览器交互循环中
3. **先写朴素算法，再优化**
    - 先写最简单、最可能正确的算法
    - 然后要求优化，同时保持正确性
4. **从命令式转向声明式**
    - 改变你的方法：从命令式到声明式
    - 让 Agent 循环更长时间，获得更大杠杆

## 编程乐趣的变化

### 出乎意料的发现

> “我没想到使用 Agent 后编程会变得更有趣。”
> 

**原因分析**：

| 被移除的部分 | 保留的部分 |
| --- | --- |
| 大量”填空式”的繁琐工作 | 创造性的部分 |

### 其他积极变化

| 变化 | 说明 |
| --- | --- |
| **更少被卡住** | 被卡住是不好玩的体验 |
| **更多勇气** | 因为几乎总有办法和 Agent 一起合作取得正向进展 |

### 观点分歧

> “我也看到其他人有相反的感受。”
> 

**LLM 编码将会分化工程师**：
- 一类人：主要喜欢**编码本身**
- 另一类人：主要喜欢**构建产品**

## 能力退化的隐忧

### 已经注意到的变化

> “我已经注意到，我手动写代码的能力正在慢慢退化。”
> 

### 认知层面的分析

| 能力类型 | 说明 | 受影响程度 |
| --- | --- | --- |
| **生成**（Generation） | 写代码 | 正在退化 |
| **鉴别**（Discrimination） | 读代码、审查代码 | 相对保留 |

**原因**：
> “主要是因为编程中涉及的各种语法细节，你可以很好地审查代码，即使你自己写起来有困难。”

## 总结

Andrej Karpathy 的这篇推文深刻反映了 AI 编程工具（尤其是 Claude Coding）对顶级工程师工作流的颠覆性影响：

1. **转变是真实的**：20年来最大的工作流变革，几周内完成
2. **但要谨慎**：模型仍有明显缺陷，需要人类监督
3. **核心价值**：不仅是加速，更是能力的扩展
4. **正确使用方式**：声明式而非命令式，给目标而非步骤
5. **个人权衡**：编程更有趣，但手动编码能力在退化
6. **行业预警**：2026年将面临 AI 生成的低质量内容泛滥
