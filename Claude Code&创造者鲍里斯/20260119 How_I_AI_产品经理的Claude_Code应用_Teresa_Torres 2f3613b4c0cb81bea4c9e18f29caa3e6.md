## Teresa Torres 背景介绍

Teresa Torres 是《Continuous Discovery Habits》的作者，这是一本产品界广为人知的书籍。她同时也是国际知名的演讲者和教练。

### AI 使用历程

> “我像大家一样，从网页版 ChatGPT 开始使用。后来我想找一个 LLM 来帮助写作，就逐渐转向了 Claude，因为 Claude 在写作方面稍微更好一些。”

Teresa 的编程背景：

- 没有版本控制系统
- 最近参与了一个需要将功能集成到真正生产级产品的项目，才开始使用 VS Code

### 转向 Claude Code 的契机

> “我丈夫说：’你可以直接在 VS Code 终端里运行 Claude。’然后，这真的改变了一切。”
> 

Teresa 在节目开始前直接在终端中导航电脑上的每个目录，展示了她对这套系统的深度整合。

## 为什么选择 Claude Code

### 核心理念：万物结对（Pair Everything）

> Teresa：“我认为工程师在使用 Claude Code 时，就是在和 Claude 进行结对编程。而我觉得结对编程的理念非常重要，我现在做所有事情都采用结对的方式，即使不是编程。就像结对任务管理、结对一切、结对所有事情。”
> 

### 从传统工具到 Claude 的转变原因

1. **数据锁定问题**：笔记被锁在 Trello 等第三方工具中，难以搜索和导出
2. **界面摩擦**：不想点击 14 个不同的按钮、不断变化的界面、日期选择器、标签选择等
3. **AI 辅助的思考方式**：每次执行任务时都会问自己”AI 如何帮助我完成这个任务？”

## 工作流一：个性化任务管理系统

### 系统概述

Teresa 构建了一个基于 Claude Code、Obsidian 和 Markdown 文件的个性化任务管理系统。

### 核心组件

### 1. `/today` 斜杠命令

每天早晨，Teresa 坐下来喝咖啡，输入 `/today` 命令：

> “我生活中的每个早晨，甚至包括周六周日，我都会坐下来喝杯咖啡，输入 ‘today’。”
> 

这个命令会：
- 查看 Trello 看板（与团队协作使用）
- 生成一个”今日”文件（在 Obsidian 中查看）
- 处理所有任务（Markdown 文件）
- 告知今天需要完成哪些任务
- 列出逾期任务
- 显示进行中的想法
- 包含每日研究摘要

### 2. 任务结构（Markdown 文件）

每个任务都是一个独立的 Markdown 文件，包含 YAML 前置内容（front matter）：

```yaml
---
type: task
due_date: 2024-08-21
tags:
- sales
- course-launch
---
```

任务文件结构：
- **类型**：task（任务）
- **截止日期**：due_date
- **标签**：自动由 Claude 添加

### 3. 自然语言创建任务

在演示中，Teresa 直接输入：

> “新任务，给 Claire 发送感谢邮件，今天完成。”
> 

Claude 自动：
- 在任务文件夹中创建新文件
- 设置截止日期为今天
- 添加到今日任务列表
- 尝试添加标签

### 为什么选择 Obsidian

1. **触感体验**：喜欢勾选复选框的感觉（虽然可以用 Markdown 标记 X，但感觉不同）
2. **文件浏览器**：左侧的文件浏览器方便管理
3. **多用途库**：不仅仅是任务，还包含：
    - LLM 背景资料
    - 笔记
    - 播客文件
    - 技能文件
    - 写作内容

### 高级搜索功能

> Teresa：“因为我在任务中记录所有笔记，即使过了一段时间我回来想找某个东西，我可以说：‘Claude，帮我找这个我不知道放在哪里的东西。’”
> 

与传统任务管理工具的搜索相比，Claude 的优势：
- 尝试各种搜索组合直到找到答案
- 能理解模糊描述
- 即使记错词也能找到正确内容

**示例**：
> “我说：‘我明天要发布新博客文章。’结果显示：’找不到叫”明天发布新博客文章”的任务，但我找到一个说”周三发布文章”的任务。这是你要找的吗？’”

### 自动标签系统

- Claude 负责所有标签工作
- 每次生成任务时都会考虑添加哪些标签
- 标签分类在项目级 `claude.md` 文件中管理
- 可以即时生成报告，如”我的销售管道现状如何？”

## 工作流二：自动化学术研究摘要

### 背景动机

> “我渴望成为一个学者，这很奇怪，但确实如此。我真的很想持续关注许多主题的学术研究。”
> 

Teresa 关注的研究主题：
- 合成用户研究（synthetic user research）
- 团队协作
- 创造力
- 探索
- 用户画像
- 教育资源

### 系统架构

### 1. 每日研究摘要

每天的待办事项列表中包含一个简短的研究摘要部分：
- 显示每日档案库（arXiv）搜索结果
- arXiv 是预印本服务器，大多数论文在正式发表前都会在此发布
- 优点：免费、快速、实时更新
- 缺点：未经编辑审核，需要自己筛选

### 2. 两个 Python 脚本（定时任务）

**脚本一：搜索脚本（每天早晨运行）**
- 搜索 arXiv
- 每周日搜索 Google Scholar
- 基于个人关键词配置文件搜索
- 记录已读论文，搜索最新论文

**脚本二：摘要脚本（每天晚上运行）**
- 查找研究目录中的新 PDF 文件
- 触发 Claude Code 代理生成摘要
- 将摘要添加到次日的研究文件

### 3. 论文保存流程

1. 浏览每日搜索结果
2. 如果论文有趣，下载 PDF
3. 保存到相应主题文件夹的 `sources` 目录
4. 次日收到详细摘要

### 摘要格式设计

> “我写了这个技能文章来专注于论文的方法论和效应量等内容，因为我必须充当编辑。这些论文目前尚未经过编辑。什么因素可以帮助我判断这本书是否值得阅读？”
> 

摘要关注点：
- 研究方法论
- 效应量
- 研究可靠性评估

### 实际案例

Teresa 分享了一个具体案例：

> “在我创建这个系统的那天，我在检查每日摘要时看到一篇关于购买意向的文章。我读了摘要，因为它的摘要格式很好，我意识到这个缺陷——他们使用的市场内调查并不十分可靠，不能准确测量购买意向。第二天我看到 Ethan Mollick 在 LinkedIn 上分享了这篇论文，我就说：’Ethan，这篇论文写得真的很差。’然后我写了一篇非常详细的 LinkedIn 文章，概述了对这项研究的批判性评价。这是我在 LinkedIn 上表现最好的文章之一。”
> 

### 开源项目

Teresa 将研究插件作为公开代码库发布：
- 目前仍在 Beta 阶段
- 只有一个用户（Teresa 自己）
- 她丈夫将成为第二个用户
- 欢迎第三个用户加入

## 工作流三：构建上下文文件库实现”懒人提示”

### 核心问题

> “最初，我把所有东西都放在我的 Claude MD 文件里。真的是所有东西都在 Claude MD 里。但后来我意识到，Claude 每次都会加载我的 Claude MD。我不想把所有这些上下文信息都放在那里。”
> 

**问题示例**：当问个人问题时（如”我的狗吃了这个安全吗？“），Claude 不需要知道营销渠道或博客文章存档。

### 解决方案：小型专注的上下文文件

### 1. 文件组织结构

创建一个专门的 Obsidian 库，命名为”LLM Context”：

```
LLM Context/
├── business/
│   ├── business_profile.md（索引文件）
│   ├── brand_guidelines.md
│   ├── marketing_channels.md
│   ├── products/
│   │   ├── course_1.md
│   │   └── course_2.md
│   └── partnerships.md
└── personal/
    └── personal_profile.md
```

### 2. 索引文件策略

- 业务简介（`business_profile.md`）作为索引
- 告诉 Claude：“你可以在这里找到公司概况，在那里找到这门课程的详情，在那边找到我的合作伙伴信息。”
- 让 Claude 根据任务需求智能加载特定上下文文件

### 3. 全局 Claude MD 配置

```markdown
如果我要求你帮助处理与我业务相关的事情，请使用我的业务简介。
如果我要求你帮助处理一些个人问题，请使用我的个人简介。
```

### 写作风格指南的创建过程

> “我首先告诉 Claude——我一开始没有告诉 Claude 我是谁——我只是说：‘去 product talk 看看，告诉我你觉得作者的写作风格是什么？目标受众是谁？写作理念是什么？语调是什么？’”
> 

Claude 访问博客并开始总结，然后 Teresa 进行修正：
> “我看了看说：‘嗯，差不多是这样。这个不对。让我们解决这个问题。’”

最终的写作风格指南：
- 非常详细
- 大部分由 Claude 完成
- 包含书籍写作与博客写作的区别
- 标题、副标题、常用短语的使用规范
- 必须使用和禁止使用的场景

### 上下文文件内容类型

1. **业务简介和个人简介**：大量业务背景
2. **市场营销**：目标受众、品牌定位、营销渠道
3. **内容架构**：内容资产、跟踪指标、发布计划
4. **产品信息**：所有课程、订阅产品等详情

### “懒人提示”的实现

> Teresa：“我可以这样说：’Claude，博客文章审核，给我一些反馈。’它会查看博客文章的主题，调出我的受众文件来了解对象是谁，以及提到的产品是什么，这真的很有帮助。”
> 

**比喻说明**：
> **Claire**：“就像你有一个文件柜，你在街上找了个随机实习生，告诉他’这是我的任务。如果你打开这个文件柜，你可以找到解决方案。’在这个文件柜上你放了使用说明。”

### 迭代构建方法

> “创建上下文文件的方式是：每当我结束与 Claude 的代码会话时，我会问：’Claude，你今天学到了什么应该被记录下来的东西？’然后我让 Claude 来做这件事。”
>

## Claude 作为写作伙伴

### 增强 vs 自动化

> Teresa：“我喜欢写作。所以我真正喜欢的是提出’增强’而不是’自动化’的问题。我不想自动化我的写作。”
> 

### 两个由 LLM 主导写作的案例

1. **Lovable 用户故事**：采访了 11 个人了解他们如何使用 Lovable，让 ChatGPT 将访谈记录转换成个人故事
2. **播客内容博客**：即将发布的博客文章基于刚发布的播客主题，让 Claude 负责撰写大部分内容

### 日常写作辅助方式

1. **实时研究验证**：
> “有时候我会想，我写了这个东西，然后我想知道这是不是真的。我就问：’Claude，这是我的想法。有没有证据表明这是真的？’Claude 会做研究，我继续写作。”
2. **开头吸引力优化**：
> “我会写一个开头，然后说：‘Claude，我写了这个开头。你能告诉我怎样让开头更有吸引力吗？’”
3. **章节审核**：
> “我会完成一个章节然后说：‘好的，Claude，请审核这个章节，看看什么写得好，什么写得不好。’”
4. **自动拼写纠正**：
> “我最喜欢的一件事是它会在我写作时自动纠正我的拼写错误。所以我可以非常随意地打字，不用担心拼写错误。”

### 写作风格指南的价值

> “有了这样详细的写作风格指南，Claude 的评论非常准确。因为它理解我的目标、我的读者、我的写作对象和我的写作风格。当它告诉我什么写得好，什么写得不好时，它是根据我自己的目标来判断的。”
>

## 闪电问答环节

### 问题一：除了 Claude Code，还使用什么 AI 工具？

**使用的工具**：
- **终端**：经常使用暗黑模式终端
- **VS Code**：写代码时仍偏好在 IDE 中使用颜色差异显示功能
- **ChatGPT**：偶尔在浏览器中使用，通常是因为做其他事情时方便切换
- **Descript**：视频编辑工具，最喜欢的非基础 AI 产品
> “从以前必须手动编辑视频到现在直接编辑文本转录，简直是世界上最神奇的事情。”

**关于 Cursor**：
> “大家都问我，Cursor 呢？我实际上从未使用过。我知道它很棒。我处理大量信息涌入的方式是：只有当我使用的产品出现问题时，我才会去寻找新产品。”

### 问题二：希望看到哪些数据源接入 AI 工具？

**首选**：LinkedIn
> “我只是讨厌 AI 生成的内容。我猜这就是为什么我还坚持自己写作，因为阅读别人的 AI 生成内容评论让我有点难过。”

**痛点**：
- LinkedIn 的 API 访问非常困难
- 需要能够在不登录的情况下在终端阅读内容
- 愿意接受内联广告

**Claire 的呼吁**：
> “我们已经申请了 LinkedIn API MCP。我们也很乐意接受广告。LinkedIn 产品经理们，只要我们不需要登录就能在终端上阅读内容，我们不介意运行内联广告。”

### 问题三：还想要什么功能？

**图像中的文字嵌入**：
> “我真的很想要在图像中嵌入文字的能力，就像他们可以为图像中引用的文字关闭引号一样。”

### 问题四：当 Claude 不听话时怎么办？

> Teresa：“我会杀掉 Claude。我过度使用清除功能。”
> 

**应对策略**：
1. 专注于上下文文件，不依赖对话历史
2. 当 Claude 卡住时，想要一个干净的开始
3. 不想重新向 Claude 解释所有背景信息
4. 使用 `/clear` 命令重新开始

**Teresa 的幽默感慨**：
> “我希望和人交流时也能这样做。就好像我们已经有了所有需要的信息，但那并没有让我们达成一致。让我们清除所有设置，重新开始。”

### 关于搜索能力

> 

### 关于上下文文件策略

> Teresa：“如果我们给它太多无关的上下文，它仍然做不好工作。我不想把所有产品信息放在一个文件里，因为如果我们在开发某个产品，它不需要知道其他产品的信息。”
> 

### 关于懒人创建上下文

> Teresa：“创建上下文文件的方式是：每当我结束与 Claude 的代码会话时，我会问：’Claude，你今天学到了什么应该被记录下来的东西？’我让 Claude 来做这件事。”
> 

### 关于 AI 写作的态度

> Teresa：“我喜欢写作。所以我真正喜欢的是提出’增强’而不是’自动化’的问题。我不想自动化我的写作。”
>
