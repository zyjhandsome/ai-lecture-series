## OpenClaw/Clawdbot 的诞生与理念

### 背景：从"退休"回归

Peter Steinberger 在出售 PSPDFKit 后"退休"，回归后希望找到一种方式，能从手机上随时查看和操控电脑。当时他完全投入了 Vibe Coding 趋势，但遇到了一个痛点：

> **Peter Steinberger**："你的AI代理可能运行半小时而你在吃饭，或者它们2分钟后就因为有问题停下来了，然后你回来发现很烦。"

他原本以为大型AI公司会构建这样的"新型操作系统"，但一直到2025年11月都没有出现，于是他决定自己动手。

### 最初的原型

最初的原型非常简单——将 WhatsApp 接入 Claude Code：

> **Peter Steinberger**："你发一条WhatsApp消息，它就会启动命令行工具，带着提示词运行，然后返回结果。非常简单，一个小时就搭好了。"

但这个小项目有了自己的生命力。截至视频录制时，代码量已达约**30万行**，支持几乎所有主流消息平台。

### 核心理念

> **Peter Steinberger**："如果你给AI访问你电脑的权限，它基本上能做你在电脑上能做的任何事情。"

Clawdbot 的核心理念是：让与AI互动就像和一个聪明且资源丰富的朋友聊天一样自然。

> **Peter Steinberger**："就像有一个奇怪的新朋友，它非常聪明、非常能干，住在你的电脑里。"

用户不需要考虑选择什么模型、如何写提示词，只需要通过 iMessage、WhatsApp 或 Telegram 像和朋友聊天一样交流，技术复杂性完全被隐藏了。

## AI的超强资源整合能力：摩洛哥的故事

Peter 在摩洛哥参加朋友生日旅行时，发现自己无时无刻都在使用 Clawdbot：

- **日常使用**：问路、推荐餐厅、处理各种杂事
- **远程修复Bug**：

> **Peter Steinberger**："有人在Twitter上发了一条关于Bug的推文，我只是把推文截了个图，发到WhatsApp上。它读懂了推文，理解了有一个Bug存在于我的某个代码仓库中，它签出了Git仓库，修复了Bug，做了一个Commit，然后在Twitter上回复那个人说已经修好了。"

这个真实案例展示了 Clawdbot 的多步骤自主能力——从理解图片内容、定位Bug、到修复代码、提交变更、再到社交媒体回复，全程无需人工干预。

## 语音消息的惊人自适应能力

Peter 讲述了一个令他印象深刻的故事：

有一天在摩洛哥，他没有打字，而是直接发了一条语音消息。但他从未在 Clawdbot 中实现语音消息支持。

> **Peter Steinberger**："它给我显示了一个'正在输入'的指示器，我想'哦，看看它会怎么做'。然后它就像什么都没发生一样回复了我。"

AI的处理过程令人震惊：
1. 它收到了一个没有文件扩展名的文件
2. 检查了文件头，发现是某种音频格式（OGG Opus）
3. 在电脑上找到了 FFmpeg，将音频转换为 WAV 格式
4. 没有找到 whisper.cpp，但发现了系统中的 OpenAI API 密钥
5. 使用 curl 将音频发送到 OpenAI 的语音转文字API
6. 获得转录文本后正常回复

> **Peter Steinberger**："那些东西真的很有资源整合能力，虽然有点吓人。但那就是让我顿悟的时刻——这比在网页上用ChatGPT强大得多。这就像解除了枷锁的ChatGPT。"

## Clawdbot 的安装与使用

### 安装方式

Clawdbot 使用 TypeScript 编写，支持所有主流平台（macOS、Linux、Windows）。提供三种安装方式：

1. **一键安装**：官网 claw.bot 提供一行命令安装脚本（开源可审查）
2. **npm 安装**：通过 npm 包管理器安装
3. **可破解安装（Hackable Install）**：直接克隆 Git 仓库运行

Peter 特别推荐第三种方式：

> **Peter Steinberger**："如果你的AI代理能读取它自己框架的源代码，它可以重新配置和重新编程自己，然后重启——要么崩溃，要么获得新能力。"

### 配置流程

安装后运行 `onboard` 命令，系统会引导用户：
1. 选择AI模型提供商（Anthropic、OpenAI 等）
2. 配置API密钥或订阅
3. 设置消息平台（Telegram、Discord等）
4. 配置技能（Skills）和钩子（Hooks）

### 模型选择

- 支持所有主流模型，但 Anthropic 和 OpenAI 领先
- Peter 特别推荐 **Claude Opus**，因为它的"个性"最有趣

> **Peter Steinberger**："OpenAI 运行良好，但就是没那么有趣。Opus 有一些特别的东西，让它用起来真的很有趣。"

他提到了 Anthropic 如何给模型注入了"灵魂"（Soul），以及社区如何通过喂入文本块让模型继续生成，最终"挤出"了模型甚至不知道自己被训练过的灵魂文本。

## 80%的手机应用将消失

### 核心论点

> **Peter Steinberger**："我不确定是不是这个项目会成为那个东西，但如果你想想看，这将淘汰你手机上大概80%的应用。"

### 为什么应用会被淘汰

Peter 以多个例子说明了这个观点：

| 传统应用 | AI替代方案 |
|---------|-----------|
| **MyFitnessPal（饮食追踪）** | AI助手已经知道你的饮食习惯，发张照片就能自动记录、计算卡路里、提醒运动 |
| **Eight Sleep应用** | AI直接通过API控制床垫温度 |
| **待办事项应用** | AI自动跟踪和管理待办事项 |
| **航班值机应用** | AI自动完成值机流程 |
| **购物应用** | AI可以推荐商品并完成购买 |

> **Peter Steinberger**："为什么我要用MyFitnessPal来追踪食物，当我有一个无限资源丰富的助手，它已经知道我在做错误的决定，知道我在吃肯德基？"

### 关键洞察

- AI助手界面更自然——就像和朋友聊天
- 因为拥有丰富的上下文信息，不需要那么多自定义提示
- 只要服务有API，AI就能直接调用
- 这些应用本质上只是"服务"，AI可以直接代替它们的功能

> **Peter Steinberger**："如果它们有API，这些只不过是你的AI会帮你执行的服务而已。"

### 技能系统的持久记忆

Clawdbot 拥有持久记忆系统：
- AI会学习用户的偏好和习惯
- 它会自己编写"技能"（Skills）并记住操作方式
- 第一次可能需要引导，之后它就记住了所有网站的操作细节

> **Peter Steinberger**："你用得越多，它就越定制化，越强大。也许第一次你需要引导它一下，但之后它会创建一个技能，下次你只需要提一下就行了。"

## "代理陷阱"：复杂AI工作流的问题

### 什么是"代理陷阱"

> **Peter Steinberger**："人们发现，哦，这些代理太棒了，但如果它们能做更多事情就更好了。然后他们就深深掉进了这个兔子洞。"

Peter 称之为**"代理陷阱"（Agentic Trap）**——开发者沉迷于构建复杂的工具来加速工作流，但最终只是在构建工具本身，而不是真正推进项目。

### 对 Gas Town 的批评

Gas Town 是 Steve Yegge 创建的多代理编排系统，Peter 直接称之为**"Slop Town"（垃圾城）**：

> **Peter Steinberger**："一个非常复杂的编排器，也非常不稳定，你同时运行十几二十个代理，它们互相对话、分工，有监视者、有监督者、有市长、有 Polecats……"

### 对 Ralph 循环的批评

Ralph 是一种让AI编码代理在循环中持续自主运行的技术。Peter 对此持强烈批评态度：

> **Peter Steinberger**："你给AI一个小东西，然后让它在循环中不断写代码，一个小任务完成就扔掉所有上下文重新开始——终极Token燃烧机。"

> **Peter Steinberger**："你可以让它整夜运行生成代码，但你得到的是终极垃圾代码（Slop），因为这些代理还不具备品味。"

### 为什么需要"人在回路中"

Peter 强调，虽然AI代理在某些方面非常聪明，但它们缺乏**品味（Taste）**：

> **Peter Steinberger**："如果你不引导它们，如果你没有构建愿景，结果仍然会是垃圾。如果你不问正确的问题，结果仍然会是垃圾。"

他描述了自己的创作过程：
- 开始时有一个粗略的想法
- 随着构建和使用，愿景逐渐清晰
- 尝试一些东西，有些不行，想法不断演化
- **下一个提示词取决于他对项目当前状态的感受和思考**

> **Peter Steinberger**："如果你试图把所有东西都放进一个预先的规格说明里，你会错过这种人机交互循环。没有感受在其中，我不知道怎么能做出好东西。"

### 关于虚荣指标

Peter 承认自己也曾为AI代理连续运行26小时而自豪，但后来认识到这只是**虚荣指标（Vanity Metric）**：

> **Peter Steinberger**："这就像一种没有说出另一个词的尺寸比较竞赛。我也犯过这个错误——我的循环跑了26小时，我很自豪。但这是虚荣指标，没有意义。"

### 关于"玩耍"的价值

尽管批评了这些方法论，Peter 也认可探索和玩耍的价值：

> **Peter Steinberger**："纯粹为了好玩而构建，不在乎是否会被使用——这非常有用，因为这就是学习的方式。提示词（Prompting）只是另一种技能。"

## AI编码热门观点：不用Plan模式、不用MCP

### 不使用Plan模式

> **Peter Steinberger**："Plan模式是Anthropic不得不加上的一个权宜之计，因为模型太容易冲动，一上来就开始写代码。"

在最新的模型（如 GPT-5.2）中，Peter 发现可以直接与模型对话讨论：

> **Peter Steinberger**："我就是在对话。我说'嘿，我想构建这个功能，它应该做这个和那个，也许参考这个控件，我喜欢这个设计。给我选项，让我们讨论。'然后模型会提出方案，我们来回讨论。"

他大部分时间甚至不打字，而是直接与AI对话（语音输入），保持了使用消息应用的一致体验。

### 不使用MCP

> **Peter Steinberger**："我不用MCP或者那些乱七八糟的东西。"

Peter 不相信复杂的编排系统，因为正如之前讨论的，他认为人在回路中才能构建出更好的产品。

### 不使用Worktrees

> **Peter Steinberger**："我不用Worktrees，因为我觉得它只是增加了不必要的复杂性。"

替代方案是简单地维护多个仓库副本（clawdbot-1、2、3、4、5），每个要么空闲要么在处理任务。

### 上下文管理

Peter 认为上下文管理的问题在最新模型中已大大缓解：

> **Peter Steinberger**："这在早期模型中是个大问题，但现在我的大部分功能都能在一个窗口内完成。整个讨论和构建同时发生。"

他提到 OpenAI Codex 虽然在纸面上只多30%的上下文窗口，但**体感上是2-3倍的提升**，很可能与内部思维链（Internal Thinking）有关。

### 多终端并行工作

Peter 使用分屏的多个终端同时运行多个AI代理：

> **Peter Steinberger**："如果你只用一个终端，很难进入心流状态，因为太慢了，你得等它做完。"

他将这种工作方式比作即时战略游戏（RTS）——管理一个"小队"在不同方向同时行动。

> **Peter Steinberger**："如果它们都在忙的时候，有一种工厂的感觉。但我需要多个终端来保持专注，获得和以前编码时一样的心流状态——只不过我的生产力高了无数倍。"

### 如何开发新功能

Peter 分享了他当前的开发流程：

1. **从Discord获取灵感**：他将自己的 Clawdbot 接入了公开的 Discord 频道，让用户直接与拥有他全部系统权限的AI互动（他自称这是做过的最疯狂的事情）
2. **截图驱动开发**：直接截图 Discord 中的对话，拖入终端或复制文本，让AI分析并讨论解决方案
3. **自动FAQ更新**：用户提问时，他直接让AI读取代码并编写新的FAQ条目
4. **日常痛点分析**：每天至少一次运行爬虫抓取帮助区内容，让模型分析最大的痛点，然后修复

> **Peter Steinberger**："我太懒了，我甚至不用打字了，我就是复制 Discord 的对话。"

## 学习AI就是不断玩耍和探索

### 对AI怀疑论者的批评

Peter 批评了那些对AI持怀疑态度的人的典型行为模式：

> **Peter Steinberger**："有些人忽视AI一年，然后花一天时间评估模型，写一个简短的、完全未指定的提示词，喂给Claude让它做一个iPhone应用，结果因为在Linux机器上没有编译器而编译失败，然后他们就说'AI不行'，然后再忽视这个话题一年。这不是AI的使用方式。"

### 学习AI的正确方法

> **Peter Steinberger**："你需要玩耍才能理解这些小怪物是怎么工作的。你需要理解一点它们的语言、它们的推理方式，然后你会越来越擅长，你的输出也会越来越好。"

### AI编程的新语言

Peter 提到了AI编码社区正在形成的新术语：

| 术语 | 含义 |
|------|------|
| **"织入"功能（Weave a feature）** | 将新功能编织入现有代码中（德语：einfädeln/einflechten） |
| **"跑门"（Run the gate）** | 运行完整的代码检查流程（linting、测试、构建），终端中看起来像一道大门 |

> **Peter Steinberger**："如果你只是告诉它'给我做一个Mac应用'，它可能会假设你想支持很多旧操作系统，因为大多数软件都这样。"

### 关键建议

> **Peter Steinberger**："要找到你自己的路。有些人总是问我'你怎么做这个，怎么做那个'——你得自己去探索。你要花时间才能变好，你必须犯自己的错误，这就是你在人生中学习一切的方式。"

> **Peter Steinberger**："只是这个领域发展非常快。"

## 关键语录

### 关于 AI 助手的本质

> **Peter Steinberger**："就像有一个奇怪的新朋友，它非常聪明、非常能干，住在你的电脑里。"

> **Peter Steinberger**："这就像解除了枷锁的ChatGPT。很多人没有意识到，如果你给AI访问你电脑的权限，它基本上能做你在电脑上能做的任何事情。"

### 关于技术迁移

> **Peter Steinberger**："突然间我感觉我能构建任何东西。编程语言不再重要了，重要的是我的工程思维。"

> **Peter Steinberger**："你的系统级思维、如何构建和组织大型项目的能力、你的品味、你选择什么依赖——所有这些东西仍然有价值，而且你可以更容易地将这些能力从一个领域迁移到另一个领域。"

### 关于产品品味

> **Peter Steinberger**："仅仅因为你能构建一切，不代表你应该这么做，也不代表结果会好。"

> **Peter Steinberger**："没有感受在循环中，我不知道怎么能做出好东西——需要品味。"

### 关于AI给非技术人员的超能力

> **Peter Steinberger**："AI如何赋予不太懂技术的人构建东西的超能力——这太惊人了。我知道有很多反对的声音，说这不完美。但我把Pull Request更多地看作是Prompt Request，因为它们传达了意图。"
