## 开发者时间分配：编码只是冰山一角

**Joel** 在与大量企业客户高管的交流中反复展示一张核心图示——软件开发者的时间到底花在哪里？

- 如果只把 GitHub Copilot 当作编码工具，它覆盖的实际只是开发者工作的一小部分
- 即使编码速度提升 50%（将两天的任务缩短为一天），由于产品规划（"漏斗前端"）和发布流程（"漏斗后端"）没有变化，功能的实际交付速度并不会真正加快
- 开发者会感觉"做了很多事"，但整体来看瓶颈只是转移了——其中最大的瓶颈之一就是**与产品团队的协作**

> **核心问题**：如何利用 GitHub Copilot 和 Agent 系统来革新产品管理流程本身？

## Ask Questions 工具：让 AI 成为真正的结对伙伴

**Harald** 分享了他在 VS Code 1.109 版本中主导开发的 **Ask Questions 工具**——这是 Agent Skills 投资中的一个重要方向。

### 背景与动机

- 很多用户对 AI 的使用模式是"帮我做这个、帮我做那个"，忽略了 AI 在**协作思考**方面的能力
- 真正的 AI 协作应该是：你给我负面反馈、审视我的代码、挑战我的想法
- 此前的 Plan Mode 有一个"进一步考虑"部分会提出问题，但回答起来很笨重——用户需要回到对话中逐条回复

### 开发过程

- **初始原型仅花了几个小时**——Harald 作为 Plan Mode 的深度用户，对 QuickPick（VS Code 的轻量级 UI 组件）非常熟悉
- 原型使用了基础的 QuickPick 界面（VS Code 标志性的轻量交互方式），作为 V0 版本
- 放到团队内部试用后，第二天就有人发现并惊叹"这个工具太酷了，居然会问我问题"
- 社区反馈推动了 UI 的进一步打磨，从简单的 QuickPick 演进为更精美的界面

### 与 Claude 集成的交集

- VS Code 同期发布了 **Claude Agent 集成**，其中 Claude 也有自己的 Ask Questions 工具
- 两个工具有一定重叠，但 VS Code 团队希望保持独立创新空间，同时相互学习

### 关键理念

- 这个功能从**发现需求 → 原型 → 团队试用 → 社区反馈 → 迭代优化**，体现了 AI 时代 PM 的工作方式
- 目前还在运行实验，验证该工具是否被过度使用——有些用户反而禁用了它，因为他们更喜欢纯自然语言对话

## PM 如何用 AI 快速原型验证

### 从瀑布式规划到快速迭代

**Harald** 指出，很多人在产品管理中过于纠结于路线图规划和待办事项清单（Backlog），但 PM 的核心工作其实是**拥有问题本身**：

- 提出假设和假定
- 设计验证方式
- Backlog 只是所有学习的**输出结果**

### AI 改变了什么

- **假设验证变得更快**：可以跳过大量规划，直接进入原型验证
- **PRD 成为活文档**：过去的产品需求文档写完就过时了，但在 AI 时代，这些文档变成了 Agent 的宝贵上下文——记录你的思考、尝试和学习
- **交互式规格说明取代长篇 PRD**：团队更需要能驱动讨论和学习的可交互产物，而非需要反复阅读的长文档

> "过去写的 PRD，你写完一次就过时了。但现在有了 AI，这些文档真的变成了活文档，你可以持续更新——它们是 Agent 的绝佳上下文。"
—— **Harald Kirschner**

## Product Brain：产品知识的中枢大脑

**Harald** 展示了他开源的 **Product Brain** 仓库（[github.com/digitarald/product-brain](https://digitarald.github.io/product-brain/)），这是他探索的"产品大脑"概念。

### 核心理念

Product Brain 是一个**完全自定义的 GitHub 仓库**，包含：

| 组成部分 | 说明 |
| --- | --- |
| **团队工作方式说明** | PM 团队如何工作、什么是重要的 |
| **模板系统** | 规格说明（Spec）的标准模板——问题陈述、解决方案、成功标准、开放问题、假设 |
| **快捷 Prompt** | 反馈综合、会议准备、创建 Issue 等日常任务的快速提示词 |
| **自定义 Agent** | 探索 Agent、数据分析 Agent、设计伙伴 Agent 等 |
| **战略文档** | 路线图、战略愿景、优先级框架 |
| **设计系统** | Agent 创建 Mockup 时遵循的设计规范 |

### 关于存储位置的思考

- 目前很多人过度依赖"Markdown 文件放在仓库里"——因为这是所有 AI Agent 最容易处理的
- 未来可能演进到 Notion、Jira、GitHub Issues、**GitHub Wikis**（已经是很好的 Markdown 结构化存储）
- 关键是通过 **MCP（Model Context Protocol）** 实现正确的认证和读写权限
- 可以是工程仓库的"侧车"（Sidecar），也可以让 Agent 直接更新 Jira Issues

### 关键洞察：会议的新价值

- 不再需要在会议中做笔记——直接将会议转录带回产品规格说明
- 所有讨论、重新表述、客户反馈都会被 AI 捕获
- 会议因此变得比以前更有价值——所有细微差别都被保留下来

## Sprint Planning 自动化演示

**Joel** 现场演示了他用 VS Code Agent 构建的 Sprint Planning 自动化工具。

### 工作流程

1. **调用 Work IQ MCP Server**：用自然语言指令，让 Agent 查看过去两周所有团队频道、会议记录、笔记
2. **发现依赖关系**：自动找出任务之间的关联，更新 Issue 标签——这些是工程师最不想做的事
3. **暂停思考**：Agent 像工程师一样暂停，遵循产品思维流程
4. **市场分析**：在执行之前先做简要的市场分析——搜索互联网，验证假设，确认是否有市场
5. **生成规格说明与实施计划**：如果验证通过，自动生成规格文档和实施计划
6. **更新 Backlog 并创建 PR**：将结果反映到 GitHub Issues 中，并开始第一个 PR

### 演示亮点

- Joel 以一个有趣的示例项目"为 Commodore 64 构建 Halo 游戏"来展示整个流程
- Agent 自动完成了市场分析报告，包括：执行摘要、市场规模可视化、目标用户画像、营销模型
- 整个过程从会议笔记到有形产出只花了**约 10 分钟**

> "我不是产品经理，我是工程师。但用今天的工具，10 分钟就能把一次对话变成有形的产品文档和可追溯的 Backlog。"
—— **Joel Norman**

## GitHub 研究 Agent 与自定义 Agent 体系

**Harald** 展示了他的 **GitHub 研究 Agent**——一个用于查询 GitHub Issues 和社区反馈的自定义 Agent。

### 工作原理

- 在 VS Code 的 Agent Picker 中选择该 Agent，提出如"VS Code 最受欢迎的功能请求是什么？为什么？"的问题
- Agent 配置了特定的工具集：可以读取代码库、搜索 GitHub Issues
- 使用更强大的推理模型（如 Opus）处理复杂查询
- 并行执行多个搜索，然后汇总报告

### 自定义 Agent 的独特之处

- 每个 Agent 都有一个 **description** 字段，说明何时以及如何使用它
- Agent 在 **agents 列表**中注册，意味着**主 Agent 可以将其作为子 Agent 调用**
- 这避免了上下文污染——研究查询在独立的子 Agent 中完成，使用更快的模型，然后将结果报告回来

### 系统设计

> "这就像一个新的系统——你可以用自定义 Agent 构建，而且就在刚发布的版本中。"

## 数据分析民主化：Jupyter Notebook + AI

**Harald** 展示了用 **Data Analyst Agent** 进行客户 NPS 评分分析的演示。

### 核心观点

- 数据分析是每个 PM 的核心技能，但数据分析团队通常被大型研究项目压得喘不过气来
- 需要**民主化数据访问**——让每个人都能提出和回答数据问题，而不需要等待数据科学家或 PM 的排期

### 工作方式

- Data Analyst Agent 可以处理 Jupyter Notebook
- 它不只是简单地查看数据——**它像数据科学家一样工作**：
    1. 创建一个 Cell 并运行
    2. 查看输出结果
    3. 推理下一步应该做什么样的数据科学操作
    4. 继续深入分析
- 输出结果包括：客户分布、可持续性关键词分析（11% 的客户对可持续性感兴趣）、人口统计分析等
- **Notebook 是可验证的产物**——团队可以查看每一步分析，指出"这里的相关性分析完全错误"，建立对 AI 分析能力的信任

## Diverge 模式：并行生成多个原型方案

**Harald** 演示了 **Diverge（发散）** 模式——面对"空白画布恐惧症"时的解药。

### 场景

Contoso 公司（一个户外装备电商）要为产品添加可持续性仪表板，但这是一个复杂且模糊的问题——不知道从哪里开始。

### 工作流程

1. 将已有的产品规格说明作为上下文输入
2. 使用 Diverge Agent 生成多个设计方向
3. Agent 提出了三个方向：
    - **游戏化可持续性**：将可持续性行为变成游戏
    - **数据透明度**：展示更多可信度信息
    - **社交社区**：围绕可持续性建立社区
4. 每个方向自动创建一个 **GitHub PR**，由 [GitHub.com](http://github.com/) 上的 Coding Agent 执行

### 丰富性思维

- 不只是 UX 问题——工程架构也可以同样操作：**同时派出三个 Agent 做三种架构方案**
- 可以将所有进来的客户 Issue 自动分配给 Agent 尝试修复——即使 Issue 写得很差，Agent 可能也有 50% 的成功率
- 关键是用"丰富"替代"稀缺"——更主动探索、更快学习

## 产品与工程的融合循环

**Harald** 强调了他认为释放团队速度的**最关键要素**：

### 传统模式的问题

- 产品有产品的循环，工程有工程的循环
- 两者之间有"交接"——PM 写 Jira 卡片扔给工程师，工程师估点

### 理想模式

- **每日 Mob Planning**：团队每天聚在一起讨论今天做了什么、Backlog 中有什么、收到了什么反馈
- **会议中即时原型**：在讨论中就开始原型验证
- **同一上下文工作**：产品研究、用户反馈、技术实现都在同一个空间中进行
- 当某件事验证通过，可以**立即开始提交 PR** 来交付功能

> "不应该是这边一个产品循环，那边一个工程循环。它们应该交织在一起，所有事情同时发生。这是释放速度的关键。"
—— **Harald Kirschner**

## 给 CTO 的建议：大型遗留代码库如何拥抱 AI

**Joel** 提出了一个来自客户的高频问题：作为拥有 200 万行代码的 25 年历史公司 CTO，应该先做什么？

### Harald 的多层建议

### 1. 培养 AI Champion

- 创造空间让团队成员尝试 AI
- 从**绿地项目**（Green Field）开始——可能是一个代码质量追踪工具，或者团队一直需要但没人做的小工具
- 目标不是代码质量，而是让人**建立与 Agent 协作的"手感"**：如何规划、如何给反馈、如何迭代

### 2. 投入 Context Engineering

- 理解 Agent 成功所需的条件
- 逐步引入 [agents.md](http://agents.md/)、指令文件、工作流映射
- 不要只做一次——需要有**持续推动者**（Champion）不断演进

### 3. 不需要立即重构整个代码库

- 最初的假设是"扔掉旧代码重写"，但实际上成本太高
- 更好的策略：
    - 用 Agent 更好地**文档化**现有代码
    - **逆向生成规格说明**（Reverse Spec）——让 AI 帮助理解代码库中实际有什么
    - 逐步拆解，就像工程师手动拆解一样，但用 Agent 加速

### 4. 将产品规划纳入代码库

- 让所有规划都**扎根于代码库**，不要在外部独立进行
- Agent 可以同时访问代码和产品文档，做出更好的权衡

### 5. 善用自动化辅助

- GitHub 正在开发 **Memory 功能**——Agent 会从代码审查中学习，避免重复错误
- VS Code 的 `/init` 命令和类似 **Primer** 的工具可以自动分析仓库并设置 AI 配置
