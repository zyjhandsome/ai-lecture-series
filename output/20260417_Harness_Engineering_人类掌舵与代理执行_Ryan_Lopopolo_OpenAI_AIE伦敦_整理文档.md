## 全局核心提要（Executive Summary）

> **AI 导读**：
> - **稀缺资源已变**：实现（implementation）不再是软件工程的主要瓶颈；在「代码近乎免费、可并行生成与重构」的前提下，**人类时间与注意力**、**人机注意力分配**，以及 **模型上下文窗口** 成为真正的硬约束。
> - **组织动作**：把工程角色上移到 **系统设计、任务分解、委托与编排**；每位工程师都更像 **Staff Engineer**，同时驱动多名（或大量）代理并行工作。
> - **Harness（驾驭层）是什么**：不仅是模型，更是 **在正确时间把正确文本指令喂给模型** 的一整套仓库与环境机制——文档、Skills、Lint/测试失败信息、评审代理、CI 中的检查，都是「对模型的持续提示注入」。
> - **反模式**：巨型单文件规则、前置堆满指令、无验收标准的「计划模式」、以及需要人频繁输入 continue——后者被讲者明确视为 **驾驭层没有给足上下文** 的失败信号。
> - **可操作的工程原则**：把失败模式 **类目化** 后 **系统性消灭**；用 **可执行的 guardrails**（定制 Lint、结构测试、评审代理）把团队的「品味与非功能需求」固化；通过 **仓库分区/包隐私/一致性** 降低 token 与注意力成本。

## 主题演讲：从「代码稀缺」到「注意力稀缺」

### 「代币亿万富翁」与九个月「只通过代理写代码」的组织实验

**核心观点**：讲者以半开玩笑的「**token billionaire（代币亿万富翁）**」自称，主张通往 AGI 的路径需要社会层面让更多人把模型用到「**把整份工作做完**」的强度；其团队通过 **禁止成员直接上手传统编辑器**、强制 **经由模型完成工作**，把这一理念推到操作极限。

**要点提炼**：

- **代理已能在更长时域、更高复杂度任务上更可靠地行动**；**驾驭层（harness）** 与模型能力同步演进。
- **实现不再是软件工程工作的稀缺资源**；讲者直言 **「Code is free（代码是免费的）」**——产出、重构、删除代码的成本结构已变。
- 「雇佣键盘前的手」主要受 **GPU 容量与 token 预算** 约束；每位工程师理论上拥有 **5、50 或 5000 名工程师量级** 的并行容量（比喻）。
- 关键不再是「怎么写」，而是 **如何把这份容量有效部署进代码库与团队流程**。

**详细内容**：

讲者描述近六个月 **编码代理（coding agents）** 能力快速跃迁，模型在驾驭层内可执行更复杂动作、在更长任务上更可靠。**软件构建方式已变**：当实现不再稀缺，技能栈更偏向 **系统思维、系统设计与委托（delegation）**。

讲者提出一个「**AGI pill（AGI 药丸）**」式信念：**相信模型能够生成你们永远需要的每一行代码**，并决定何时删除、重构或提升可靠性；工程师的职责转为 **解除阻塞**，让人类与代理在 **长周期任务** 上能把工作 **从头到尾做完**。

在旧世界里，若 **人类时间是瓶颈** 且代码主要靠人写，工作会退化为 **只有 P0 / P2**，**P3 永远做不完**。在新世界里，若 **代码免费且可大量并行**，**P3 可以立刻并行开四条线**，选一条能解决问题的合并进去。

讲者举例：在 OpenAI 内部做了大量代理以提升同事效率；**代码免费**后，内部工具可以 **从第一天起就做好本地化/国际化**，让伦敦、都柏林、巴黎、布鲁塞尔、苏黎世、慕尼黑等地同事用母语使用工具，而不必在「团队容量」上做痛苦权衡。

### 「每个人都是 Staff Engineer」与「让事物保持一致」

**核心观点**：工程师要把时间视野拉到 **一天、一周、六个月**，预先设计 **结构、流程与代码空间**，以驾驭近乎无限的代码产出能力。

**要点提炼**：

- **一致性**降低模型每次任务需要「激活」的规则数量，使 **token 消耗更可预测**。
- **大规模重构**在新世界被描述为「近乎免费」；**迁迁移不再应拖半年**——可以并行启动多名代理收尾。
- 代码之所以过去被视为负担，是因为它消耗 **人类同步注意力**；模型则被描述为 **极有耐心、可无限并行**，从而改变维护与删除的决策约束。

**详细内容**：

讲者强调：**重要的不是代码本身，而是把它带到那里的 prompt 与 guardrails（护栏）**。因此 **面包屑路径**、文档、**ADR（架构决策记录）**、面向角色的说明、工单与 Code Review 历史，本质上定义了「何为好工作」；这些必须对代理 **可读、可检索、可执行」，而不仅是给人看。

**代码即文本即提示**：仓库里的代码与文档，会持续作为 **上下文提示** 影响代理行为；因此要想办法 **让事物尽可能相同**（同一套模式、同一套工具链入口），减少模型为理解差异而支付的注意力。

## 三大稀缺资源与优先级世界的改写

### 人类时间、注意力与上下文窗口

**核心观点**：当前世界的三类稀缺是 **人类时间**、**人类与模型的注意力**、以及 **模型上下文窗口**；工程组织应系统性地 **自动化低杠杆同步时间**，把人类时间挪到更高杠杆活动。

**要点提炼**：

- 当 **P3 级需求** 也能并行尝试，产品/工程策略会从「永远不做」转向 **快速并行探索与取舍**。
- 「模型与人类在注意力经济学上不同」：模型适合承担繁琐、并行、反复的代码维护动作。

**详细内容**：

讲者认为应默认：**软件工程里那些最好的实践**，应当 **始终可及**——不是奢侈品，而是基线假设。实现细节可以更多交给代理，人类更关注 **系统与结构**。

## 文档、ADR 与「可执行品味」：让代理做对「未写明的非功能需求」

### 好工作 = 成千上万次「未写明的非功能选择」

**核心观点**：写出可维护、可扩展、可协作的高质量代码，依赖大量 **未被规格逐条写清的非功能需求（non-functional requirements）**；模型在训练里见过 **海量代码与各种取舍**，因此 **关键在人类团队把这些取舍显式化**，否则代理会在无数「可行但不符合你团队品味」的路径中走偏。

**要点提炼**：

- **「别产出 slop（低质凑数代码）」** 不是口号，而要落到 **可重复的约束**：不接受，就 **不会进入代码库**。
- 这需要愿意承受 **短期速度损失**：停下来 **复盘代理为何在你的环境里失败**，补齐文档、Lint、测试与评审代理，再恢复高速。
- **多样化专家人格（persona）** 仍重要：前端架构、后端扩展性、产品意识等；让专家把「何为好工作」写下来，可使 **每一名驱动代理的工程师都复用全队的最高标准**。

**详细内容**：

讲者指出：不必再用 **低信号的人工 Code Review** 来「现场学习怎么写 QA 计划」；可以由一位专家 **一次性沉淀** 可执行文档，让后续 **每条代理轨迹** 都带上高质量 QA 计划模板与要求。

## Harness 机制：JIT 指令、评审代理、Lint/测试即提示

### 自动上下文压缩时代：持续刷新「何为成功」

**核心观点**：随着 **自动上下文压缩/整理（auto context compaction）** 变好（讲者点名 **GPT-5.4** 与 **Codex** 在该能力上很强），工程师要假设 **上下文会被逐出（paged out）**；Harness 必须 **在任务全程反复提醒** 成功标准，而不是指望一次前置提示管到底。

**要点提炼**：

- **`AGENTS.md` 类文件** 仍有价值，但不能假设代理永远完整记住；需要 **评审代理** 在推送/CI 中持续对照「成功定义」检查补丁。
- **安全与可靠性评审代理** 可问简单但高价值问题：网络代码有没有 **超时与重试**？新接口是否 **难以被误用**？
- 讲者以自身为例：人很容易 **合并了带 retry/timeout 的修复** 却在别处继续忽略同类问题——**人不是可靠评审员**；但 **一次性写文档 + 定制 Lint**（例如每次 `fetch` 都必须带重试/超时包装）可以 **持久消除该类失误**。

**详细内容**：

讲者建议 **观察人机反复犯的失败类目**，系统性消除：**为何在这类事上反复花时间 → 设计机制让该类行为不再发生 → 继续观察与迭代**。

**结构测试（tests about source）** 示例：限制 **单文件不超过约 350 行**，把代码库 **适配模型与 harness**，提升上下文效率。

**错误信息即 prompt**：不要只报「Lint 失败：在循环里 await」或「深处出现 unknown」；要让失败信息 **告诉代理下一步怎么改**——例如强调 **在边界 parse、不要 validate**、以及 **此处类型来自 Zod（Zod 为讲者口述的负载承载基础设施）** 等。讲者强调：**这些都可以在不动模型权重的情况下完成**。

**提示注入渠道会爆炸式增长**：rules、skills、lint 报错、PR 上要求代理回应的评审评论、嵌入在测试里的 **Agent SDK** 评审等。讲者甚至描述 **让 Codex 阅读 OpenAI 开发者文档中的 prompting cookbooks，合成一个「如何写 prompt」的 skill**；需要写 prompt 时，再用该 skill 辅助——形成 **用代理写 prompt 来写 prompt** 的递归杠杆。

**QA 计划闭环**：当产品同学把功能、关键用户旅程、触点写清，并规定「所有面向用户的工作都要有 QA 计划」，**评审代理** 就能断言 PR 是否 **证明** 特性完成；附件与媒体要求让人与代理都更少需要 **盯着代理肩膀看**。

讲者以「**模型渴望 token**」收束：**可以用子代理（sub-agents）** 等方式，把 token 花在对齐与打磨输出上，从而把人从同步驾驶位上解放出来。

## 规模化与代码评审：冲突、垃圾回收日与代理评审员

### 高并行 PR 与「周五垃圾回收日」

**核心观点**：早期每人每天 **3–5 个 PR**、团队很小时也会 **合并冲突地狱**；解决路径一是 **树状拆分代码降低冲突面**，二是 **缩短 PR 打开时间**；而 PR 长期悬开往往因为 **人类评审成为瓶颈**。

**要点提炼**：

- 设立 **「垃圾回收日」（例如周五）**：全员专门处理一周内观察到的 **slop 模式**，把「让人在 PR 里反复说同样的话」转化为 **仓库内的永久机制**（文档、测试、Lint、评审代理）。
- 把人类评审意见 **按人格/职能分桶**（前端架构、可靠性、扩展性等），为每桶配置 **在每次 push 触发的评审代理**：是否达到文档定义的「好工作」？是否存在 **P2 及以上阻塞合并项**？
- 随着文档持续增厚，**slop 逐步下降**。

**详细内容**：

这一机制的本质，是把 **同步的人类 Code Review 时间** 换成 **可自动 surf 的知识**，让代理在失败时 **自愈式对齐**。

## 现场问答：工作流、工具链、协作平台与入门路径

### 工作入口：以 Codex 为中心「从外到内」

**核心观点**：团队以 **工单（tickets）** 为单元，把工单交给代理并附上少量 **Skills**；**开发流程的入口应是 Codex**，而不是先造一个把人或代理包进去的环境。

**要点提炼**：

- Skills 教 Codex：**如何启动应用**、如何起 **本地可观测性栈**、如何用 CLI 连接 **Chrome DevTools（通过本地守护进程/daemon）** 等。
- 仓库内有许多「小 harness」：例如 **覆盖每个 PNPM 包** 的定制 ESLint 规则集；更高层的 **结构健康测试**（包隐私、依赖边、跨文件 Zod schema 去重、异步 helper 单一实现等），专门克制代理 **「局部自洽但违背全局复用」** 的倾向。
- **Skills 数量**：更偏向 **5–10 个高质量 Skills**，持续加厚，而不是铺几千个；因为本地工具链变化快，复杂度应 **藏在 skill 后**，由人触发，让代理自行摸索执行细节。
- 讲者举例：从直接用 **Chrome DevTools Protocol** 迁到 **daemon** 架构后，自己 **三周没察觉**，因为 Codex 仍能用文档与流程把事做成。

### 如何避免过度工程化 Harness（「bitter lesson」视角）

**核心观点**：驾驭层不应赌某种具体模型技巧长期不变；**最不容易被模型进步淘汰的** 是 **最小化的上下文管理**：在任务全程 **准时（JIT）把要求递给模型**。

**要点提炼**：

- 例：不必前置加载「React 组件要如何拆分才利于 snapshot 测试」；可以让代理先 **把 UI 做出来**，再在 **lint/test 阶段** 要求拆分小组件、减少 prop drilling、hooks 本地化等——代理会 **基于新指令改写补丁**。

### 第一方 Harness 与「模型在后训练里对齐工具语义」

**核心观点**：实验室不仅后训练模型，还在 **主要部署 harness** 的语境里后训练——例如 **apply patch 工具**、bash 工具的 **引用/转义语义** 等；因此 **依赖第一方 harness/SDK** 能吃到 **与工具链共同演化** 的红利。

**要点提炼**：

- 个人工作重点应上移到 **不同模型版本的行为差异** 与 **如何引导行为**，而非深陷 harness 内部实现细节。

### 协作平台：Markdown + GitHub PR 作为「广播域」

**核心观点**：协作主战场仍是 **仓库里的 Markdown** 与 **GitHub**；**PR** 类似「干净房间」里的协作文档：人类与代理都在此交流。

**要点提炼**：

- 为吞吐优化：**不强制**每个人必须贡献评审；代理也可评审或不评审。
- **实现代理** 可以对反馈 **确认、延期或拒绝**；避免「每条反馈都必须照做」导致 **代理被评审霸凌**。
- 价值观：**偏向合并可接受但非完美的代码**，而不是在细枝末节里淹死。

### 新手如何过渡：测试与时间审计

**核心观点**：两条入门路径并行：

1. 用代理 **为现有代码补测试**，提高你对代码与代理导航代码的信心。
2. 审计你把时间花在哪：**写代码、等测试、等评审、CI 慢、 flaky 测试**——用代理 **逐步自动化** 这些摩擦点。

讲者补充：**抽象积木**（例如让响应 Kafka 事件很容易正确）能降低每个工程师都要手搓消费者的必要性；这类 **可组合基础设施** 对人与代理都 **可堆叠放大**。

### 「在车里怎么用代理」

**核心观点**：讲者未使用新推出的 **CarPlay 语音模式**（自称还没准备好）；常见做法是下班前 **开任务**，笔记本 **绑在后座**、**手机热点**，通勤 30 分钟让它跑；Skills 告诉代理 **跑到测试全绿**。愿景是 **24/7 跑 50 个代理** 且几乎 **零交互**。

**金句**：**每次你必须对代理说 continue，都是 harness 没给够「如何继续直到完成」的上下文**——这是失败信号。

### 组织与代码库变大：渐进披露与「750 个包」

**核心观点**：从空仓库单包起步会 **失控**；团队最终走向 **约 750 个包的 PNPM workspace**，按业务域与栈分层隔离，并用 lint **强制正确使用共享能力**。

**要点提炼**：

- 即便不是微服务，也要让 **大多数改动能局部发生在某个子树**。
- **代码风格与基础设施一致性**（一种并发 helper、一种可观测命令构造方式、一种 ORM、一种 CI 脚本写法……）提高 **可迁移的上下文**，让模型在不同目录产出更 **可预测的 token**。
- 需要有人/集体扮演 **「独裁者标准」**：定义「必须这样做」，并可用代理 **迁移全库对齐**。

## 代币预算、计划模式与「代码是否一次性产物」

### 日均十亿 token 怎么花：大约三等分

**核心观点**：讲者估计其消耗大约 **三分之一在规划/工单整理/文档**，**三分之一在实现**，**三分之一在 CI 里跑的东西**。

**对 Plan mode / exec plans 的态度**：

- 用过早期类似 **exec plans** 的原型（带里程碑与验收条件的计划 skill），但 **不指望 plan mode 成为 harness 核心**。
- 预期是：**丢工单就应能做完**；若计划 **你自己也不会读**，却批准执行，等于 **注入一堆你不真想遵守的指令**。
- 若要用计划：**把计划单独做一个 PR**，**人类逐行审阅并阻塞合并**，再启动 rollout；减少「坏指令被默默执行」的浪费。
- **CI 消耗 token 是必要的**：写代码不难，**让代码被接受并推动产品前进**才难；**资深工程师给好 Code Review**——也应期待 **资深工程师代理** 做到同样的事。

### 「代码是不是可丢弃的构建产物？」

**核心观点**：**是（Yes）**。讲者连接到 OpenAI 发布的代理编排方向（讲者口播名称经公开播客材料核对为 **Symphony**；听录常被写成 symfony）。

**要点提炼**：

- 可以把库看作 **非常清晰的 spec**，而代码是 **在该 spec 约束下编译出的产物**。
- **LLM 像模糊编译器**；仓库里为 harness engineering 堆的上下文，相当于 **约束与优化 pass**，决定 **哪些代码「可被判为可接受」**。
- 类比：**替换模型** 像 Rust 编译器换代码生成后端（LLVM vs cranelift）；**可接受 Rust 的静态规则**仍应保证输出 **_sound_** 的机器码——同理，**围绕代码的结构** 应限制写法，使不同模型仍产出团队可接受的补丁。

## 对未来的想象：按季度分配 token 预算的自主推进

**核心观点**：理想状态是拿 **一个季度/半年/一年的 token 预算**，把人类排序的 **成功指标、可靠性指标** 交给机器，让产品 **持续向前**，人不必手扶方向盘。

**要点提炼**：

- 软件工程里许多环节会经历 **从 0 到 1 的能力爬坡**（讲者用 **雷达图/五边形能力图** 比喻）：例如早期 **对构建产物做 QA 冒烟** 很弱，就要补 **文档、工具、代理可执行旅程**。
- 「不写代码」后，人的脑力会更多落在 **用户反馈分流、on-call、日志 PII、舆情、用户运营 runbook** 等更「软」的工作——但讲者认为 **这些也可被代理承担**，关键是 **流程与验收标准** 的可写性与可执行性，即一种 **元编程（meta-programming）**。
