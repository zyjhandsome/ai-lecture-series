## 全局核心提要（Executive Summary）

> **AI 导读**：
> - **实现不再是稀缺资源**：在讲者观察中，软件工程的关键瓶颈已从「手写代码」转向如何部署算力与 token、如何设计套索（工具链、规则、测试、文档）让代理可靠地完成长周期任务。
> - **「代码免费」是工作假设**：生产、重构、删除代码的成本在模型侧可并行、可忍耐；人类应把注意力放到系统设计与委托；维护负担通过统一规范、自动化审查与一次性迁移消化。
> - **三类真正稀缺资源**：人类时间、人类与模型的注意力、上下文窗口；工程化目标是让 token 可预测、让约束在 PR 全周期「按需浮现」，而非一次性堆进提示词。
> - **从人肉评审到「可附着文档」**：周五「垃圾回收日」把本周评审里重复的「糊涂代码」类反馈沉淀为文档、lint、测试与审查代理，使人机时间从同步评论转向高杠杆结构。
> - **协作枢纽仍是 GitHub + 仓库内 Markdown**：PR 被视作洁净室；为吞吐优化，不强制每条反馈必须被采纳，避免代理被多评审者「围攻」导致失效。

## 主题演讲：从「token 亿万富翁」到套索工程

### 代理时代的工作假设与「AGI 药丸」

**核心观点**：讲者近九个月**只通过代理**构建软件，并曾**禁止团队直接打开编辑器**、强制经模型完成工作；他认为要走向「AGI 未来」，应让每个人都成为能负担大规模模型使用的 **「token 亿万富翁」**——即把模型当作**完整软件工程师**来用，而不是辅助打字。

**要点提炼**：

- **行业变化**：约六个月内，编码代理能力与套索（运行环境、工具、工作流）同步进化，能在更长时域、更高复杂度上更可靠地行动。
- **「代码免费」（code is free）**：日常问题上有「代码供给过剩」；招聘键盘工程师主要受 **GPU 容量与 token 预算**约束；每位工程师理论上拥有「5、50 或 5000 个工程师当量」的 24/7 产能，关键是**如何productive 部署**。
- **技能迁移**：更多转向**系统思维、系统设计与委托**；讲者将转折点放在 **2025 年末**，个人「魔法时刻」是 **GPT-5.2**——他认为该版本起模型在真实代码库中产出高质量代码、解决真实用户问题的能力，已与人类工程师在该维度上**同构（isomorphic）**。
- **维护焦虑的再框架**：人类讨厌代码负担，常因代码是**同步注意力黑洞**；模型极有耐心、可无限并行，因此产码/维护/重构/删码**不应再是**团队资源分配的主约束。
- **人人即「员工工程师」**：并发能驱动的代理数、token 能支撑的规模，决定你的「团队规模」；需提前规划**一天、一周、半年**的结构以吸收无限产能。
- **P3 任务范式转移**：在人类稀缺世界里只有 P0/P2，**P3 永不做**；在代码免费世界里，P3 可**立刻并行开 4 倍尝试**，选能解决问题的合并。
- **国际化示例**：在 OpenAI 内部做大量代理工具；代码免费时，内部工具可**从第一天起**做好本地化，伦敦、都柏林、巴黎、布鲁塞尔、苏黎世、慕尼黑同事可用母语体验，而不必严重挤占其他团队容量。

**详细内容**：

讲者自称 **「token billionaire」**，并希望未来人人都用模型「做满全套工作」。implementation 已非软件工程稀缺资源；**键盘人力**仅受 GPU 与 token 限制。技能转向系统思考与委托；**GPT-5.2** 是他眼中模型能承担**完整软件工程师工作**的分水岭——产出与真人同构的高质量代码。代码可免费生产、重构；维护负担应通过模型并行消化。工程师的职责是**解除代理与人类驾驶者**在长周期任务上的阻塞。每个人都是 staff engineer，结构要吸收无限并行产能。

真正稀缺的三样东西：**人类时间**、**人类与模型的注意力**、**模型上下文窗口**。自动化同步低杠杆时间；当代码免费时，低优先级工作可并行试错。

### 文档、ADRs 与「对代理可读」的系统

**核心观点**：人类不必再执着于手写实现本身；重要的是**把你带到正确代码的提示与护栏**——文档、ADRs、人格化（persona）说明、工单与代码评审历史，都是过程资产，必须让代理也能消费。

**要点提炼**：

- **可替代性**：应假设「我们所知的软件工程最佳实践」在**任何产品**中随时可用。
- **对代理可读（legible）**：系统、软件与流程要**对驾驶者透明**；以代理原生方式组织；尊重上下文稀缺；让完成任务所需 **token 可预测**。
- **一致性降低激活能**：尽量让代码与流程「长得一样」，减少模型每次任务要「激活」的规则总量。
- **大规模重构免费**：不会再有拖半年的迁移——可派 **15 个代理**把尾部收口做完（讲者现场带动鼓掌）。

**详细内容**：

 breadcrumbs、文档、ADRs、关于「何为好工作」的人格化文档、历史 ticket 与评审记录——这些塑造了今日产品与代码，也必须塑造代理路径。要把结构写得**省上下文**、**可预测 token**。迁移类工作应快速闭环。

## 主题演讲：规范「好工作」——非功能需求、防糊、可执行错误信息

### 非功能需求必须写下来，并接受短期速度损失

**核心观点**：「做好工程」依赖大量**未写明的非功能需求**；模型在训练里见过海量代码与各种非功能取舍，**人类工作是把 acceptable job 写成代理可执行的约束**；若输出不行，就收紧护栏，而非默默接受糊代码（slop）。

**要点提炼**：

- **反糊策略**：可以明说 **「不要产糊、不要收糊」**，但通常需要**短期降速**，回头分析代理在你环境里卡在哪里，再加规则，然后回到高杠杆活动。
- **团队人格互补**：前端架构、后端扩展、产品思维等不同人格，各自贡献对非功能需求的解法；**一人写清、全员代理受益**，减少低信号人肉评审。

**详细内容**：

单次 patch 背后可能是数百个微小非功能决策；代理见过所有可能组合，你要指定**哪一种是你们可合并的 acceptable**。不愿接受 slop 就要承受短期 velocity hit，定位失败模式，建立护栏，再扩展杠杆。

## 主题演讲：把提示注入到仓库与 CI——审查代理、测试元数据、QA 计划

### 上下文会被「分页」：要靠审查代理与失败路径反复注入指令

**核心观点**：自动上下文压缩（讲者口播作 *autoco* compaction）持续改进；**GPT-5.4 与 Codex** 在该能力上极强，使他几乎不必再 `/new` 开新会话。必须假设**上下文会随时间被挤出**，要在任务全程**刷新**「何为成功」。

**要点提炼**：

- **agents.md / 规则文件**：可写，但不能只靠一次性大提示。
- **审查代理**：在每次 push/CI 中对照文档与补丁做**安全、可靠性**等检查（例如网络代码是否有 **timeout/retry**；接口是否**难以误用**）。
- **人肉不可靠的例子**：讲者自嘲曾只为救火加 retry/timeout，合并后便不再系统推广；**定制 lint**（例如每次 `fetch` 必须包 retry/timeout）可**一劳永逸**，且依赖「代码免费」可全库迁移。
- **失败分类闭环**：观察代理与人类**重复犯的错** → 弄清时间花在哪 → **系统性消灭该类行为** → 持续迭代非功能清单。
- **关于源码的测试**：除 lint 外，可规定**单文件不超过 350 行**等，使代码库适配模型与套索，提高上下文效率。
- **错误信息即提示**：不要只报「lint 失败」或深层 `unknown`；要在失败输出里写清**修复语义**（例如：边界处 parse don't validate、类型来自 **Zod** 等）——**不碰权重也能调行为**。
- **提示注入渠道激增**：rules、skills、lint 文案、PR 评论、合并前必须处理的审查意见、嵌 **Agent SDK** 的测试等；若写提示太多，可让代理读 OpenAI 开发者指南里的 prompting cookbooks **合成 skill**，再用 skill 写提示（「用代理读提示书写提示」）。
- **QA 计划**：产品型同事定义何为好的 QA plan（功能、关键用户旅程、触点）；要求所有面向用户工作附带 QA plan 后，**审查代理可断言**「如何证明功能完成」；PR 应附媒体证据；结果：**更信任输出、更少盯着代理**。
- **子代理与 token**：模型「渴望 token」；可通过子代理与工程化给足 token 推进任务，减少人类当同步司机。

**详细内容**：

讲者提到把笔记本绑在车后座通勤时仍跑推理（社交媒体配图）。CI 里跑安全/可靠性审查代理；用 bespoke lint 固化 retry/timeout；用**可教学的失败信息**把组织知识灌回模型；QA plan + 审查代理减少 shoulder surf。结论是：**大胆让代理做满工作**。

## 问答：工作流、技能与仓库结构

### Codex 为入口、「外向里」开发与迷你套索

**核心观点**：工作从 **ticket** 开始，交给代理并附上少量 **skills**；**开发入口应是 Codex**，而不是先造一个再塞 Codex 进去的环境；仓库与本地工具链按 **Codex 先调用**来设计。

**要点提炼**：

- **Skills 示例**：启动应用、拉起本地可观测性栈、用本地 CLI 打开 **Chrome DevTools** 并 attach（经 **daemon** 连接，非长期手搓协议细节）。
- **嵌套套索**：自定义 **ESLint** 规则包接入每个 **pnpm** 包；本地「高级 wholesome 测试」断言**代码结构**（包隐私、层间依赖边、跨文件 **Zod** 去重、异步 helper 单一路径等），防止代理**只在包内局部最优**而忽略共享工具。
- **技能数量**：杠杆集中在约 **5–10 个 skills**，宁可加深现有 skills；底层工具变得快，人类带宽有限，复杂度藏在 skill 后，由代理自行消化。
- **工具替换无感**：从直接用 Chrome DevTools Protocol 换到 daemon 方案后，讲者**三周没察觉**，因 Codex 靠文档与结构仍能完成——说明套索与文档到位时，底层替换可不打断人类。

**详细内容**：

优化目标是代理完成任务、人类不必跟踪高 churn 细节。

### 如何避免过度工程化套索（Bitter Lesson 视角）

**核心观点**：好套索 = **在正确时间给模型正确文本**；模型被训练为遵循指令；套索不应被能力升级轻易作废——**上下文与需求传递**不会被淘汰，但要**最少化**上下文管理；**不要 frontload 所有指令**（会压垮代理），也不要让要求只在开头出现——**PR 全程**都要能浮现。

**要点提炼**：

- **延迟指令示例**：React 可先让代理自由原型，再在 **lint/测试** 阶段要求拆小组件、少状态、hooks 本地化，避免 prop drilling——代理收到「新指令」后改补丁再推 GitHub。
- **结论**：关键不是模型变多强，而是**何时注入哪段约束文本**。

**详细内容**：

这与「better lesson」一致：避免做会被模型进步一次性废掉的重 harness；**按需 surfacing** 更耐久。

## 问答：套索设计原则与第一方套索（Codex 等）

**核心观点**：实验室不仅 post-train 模型，还在模型**主要部署的套索语境**里 post-train（如 **apply patch** 工具、bash 引号语义等）；因此**直接依赖第一方套索**（经 **SDK** 或操作 **Codex app server**）可吃到后续训练红利；个人应专注「正确代码长什么样」，而非套索每一颗螺丝。

**要点提炼**：

- **Claude Code、Codex** 等会持续变好，那是产品团队责任；讲者角色是**接入并转向**代理行为。
- **工作重心上移**：关注**版本间模型行为差异**与可观察输出，而非套索内部机制。

**详细内容**：

ASR 将「Claude Code」误为 *clog code*，已校准。

## 问答：协作、上手路径与规模化知识

### 协作平台：GitHub PR + 仓库 Markdown

**核心观点**：协作主形态仍是 **repo 内 Markdown + GitHub**；**PR** 类似 Google Docs 的洁净室；人类与代理都在此广播式协作；为吞吐**不强制**每条评论必须处理；实现代理可对反馈 **acknowledge / defer / reject**。

**要点提炼**：

- **避免「每条反馈必须改」**：否则代理会被多评审**霸凌**；应偏向**合并可接受、非完美**、不淹没在琐事中，保留模型推理空间。

**详细内容**：

优化的是整体 diff 产出与接受度，而非会议式一致。

### 传统工程师如何上手

**核心观点**：两路并进——(1) 用代理为**现有代码补测试**，提升人与代理双方信心；(2) 审计时间花在哪（写码、等测试、等评审、CI、flaky），**逐项自动化**；高杠杆在定义/排序工作并赋能他人；为 Kafka 等建**可复用原语**后，不必跟每个消费者实现死磕——同样适用于代理堆栈。

**详细内容**：

从「增强确定性」与「消除等待」切入，自然过渡到编排角色。

---

### 车里怎么用代理

**核心观点**：尚未用 **CarPlay 新语音模式**；常在下班前开任务，笔记本**手机热点、固定在后座**，通勤约 **30 分钟**跑到**测试全绿**；skill 告诉代理「做到 green 为止」，无需一路点 continue。

**要点提炼**：

- **金句**：**每次不得不跟代理交互，都是套索没给够「如何跑到完」的上下文**的失败。
- **愿景**：**50 个代理 24/7**、人零点击；靠任务定义与自动调度，而非手点按钮。

**详细内容**：

目标是最大化「无人值守 token 消耗」。

---

### 组织与代码库规模化：750 个包与「代码即提示」

**核心观点**：从空仓库单包 Electron 起步会**变乱**——缺包隐私与域边界时代理缺「文件系统钩子」；他们走到 **「一万工程师组织级」** 架构：**pnpm 工作区约 750 包**，按业务域与栈分层；小 util 包封装可复用能力并用 lint 强制使用。

**要点提炼**：

- 即使不搞微服务，也要让**多数改动可局限在子树**，缩小代理视线。
- **代码即文本即提示**：越统一，越可迁移的上下文；**一种**有界并发 helper、**一种**可观测副作用命令范式、**一种** ORM、**一种**语言、**一种** CI 写法、**一种**加 lint 规则的路径——token 更易预测。
- **迁移策略**：定「独裁式」规范或共识成文，再让代理**整库迁移**到一致形态。

**详细内容**：

这是 progressive disclosure 的工程版：结构即地图。

## 问答：代码评审、token 配比、计划模式与「代码即产物」

### 高吞吐下的评审：周五垃圾回收日与审查人格

**核心观点**：早期每人每天 **3–5 个 PR**，三人团队即**合并冲突地狱**（PR 大、改动区重叠）；解法：**树状拆分**减少冲突 + **缩短 PR 开放窗口**；窗口长的主因曾是**人肉评审**。

**要点提炼**：

- **周五垃圾回收日（garbage collection day）**：全员把本周阻碍合并的「糊」分类，**从机制上杜绝再犯**——人肉评论反映的是**代理上下文失败**，应进仓库，经失败测试或审查代理**自愈合**。
- **人格分桶**：前端架构、可靠性、扩展性等；每类人格spawn **审查代理**，每次 push 问「是否达标」、 surfacing **P2+ 阻断项**；文档持续追加后 **slop 持续下降**。

**详细内容**：

从同步评论转向可附着、可执行的系统知识。

### 每日十亿 token 怎么分、预算收紧怎么花

**核心观点**：约 **三分之一 × 三份**——**规划/工单整理**、**文档**、**实现**，以及 **CI 里跑的东西**；CI 消耗 token 是**必要**的，因为难的是**让代码被接受并推进产品**，而非写出代码。

**要点提炼**：

- 主持人口播：**日均十亿 output tokens**、约合 **一千多美元**量级（粗略消费量级，非精确账单）。

**详细内容**：

对预算是「结构性」思考：接受与证明比打字贵。

### Plan mode 与 exec plans

**核心观点**：用过早期公开的 **exec plans**（带里程碑与验收标准的类 skill）；**未在套索里用 plan mode**；期望是**丢 ticket 就能做完**，多数计划自己**不会读**——若批准计划却**不读**，等于写入**可能不该被执行**的指令。

**要点提炼**：

- 若用计划：应**单独 PR 只含计划**、**人类逐行审**、**批准后再**启动执行，减少坏指令 rollout。
- **金句传承**：**资深工程师给好 code review**——**资深工程师代理**也应如此。

**详细内容**：

计划是可选分支，默认直开任务。

### 代码是否是可丢弃构建产物

**核心观点**：**是**；与 OpenAI 发布的代理编排项目 **Symphony**（讲者口播作 *symfony*，指 [openai/symphony](https://github.com/openai/symphony)）相关——可发布「规范极明」的库，**代码是该规范的编译产物**；**LLM 是模糊编译器**；仓库里为 harness 堆的上下文=**可接受代码的约束与优化遍**。

**要点提炼**：

- 类比 **LLVM** 编译 Rust：静态分析与优化遍决定可接受 Rust；换模型像换代码生成后端（如 LLVM 与 Cranelift），仍期望**同一套前端规则**产出**合法机器码**；换 LLM 同理，用结构限制**可接受代码集合**。

## 问答：愿景——按季度预算交给机器持续推进

**核心观点**：理想状态是拿**季度/半年/年**的 token 预算与工单优先级、成功与可靠性指标交给机器，**持续推进产品**，人手不必时刻握方向盘；从原型到内测、外测，软件工程能力像**雷达图**——首次部署分发时，代理对构建产物做 **QA 冒烟**很弱，于是让代理**自建工具**下载产物、启动、验证关键旅程。

**要点提炼**：

- **写代码之外的全集**：用户反馈分流、on-call、日志 **PII**、舆情、用户运营 runbook、把运维痛点回填到代码等；人脑可转向更「软」的活动，而这些也可被代理做——**把工作流程与验收标准写下来**成为新的元编程。

**详细内容**：

长期是「定义与度量」驱动自治迭代。
