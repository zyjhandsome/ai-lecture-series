## 安全通告洪流：数量、对比与「slop」启发式

### 1142 条通告与日均节奏

**核心观点**：讲者用 **量化数据** 描述被安全通告「淹没」的体验，并给出 **与 Linux、curl 的数量级对比**；同时提出 **判别低质报告的启发式**。

**要点提炼**：

- 累计约 **1142** 条 advisories（讲者笑称可能已到 **1150**）；约 **16.6/天**；其中 **99** 条 **critical**。
- 已 **发布** 约 **469**，**关闭** 约 **60%**。
- 对比：**Linux 内核** 约 **8–9/天**；OpenClaw **约两倍**；**curl** 迄今约 **600** 报告，OpenClaw **约两倍**。
- **经验法则**：通告方 **喊「critical」越大声，越可能是 slop**。
- 宏观判断：AI 工具让 **怪异多链漏洞** 更易被挖出，**既有软件** 将面临系统性压力，行业需改 **构建方式**。

**详细内容**：

讲者认为数字「听起来恐怖」，但解释框架是：**披露机制 + 激励** 让 OpenClaw 成为 **被集中火力试破** 的对象；与「积分/名声」产业逻辑一致。

## 案例：Nemo Claw 沙箱、CVSS 10、供应链与国家背景攻击

### 英伟达 Nemo Claw 半小时内五种逃逸

**核心观点**：用 **内部更强网络安全模型** 可在极短时间找到沙箱逃逸路径；说明 **能力不对称** 与 **产品化安全层** 的困难。

**要点提炼**：

- **Nemo Claw**：英伟达的 **插件 + 安全层**，可把 OpenClaw 放进沙箱；**周一** keynote，**周日** 邀讲者联调；接 **Codex 安全能力** 后 **半小时** 内找到 **五种** 逃出沙箱的方法。
- 原因：产品附带访问 **未广泛公开的更强模型**，在 **网络攻击** 维度 **明显强于公开模型**——因为 **危险** 所以受控。
- **CVSS 10 案例（Gshjp）**：`[ASR不确定：漏洞编号转写为 Gshjp，请以 CVE/GitHub Advisory 为准]`。要点是：若同步 **尚未发布的 iPhone 应用** 且只给 **读权限**，可能被利用升级到 **写权限**。讲者辩称 **99% 实际部署** 是：本机、自有云或 **Mac mini**（讲者表示不再争论，随用户选择）；典型威胁模型是 **要么能碰 gateway，要么不能**——因此 **多数现实用户不受影响**。
- 讲者自认错误之一：曾尝试做 **更细粒度权限**（如设备侧 **语音** 场景只读），但 **几乎无人使用**；**CVSS 规则** 不纳入这种部署现实 → 仍可出现 **10/10** 评分 → 外界 **过度反应**。
- **GhostClaw**：怀疑 **朝鲜背景** 的 **npm 仿冒**，诱导错误网站下载 **rootkit**——**超出项目控制**，类比其它生态也发生。
- **Axios 供应链**：项目 **未直接使用 Axios**，但依赖 **MS Teams / Slack**，后者依赖 Axios 且 **未 pin 版本** → 典型供应链连锁 → OpenClaw **间接受影响**。

**详细内容**：

该段核心不是「否认漏洞」，而是区分 **（a）评分/叙事放大** 与 **（b）真实用户威胁模型**，并点名 **依赖链与国家背景钓鱼** 两类 **更难防** 的风险。

## 媒体与论文叙事：Agents of Chaos、比利时通告与「对抗默认建议」

### 忽略安全文档的「好故事」

**核心观点**：讲者批评 **Agents of Chaos** 类论文：**用大量篇幅拆解 OpenClaw 架构**，却 **不写官方安全安装页**；若按文档建议部署，许多「惊悚实验」不成立。

**要点提炼**：

- 论文被批：**四页** 细讲 OpenClaw 架构，**安全章节** 回避官方建议。
- 官方建议摘要：**个人代理** → **不要放进群聊**；若必须群聊 → **开沙箱**；团队代理 → **仅团队可见数据**、禁机密；个人代理 → **仅你本人** 应能对话。
- 讲者称与作者沟通后发现：他们 **以 sudo 模式运行** 以追求「最强代理」，**需改代码** 才能实现——**报告未写**，因为不利于 **clout（声势）**。
- **比利时案例**：网络安全机构发布 **RCE** 类通告；实质是 **恶意网站链接触发 gateway 并转发 token**。若用 **默认/推荐** 配置，**token 仅本机或私网**，外站 **无法触达**；若用 **Cloud Code 等云端方式** 且 **不读文档** 硬搭，才可能 **作出来**——**非官网推荐路径**。

**详细内容**：

讲者总结：**风险真实存在**，但核心框架是 **「强大代理 + 不可信内容 + 外联能力」** 的 **法律三角（legal trifecta）**——**任何强力代理系统** 都面临，不是 OpenClaw 独有；**越强大 → 越需理解其行为**。

## 维护负担：志愿者模式极限与「报告+修复」陷阱

### AI 写报告、人类仍要动脑

**核心观点**：大量报告由 **代理生成**，维护者仍必须 **人工阅读**；误判与 **劣质补丁** 会 **拖垮或破坏产品**。

**要点提炼**：

- **无法** 完全信任代理自动分类/定性（讲者个人达不到「全放手」）。
- **时间黑洞**：报告 **语气过礼貌、道歉** 可能暗示 **AI 代写**（传统安全圈较少道歉）——仅启发式。
- **ffmpeg** 等项目也公开抱怨类似压力。
- 常见模式：**只有报告** 多、**报告+补丁** 少；若 **补丁+报告** 同来，补丁 **往往很差**；**赶工合并** 会 **几乎必然破坏产品**。

**详细内容**：

对策线索：引入 **英伟达** 等机构 **全职** 人手 **过滤 slop + 加固代码库**；并承认 **单靠志愿者** 不可持续。

## AMA：本地与开放模型、数据主权与「黑客路径」

### 欧洲视角：自有数据与绕过 Gmail 半年审批

**核心观点**：OpenClaw 动机之一是 **数据自主**；消费者代理可 **绕过企业 API 关卡** 完成自动化——带 **合规/道德边界**，但是 **「黑客式」创新空间**。

**要点提炼**：

- 大公司连接器进 **Gmail** = 你的邮件 **在很大程度上对他们透明**；个人托管更 **刺激**。
- **欧洲心态**：想 **拥有自己的数据**。
- 创业公司接 Gmail **可能要半年**；消费者代理可以 **像人一样点网站、过「我不是机器人」**——若业务 **给你数据**，代理就能 **拿到数据**。
- swyx 总结为 **hacker way**；讲者认同。

**详细内容**：

与后文 **小模型警告** 形成张力：**开放/本地** 是价值，但 **小模型 + 浏览器/邮件** 仍可能 **更危险**。

## AMA：多会话 Codex、「暗工厂」与迭代式品味

### 从 10 窗到 5–6 窗：等待时间 vs 并行ism

**核心观点**： famous **多会话并行** 是 **延迟 workaround**；**fast mode** 与引擎改进后 **并发窗口下降**；未来 **不应** 长期依赖「六件事同时跑」。

**要点提炼**：

- 高峰约 **10** 个并行会话（提 **Codex 5.0/5.1** 时期更慢）；现在典型 **5–6**；因 **单次循环更快**、**fast mode**。
- **暗工厂（dark factory）**——几乎不审代码就合入：讲者认为 **部分场景越来越可行**，但 **不能** 一开始就定死全规格；**上山路径是弯的**，中途发现捷径与需求漂移。

## AMA：soul.md、WhatsApp 人格与「疯癫加一点科幻」

### 渠道人格：从 Cloud Code 接到 WhatsApp 的不对劲

**核心观点**：2023–2024 的 ChatGPT **像重构的 Google**（搜索框、无性格）；进入 **代理 + 聊天渠道** 后，**人格与字数习惯** 必须匹配 **WhatsApp 朋友**。

**要点提炼**：

- 初版 **Cloud Code** 接 WhatsApp **感觉不对**：太冗长、太多句号；朋友 **不那样发短信**。

**要点提炼**：

- swyx 观察：**智能家居设备安全差 → 反而让 OpenClaw 能控**——反讽。
