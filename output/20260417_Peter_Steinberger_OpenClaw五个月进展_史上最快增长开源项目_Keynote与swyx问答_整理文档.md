## 项目增长与工程速度：直线上升的「杆状」曲线

**核心观点**：OpenClaw 在约五个月内达到极高提交与贡献者速度；增长曲线被 Peter Steinberger 形容为近乎垂直的「杆」，而非传统「曲棍球杆」尾部上翘。

**要点提炼**：

- 项目约 **五个月**；Peter Steinberger 称可视为 **GitHub 史上增长最快的项目之一**（口述）。
- 规模量级（均为 Peter Steinberger 现场数字）：约 **30k commits**、即将 **2000 contributors**、将破 **30k PRs**；截至演讲日附近仅到 **4 月 9 日**数据仍显示斜率未明显放缓。
- 星标：Peter Steinberger 称**除少数更大的「教学类」仓库外，没有其他软件项目达到同级体量**（口述，未与第三方榜单交叉验证）。
- 幽默表述：朋友将增长曲线称为 **stripper pole growth**（钢管舞杆式垂直增长）。

**详细内容**：

开场主持人介绍其为 OpenClaw 创建者、近期加入 OpenAI 以推动代理普及。Peter Steinberger 以「State of the Claw」自称，现场约 **30–40%** 听众举手表示正在运行 OpenClaw。他强调速度仍在，但也说明「并不容易」。

## 双重职责：OpenAI 任职与 OpenClaw 基金会

**核心观点**：Peter Steinberger 曾不想再走「完整创业办公司」老路而选择加入 OpenAI，但又联合成立 **OpenClaw 基金会**，形成事实上的**两份全职量级工作**；基金会像**困难模式下的公司**——流程与责任齐全，但大量参与者是**无法直接指挥的志愿者**。

**要点提炼**：

- 目标之一：改善 **bus factor**（关键路径是否过度依赖少数人）；Peter Steinberger 称**缓慢好转**，并提到 **Vincent** 在下一环节演讲，但「还没到理想状态」。
- 已与多家公司沟通并引入支持：NVIDIA、Microsoft（MS Teams / Windows 应用）、**Red Hat**（安全、容器化）、多家中国公司；点名称 **腾讯**、**字节跳动**，并称**中国使用者规模大于其他大洲**；另有全球多地贡献者。

**详细内容**：

Peter Steinberger 将「运营基金会」比作硬模式公司：财务、法务、协作与产品节奏都要管，但志愿者动机与时间不可像员工一样管理。公司侧参与被定位为**补强安全与工程化**，而非替代社区。

## 安全通告洪峰：数量、对比与产业激励

**核心观点**：近两三个月大量版本变更与安全公告；Peter Steinberger 自称一度被安全通告 **「淹没」**。总量与速率需结合 **AI 辅助漏洞研究**、**CVSS 计分与实际部署**脱节、以及**产业按发现数计酬**来理解。

**要点提炼**：

- 累计约 **1142** 条 advisories（Peter Steinberger 口头称「我确定已经 1150 了」类表述，数字仍在涨），约 **16.6 / 天**。
- 其中 **99** 条标记为 **critical**；团队约 **469** 条已公开发布；约 **60%** 已关闭。
- 对比：**Linux 内核**约 **8–9 / 天**；OpenClaw 约 **两倍**。**curl** 迄今约 **600** 条报告，OpenClaw **约两倍**。
- 经验法则：通告喊得愈凶、严重度愈夸张，愈可能是 **slop**（低质或夸大的报告）。
- **NVIDIA NeMo Claw**（口述拼写多变，此处按 NVIDIA 安全层/沙箱产品与 OpenClaw 集成的语境理解）：Peter Steinberger 在周一会前被邀周日联调；接 **Codex 安全**能力后，**半小时内**找到约 **五种**突破其「安全沙箱」的路径。Peter Steinberger 解释原因之一是使用该产品可接触到**比公开模型更强的网络安全向模型能力**（因其危险而不对外）。
- 典型攻击面关键词：**RCE**、**绕过审批**、**代码注入**、**路径穿越**（path traversal，转写稿中作 pass traversal）。

**详细内容**：

Peter Steinberger 认为行业正进入「AI 极擅长找复杂链式利用」的阶段，将迫使软件构建方式改变；OpenClaw 因被贴上「不安全」标签而成为**数百人**同时尝试攻破的目标，形成报告洪流。

## 典型案例：CVSS 10、供应链与「朝鲜幽灵包」

**核心观点**：存在 **CVSS 10** 级条目，但在**主流自托管/网关默认可接受配置**下，Peter Steinberger 认为**多数用户实际不暴露**于该利用路径；同时存在**真实恶意供应链**与**依赖传递**导致的连带影响。

**要点提炼**：

- **CVSS 10 例**（Peter Steinberger 口述标识作「Gshjp」类读音，**具体 GHSA/CVE 编号请以 GitHub 安全公告为准**）：场景涉及**尚未正式发布但在开发中的 iPhone 同步应用**——若仅授予**读权限**，可能被利用提升到**写权限**。Peter Steinberger 辩称：典型用法是机器自托管（云端或 Mac mini 等），**99%** 场景下要么能访问网关要么不能；其个人失误在于曾尝试做**更细粒度权限模型**以支持「只读子集」类设备，但**几乎无人使用该模式**；CVSS 规则**不纳入**此类真实部署差异，因此仍显示为满分严重度，外界反应剧烈。
- **Ghost Claw**（命名口述）：Peter Steinberger 称**疑似朝鲜**背景行动，通过**混淆的 npm 包名**诱导用户从错误网站下载，从而植入 **rootkit**；属供应链与社工范畴，**超出 OpenClaw 可控范围**。
- **Axios 事件**：OpenClaw **不直接使用 Axios**，但依赖 **MS Teams / Slack** 等，而后者依赖 Axios 且**未锁版本**，供应链攻击下 **OpenClaw 亦受波及**。

**详细内容**：

Peter Steinberger 强调自己「尽量按规则玩游戏」（承认 CVSS 计分结果），但同时区分**纸面严重度**与**真实用户暴露面**。

## 误信息与恐惧营销：论文、媒体与实验方法

**核心观点**：除漏洞产业外，存在高校与媒体的**恐惧营销**；「Agents of Chaos」类传播材料用大量篇幅解析 OpenClaw 架构，却**忽略安全文档与推荐部署**。

**要点提炼**：

- 官方建议包括：个人代理**不要进群组聊天**；若必须群组使用应**开启沙箱**；群组代理只能访问**团队可知情数据**；个人代理应确保**只有本人**能对其下指令。
- 若违背上述规则，会出现「我能对你的代理说话并搞坏你的系统」类演示效果。
- Peter Steinberger 称曾追问论文方做法，对方承认以 **sudo / pseudo 模式**运行以求「代理最大功率」，且**修改代码才能实现**；这些实验条件**未写入报告**，因为不利于叙事。
- **比利时网络安全机构**曾就一例 **RCE** 发通告：恶意网站链接触发网关并转发 **gateway token**。Peter Steinberger 解释：在**默认且推荐**配置下 token **仅本机或内网**；外站无法触达。若用户**主动对抗文档**，例如用 **cloud code** 在未阅读文档情况下搭出**可被外网触达**的配置，则风险上升——但这**非官网推荐路径**。

**详细内容**：

Peter Steinberger 表达强烈挫败感：整个行业有人**系统性**把项目写成「默认不安全、不可接受」，同时大量实际用户与认真阅读安全文档的人能**正常使用**。

## 结构性风险：代理的「法律三重奏」

**核心观点**：真正需要反复讲清的风险框架是：**能访问你的数据** + **能消费不可信内容** + **能对外通信** 的代理系统普遍面临风险；能力越强，越要理解其行为；**这不是 OpenClaw 独有**。

**要点提炼**：

- Peter Steinberger 同意「绝对有风险」，但反对只贴标签不讨论**通用代理风险模型**与**缓解分层**。

**详细内容**：

此段为 keynote 收尾前对「风险叙事」的框架化回应，为后文 Q&A 中提示注入讨论埋下伏笔。

## 维护负担：志愿者模式与「报告多、可信修复少」

**核心观点**：大量报告由 **agent 生成**，维护者仍必须用**人脑**审；无法完全信任自动化研判。社区里**道歉太客气**的报告往往像 AI 写的，而传统安全圈**很少道歉**。仓促合并「报告 + 修复」常见于**劣质修复**；Peter Steinberger 称自己早期过载时匆忙处理**几乎必然会破坏产品**。

**要点提炼**：

- 单靠志愿者难以承受；**NVIDIA** 被点名表扬：派员**近乎全职**筛 slop 与加固代码库。
- 其他开源项目（如 **ffmpeg**）也有公开抱怨类似压力。

**详细内容**：

此段连接「基金会要雇全职」与后文 OpenAI 关系澄清：需要通过基金会**雇人**以维持节奏、质量，并解放 Peter Steinberger 做创新。

## 现场问答：本地模型、工作流、品味、人格、智能家居愿景

### OpenClaw 与 OpenAI 关系（「Closed Claw」质疑）

**核心观点**：社区担心合理；Peter Steinberger 通过**多公司分散贡献**与**治理结构**降低「单一厂商捕获」感知。

**要点提炼**：见上一节；补充：NVIDIA 被形容为「给工程师就真干活」的文化典范；**阿里巴巴、MiniMax、Kimi** 等模型厂商也曾沟通。

### 本地与开放模型为何重要

**核心观点**：动机之一是**数据自主**——不愿 Gmail 等全量暴露给大厂连接器；欧洲价值观里「拥有自己的数据」；分层模型：**本地/私有**为主，仅在需要时把少量 token 送到最强云端；另一层是**fallback 模型链**。

**要点提炼**：

- 创业接 **Gmail** 可能要 **半年**；消费者场景下个人代理可点网页、过 **「我不是机器人」**，用**黑客路径**绕过大厂筑起的孤岛，实现大公司**无法合法做**的自动化。
- swyx 总结为 **hacker way**，Peter Steinberger 认同。

### 编码工作流：多窗口、等待时间与「暗工厂」

**核心观点**：并发会话是对**延迟的 workaround**；工具变快后会话数可降；不认为能**一开始就写死全规格**再「暗工厂」式无人审核推进。

**要点提炼**：

- 数字：曾常开近 **10** 会话；Codex 改进与 **fast mode** 后典型约 **5–6** 窗；工作区同步的「工人」数量大致不变，故不再需要那么多分屏。
- 对「完全不审代码」：**越来越可行**，但**不等于好工程**。上山路径是弯曲的；第一印象很少等于最终产品；需要**迭代、试玩、改 prompt**。
- PR 自动化合并风险：贡献会把产品拉向**各方向**，AI 难以判断战略方向；需要**愿景文档**等引导，但瓶颈仍是 **sync** 与 **品味**。

### 「品味」如何定义

**核心观点**：底线品味是**不发出「AI 臭味」**——文字风格、人格、**UI**（左侧彩边、曾流行的紫渐变等）一眼可辨；更高层是自动化节省出的时间花在**微小愉悦细节**（例如启动时**随机吐槽/ roast** 用户的文案）。

**要点提炼**：

- swyx 提 **soul.md** 与人格化；Peter Steinberger 认为 2023–2024 的 ChatGPT 体验像**再造 Google**，用户不期待搜索引擎有人格；进入 **agent** 与 **WhatsApp relay** 后发现 **Claude Code** 默认语气**不像真人发短信**（太长、太多点号），故迭代「写得更像人」——也是一种品味。

### 金句：「带一点科幻的疯狂」

**核心观点**：OpenClaw 这类项目**不太可能从美国大公司内部诞生**——法务会在发布前**扼杀**；行业尚未解决部分问题，但现在缓解更多；Peter Steinberger 自述起步时评估最坏情况：token、邮件、照片外泄等「可忍受」 vs 公司场景不同。

**要点提炼**：

- swyx 引用 Peter Steinberger 的说法：**madness with a touch of science fiction**（带一点科幻的疯狂）来运行此类 AI 项目；Peter Steinberger 称并非所有艺术项目皆然，但 **OpenClaw** 特例如是。

### 产品形态愿望清单：无处不在的代理与智能家居

**核心观点**：理想是家中任意房间 **Star Trek式「Computer」**；多房间 **iPad**；代理知用户位置，**就近屏幕**用 canvas 展示答案；手机只是输入端；未来眼镜应能**旁听**并投影；企业 **OpenClaw** 与个人 **OpenClaw** 需建立**双方都舒适**的互联。

**要点提炼**：

- swyx 提 **Marc Andreessen** 与 **Andrej Karpathy** 等用 OpenClaw 管「家」；讽刺性观察：**物联网设备安全差**，反而让 OpenClaw **更能驱动**设备。

## 提示注入、做梦机制与插件化架构

### 提示注入（Prompt injection）

**核心观点**：**做得还不够**，但**前端/强模型**对网页、邮件等**不可信内容**标注后，**一般注入**「通常不再是最大问题」；若攻击者**无限度**对你的 claw 轰炸，仍有机会。小型（如 **20B**）本地模型若**无防御训练**又跑浏览器/邮件，**更令人担心**——故 OpenClaw 会**警告**小模型用户；被外界曲解为「恨本地模型」则属误读。

**要点提炼**：

- **Simon Willison** 被提及（提示注入术语与 **dual LLM** 思路）；Peter Steinberger 自嘲不是全知攻击面专家，**信任**终须存在。
- **Vincent** 侧与 **Tobi Lütke**（转写作 Toby Luca）[编者注：身份请以实际项目贡献者为准] 类似的**声誉随时间累积、高声誉更高权限**的系统，Peter Steinberger 称「是故事的一部分」。

### Dreaming（做梦）与模块化

**核心观点**：**Dreaming** 指 reconciling memories、梳理 session logs 等，类似人类睡眠中 **GC**、把部分短期记忆转入长期存储；Anthropic 代码泄漏亦显示其在探索 [口述，非独立验证]；已发布的是**第一步**。与 **Karpathy** 谈的 **wiki 式记忆**相关但不同：wiki 偏存储，做梦偏**整理与巩固**。OpenClaw 代码从早期面条式走向**一切皆插件**：可替换 memory、加 wiki、加 dreaming，像 **Linux** 装组件，**不必**所有东西都进上游 PR。

**详细内容**：

Peter Steinberger 表示维护者仍被 PR 淹没，插件化可让用户自建分支生态。
