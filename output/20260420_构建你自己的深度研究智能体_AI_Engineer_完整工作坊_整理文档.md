## 全局核心提要（Executive Summary）

> **AI 导读**  
> - **问题**：通用聊天模型直接写 LinkedIn/技术内容易出现「AI slop」、幻觉、过时信息与空洞套话；高质量 AI 工程教学内容生产成本高、需要工程师+写作者+编辑协作。  
> - **方案**：拆成两套系统——**探索型深度研究智能体**（MCP + 工具：联网检索、YouTube 分析、汇总为带引用的 `research.md`）与**约束型写作工作流**（大 system prompt + 档案 + few-shot + 评审–编辑循环），顺序脚本串联、弱化编排。  
> - **实现要点**：MCP 暴露 tools / prompts / resources；FastMCP 本地跑；Claude Code 等作 agent harness；可选 **Skill** 替代冗长 MCP prompt 以实现渐进式披露；**Opik** 做线程/追踪与评估实验。  
> - **写作侧**：指南 `guideline.md` 结构化用户意图；结构/术语/人物三类 profile；评审者与写作者**不同上下文**；结构化（如 Pydantic）评审意见；冲突时优先级 **指南 > 研究 > profile**。  
> - **评估**：自建 pass/fail 数据集 → LLM Judge 对齐专家 → dev/test 上 F1；小样本易「满分」过拟合，需扩大样本并在在线轨迹上复验。

## 开场：LinkedIn 式「AI slop」与工作坊目标

**核心观点**：幻灯片用典型「ChatGPT 式 LinkedIn 帖」说明问题——不仅是禁用词式的 slop，还有**泛化措辞**（如反复「大多数团队/人/项目」）、**幻觉**、**过时信息**（举例称某稿写于上周却仍写「GPT-4 为最先进」——此为讲者举例语境）、套话（如「rapidly evolving」「不是 A 而是 B」），以及**整体浅薄无增量**；解决路径是**严肃研究 + 严肃写作**，而不是单次聊天。

**要点提炼**：

- Towards AI 大量生产 AI 工程课程与视频，需要资深工程师参与写作与审稿，**成本高**；团队尝试将「深度研究 + 技术写作」流水线**部分自动化**，并曾用该流水线**生成一门教别人搭建该系统的课**，以便迭代与学生反馈。
- 本次两小时工作坊压缩为：**更小的深度研究示例 + 面向短内容（LinkedIn）的写作智能体**；前约 30 分钟偏幻灯片，之后同事走代码；**公开 GitHub 仓库**（简介与 QR 码，具体 URL 以视频为准）。

**详细内容**：

讲者展示「像 LinkedIn 今年泛滥的那类」生成内容：除 **AI slop 用词**外，还有排版/行距问题、**高频泛化主语**（most teams/people/projects）、**幻觉与过时陈述**、以及**无实质信息**。结论是需要 **proper research and proper writing**。Towards AI 的动机是降低**高质量技术内容**成本，同时承认**故事性、幽默感、与读者共鸣**仍高度依赖人类，系统定位偏向**写作者增强**而非完全替代。

## AI 工程约束：自主性滑块与 Workflow / Agent / 多智能体

**核心观点**：AI 工程师的决策受**单任务成本**、**延迟**、**质量**、**数据隐私**等约束；从 prompt 到编排、自研评估构成「栈」。可用**自主性滑块**理解：**自主性越高，控制越弱、成本往往越高**；应优先最简方案。多数客户口中的「智能体」实际常可落成**简单可预期的 workflow**。

**要点提炼**：

- **Workflow（工作流）**：在基础 LLM 上增加数据、**工具**、**记忆**；可链式 prompt、路由、并行、基于评判的循环——仍**不必**是 agentic。
- **Agent（智能体）**（讲者采用 Anthropic 式表述）：需**采取行动**且能**随环境反馈调整**；能规划、选工具，且 importantly **决定不用哪些工具**。
- **何时用 workflow**：步骤固定、顺序不变（例：**工单系统**——收票、分类、路由、草拟回复、按政策校验、发送）；做成 agent 只会增加开销。  
- **何时用 agent**：需**分支**、调不同 API、写库等动态行为（例：加拿大某 CRM **营销内容聊天机器人**——客户最初要「多智能体」以迎合资助申请，实际需求是固定顺序：**规划 → 按客户取数 → 生成 → 校验修复**；团队采用**单智能体 + 多工具**，工具可自带 system prompt、校验逻辑甚至独立 LLM 调用）。

**详细内容**：

简单决策树：**模型已内知** → 直接 prompt（可加 few-shot）；**上下文可贴进 prompt**（量级约 **20 万 token** 内）→ 粘贴并可用 **context caching**；**上下文事先未知** → 检索注入（私有数据/新事实/领域知识）；需按条件分支 → **预定步骤的 workflow**。CRM 案例中，多智能体切分会**割裂上下文**、增加调用错误；**工具即专家能力**，全局规划留在单一 planner agent。

## 上下文腐烂与委托：何时多智能体

**核心观点**：Agent 上下文包含 system prompt、工具 schema、few-shot、检索结果、对话历史等，**随步骤增长性能下降**即 **context rot**；往往在远未达到百万 token 上限前就会恶化，讲者提到约 **20 万 token** 后明显变差，并与 **lost in the middle** 训练方式有关——模型被训练成「长文中找单一事实」，而非真正综合全书。

**要点提炼**：

- 管理手段：**裁剪、摘要、按条件检索**；讲者推荐阅读 **Claude Code 泄漏的 compaction** 等压缩技巧。
- **委托**：把子任务交给**工具**或**子智能体**（各自上下文），多数 **harness** 如此设计。
- **多智能体**：常见触发是**工具极多（如 20+）**或上下文过大；也可能因**自主决策分工、工具差异、合规**（如医院本地部署）——讲者曾从事医疗场景。

**详细内容**：

真实 AI 产品往往**混合**工具、workflow、固定与灵活模块；**深度研究**被作为整合这些技术的综合学习项目。

## 深度研究系统定义与 Towards AI 的产品动机

**核心观点**：**Deep research** 是规划研究、具备自主性、带网络/工具、**可靠引用**、含自我与人类反馈的推理系统；**目标驱动**，只给研究主题而非逐步指令；比聊天更接近「替人做文献式功课」：搜索、检视、转向、综合、迭代。

**要点提炼**：

- Towards AI 的目标是端到端产出**可运行代码、有用配图**的技术文章/课程稿，而非随机插图。
- 挑战：**高精确率与高召回**的平衡（上下文有限）、抑幻觉与 slop、写作侧融入人类反馈。
- 是否自研：市面 deep research 工具团队**每天用**，但常**信息过载、噪声大**；故自研以贴合流程；定位仍是 **augment 写作者**，因「与人连接」的故事与幽默仍难完全自动化。

**详细内容**：

讲者强调从项目伊始用**问题清单**驱动设计（是否值得做、有无现成方案等）。结论：**值得做**，因精品技术内容**昂贵且难规模化**；研究要「宽」，写作要「严」，二者架构需求冲突。

## 架构决策：研究灵活 vs 写作约束、顺序交付与工具栈

**核心观点**：**研究**要探索、迭代、转向；**写作**要语气、结构、禁词与格式——故拆成**两个系统**：探索型研究智能体 + 确定性写作工作流。二者**无需复杂编排**：实践发现用户要么**同时用两套**，要么主要在最后用写作侧微调；大改则**整轮重跑**；研究先做出较完整 `research.md` 再交给写作。

**要点提炼**：

- **通信方式**：两智能体同仓库、**共享文件**（如 `research.md`、`guideline.md`），用**简单顺序脚本**连接，便于 **AI 辅助编程**（Claude Code 等）维护。
- **研究行为**：依赖人写的 **guideline**（主题、想覆盖的点、**推荐链接**）；**约束越细、自由越少，效果越好**（承认存在创作者偏向）；需 **抓取链接网页、消化 YouTube、读 GitHub（含私有，token）**，再补全网检索，汇总为终稿。
- **工具选型（口述）**：网页抓取用 **Firecrawl** 等支持动态/反爬站点；搜索曾用 Perplexity，现多用 **Gemini 带 grounding** 得可溯源答案，再按需深抓；YouTube 用 **Gemini 直接吃 URL** 提问或要逐字稿，避免自行下载转写；GitHub 用 **gitingest** 类库转 Markdown；框架层 **MCP + FastMCP**，用 **MCP prompt** 给「配方」；依赖管理 **uv**；模型侧以 **Gemini** 为主（YouTube 能力、Pro/Flash 按任务切换、对学生有免费额度），但强调须与 **OpenAI 等**对比选型。用户交互步骤写在 MCP prompt 里，通过文件与研究者协作；README 链至 **cheat sheet**。

**详细内容**：

[Samridhi 部分] 设计哲学：**智能体 = 大脑**负责推理与补全研究缺口；**MCP 服务端**暴露能力与数据。三大工具：**deep research**（Gemini、grounded 搜索，结构化 URL/标题/snippet）、**analyze YouTube video**（URL 作 file URI，模型**逐段「看」视频**而非仅用现成字幕，故单次常需 **2–3 分钟**）、**compile research**（读 `memory` 文件夹写出 **`research.md`**）。前两者写入 **`memory/`** 便于日志与调试。实现上工具注册需 **名称、参数、描述、参数 schema、实现**；deep research 内建 prompt 强调**权威来源与明确引用**。资源层暴露模型名、版本等；`.env` 配 **Google API key**、可选 **Opik**。

## 连接 Claude Code：mcp.json、本地 stdio 与演示

**核心观点**：项目根目录 **`mcp.json`** 声明服务器：本示例用 **`uv` 运行 `fastmcp run` + 服务端脚本**，**本地 stdio** 与 Claude Code 通信；若换 **托管 MCP**（如 Notion），则把命令换为 **URL**。Claude Code 同时是 **LLM** 与 **MCP 客户端**。

## Agent Skill：渐进式披露与 prompt 的取舍

**核心观点**：**Skill** 以紧凑 front matter（名称+描述）被加载，**运行时**再展开全文，结束后从上下文卸掉——**渐进式披露**，避免 MCP prompt 长期占满上下文；便于团队**版本化、分享、维护**。团队可将原 workflow prompt **迁入 Skill**，但仍保留 MCP 服务端以承载复杂逻辑与工具。

**要点提炼**：

- 描述字段要写清，便于路由匹配 query。
- 与「仅用 prompt」对比：prompt 常持续占用上下文。

**详细内容**：

讲者展示项目内 **research skill** 复用工作流说明；Claude 中 `/skills` 仅见 front matter。现场对话提到在复杂服务端组件很多时仍保留 MCP 而非全盘换 Skill。

## LinkedIn 写作工作流：三阶段与可控生成

**核心观点**：写作是**静态步骤链**，**不必**用 agent（若加也只局部）；三阶段为：**（1）拼装上下文 → 大 system prompt**；**（2）LLM 出初稿**；**（3）评估器–优化器循环**（多轮评审与修改）。输入包括 **`research.md` + `guideline.md`（用户意图）** + 静态 **profiles** + **few-shot**。

**要点提炼**：

- **guideline** 用模板约束：主题、角度、要点、叙事线等——**每条帖唯一主要人工编辑点**。
- **三类 profile（Markdown，数百行级）**：**结构**（篇幅、hook/正文/CTA）、**术语**（主动语态、禁用 slop 词如 delve/tapestry/vibrant 等）、**人物/声音**（如绑定 Paul 个人简介与写作习惯）。
- **few-shot**：示例宜少而精（讲者用 **3 条**自称「猜的」）；从多例裁剪到仍能工作的最小集合，以控制成本、延迟与性能；仓库提供约 **20 条**高互动帖作数据集；**微调**可减少 system prompt 长度但摩擦大。
- 演示：生成帖粘贴某 **slop 分数检测器**，结果 **非 slop** 且分数优于对照人类稿**（单次演示，非保证）**；强调仍带 **LinkedIn 语域**，但避免乱用 em dash 等。

**详细内容**：

另有**小图生成**：写作工作流出文后，再调工具**自动生成配图**（投入不多）。运行可走 **MCP + write post skill**，示例输出 **v0…v4 多版本**帖文与 guideline；**make** 目标用于本地端到端自检。

## 评估器–优化器循环：评审、编辑与优先级

**核心观点**：**评审者**与**写作者**使用**不同上下文窗口**，减轻「偏爱己稿」偏差。评审对照 **guideline、research、profiles** 查**依从性与幻觉**；编辑常由同一 writer 模型兼任。循环 **3–4 轮**；保留目录下**多版本**供主观挑选。用**固定轮数**而非单一分数阈值——因创意质量**难客观量化**，阈值易噪声。

**要点提炼**：

- 评审输出为**结构化对象列表**（讲者以 **Pydantic** 为例）：字段可含 **profile 类型、位置、评论**，便于编辑精准修改；经验上**结构化输出优于自由文本喷评审意见**。
- **评审冲突优先级**：**指南（用户意图）> 研究事实 > profile 风格**。

**详细内容**：

示例对比 **v0 与四轮后**：首句更 punchy、结构更符合 LinkedIn 预期；示例见仓库 `examples`。

## 可观测性：Opik 的 Thread 与 Trace

**核心观点**：仅靠终端日志难以调试 agent/workflow（尤其「thinking」隐藏信息）；需要平台捕获 **LLM 与工具调用**、**输入输出**、**元数据**、**延迟与费用**；并沉淀轨迹供评估。

**要点提炼**：

- **Thread**：一次端到端任务（如写帖+配图）可含 **数十条消息**，像对话线程。  
- **Trace**：子调用粒度；例如 **generate post** 下可展开各模型费用、延迟。研究侧 thread 聚合多次 deep research 与最终汇总；单 trace 可对应**单次工具内一次 LLM 调用**。

**详细内容**：

团队对**研究**与**写作**均接入 **Opik**（口述拼写为 Opik）。

## 评估体系：数据集、LLM Judge、F1 与过拟合

**核心观点**：「vibe 检查」无法扩展到百篇；prompt 改动**牵一发动全身**，需要**评估层**。评估用途分三层：**优化**（每次改动看分是否提升）、**回归**（新功能不破坏旧表现）、**线上**（对真实流量告警）。

**要点提炼**：

- **数据集**：从真实 LinkedIn 帖出发约 **20 条**（工作坊规模；讲者认为生产宜 **≥100**）；**反向构造** guideline 与 research（用深度研究补材料），再用**真实写作流水线**生成待评帖子——**不要让 LLM 直接伪造最终输出标签**。人工 **二值 pass/fail + 短评**，**见第一处错即标 fail** 以降低标注噪声。划分 **train/dev/test**。
- **LLM Judge**：输入为**生成帖 + guideline + research + profiles**；输出 pass/fail + critique；**train split 作 few-shot** 可让 judge 的 system prompt 极简。Judge 本质是**二分类器**，可与专家标签对齐。
- **校准**：在 **dev** 上调 prompt 与示例使 **F1** 最大；再跑 **test**；若 **dev 极高而 test 崩**则**过拟合**。演示中每 split 仅 **5** 条故 F1=1，讲者明确**扩大样本后分数会下降**；在线模拟中曾出现**全判 pass** 与人工标签不符，需**把失败例收回 dev** 再迭代。

**详细内容**：

平台 **Experiments** 视图对比 dev/test；界面可悬停看 critique。配置用 **YAML** 将各文件链接，便于复现。
