## 裂口何来：工程世界与知识工作的结构差异

### 💡 核心洞察

> 裂口不仅来自话术对立，而是**工作流与技术集合本身不同**：工程的**高技术强度、工具的自主可控、结果可验证、模型偏好代码**，与多数知识岗位的**更低技术成熟度、严重碎片化的数据与更陈旧系统**，构成难以短期弥合的 **gulf**。

### ✍️ 深度解析

**Aaron Levie** 列出工程侧常有的一组优势：**技术能力极强、对互联网上发生的事极度敏感、自由选择并调试自己的工具、系统坏了能自己修**；模型擅长代码、工作可被验证，因而 **five or ten things** 使 Agent（含会用计算机环境的 Agent）在工程或至少是「类初创」环境里成立。

但一旦离开这一语境：**工作流不同、用户更不技术化、数据更碎、遗留更重**—— diffusion 需要时间，从硅谷与全球科技公司圈外溢到更广泛知识劳动，不会是「几周部署」量级。

## 自上而下失败：董事会、咨询与「95% 失败率」到底在说什么

### 💡 核心洞察

> **Martin Casado**：凡「互联网级」的长期趋势通常**自下而上**先在个体层面发生；大企业却习惯**中心化决策**。故「大厂 95% AI 努力失败」若按字面全盘接受是**草率**——很多人已在日常里用 ChatGPT。**真正失灵的是另一类项目**：董事会喊「多上 AI」，CEO 转手交给咨询公司做**中央集权式攻关**，不与运营对齐，**注定要摔**。

### ✍️ 深度解析

**Martin Casado** 还补一刀：**董事会想要什么 vs 第一波失败留下的瘀伤**，让企业在第二轮投入前需要额外勇气；近几个月终于有了些**试探性（tepid）**的企业落地，但整体仍谨慎。

自上而下项目另一经典坑——被塞去「公司最疼的单一系统」，而一线 IT 知道那是**最差下手点**：**Aaron Levie** 与 **Martin Casado** 都暗示，这种「挑**最尖锐痛点**给 AI」与组织政治绑定的立项，常与真实改造顺序相悖。

附带一条「约 5% 的问题」：**Steven Sinofsky** 回忆旧时工程扯皮：选型、架构路线能吵数月；如今 **AI 范式切换更快**（Agent 放在哪里、是否托管、工具链、云或本地），**实验室彼此 leapfrog**，企业架构组会问「我该押哪匹马」——过去几年曾选错路线的记忆，加重了**决策瘫痪**。

「快」本身削弱了技术扩散进入核心工作流的速度：看似吊诡，却与仍需完成的**基建、数据、系统升级工作**并行存在——CIO 们经常被问到：核心 chat、Agent 编排底层到底选哪条路，答案是「我们正激烈辩论两三个范式」，而且**几乎每个客户都这样**。

## 架构转折：从「AI 即软件」到「AI 即用户」

### 💡 核心洞察

> **Martin Casado** 提出近两季产品圈里极关键的心智迁移：与其把 AI **焊进界面**做成「混合型产品」，不如把产品线想成 **CLI/可被调用的能干活的工具**，**让外面的 Agent（如 Claude Code 一类栈）来当操作者**——**不是融合，而是服务「机器用户」**。

### ✍️ 深度解析

这等于**同一时期两次重构**：先是从纯产品跃迁到不成熟的「AI + 软件」拼贴；再跳到 **Agent** 作为主语的消费方式——行业里大量团队正在 **speed running**「云演进史」：**Steven Sinofsky**、**Martin Casado** 以早期「杂糅远程桌面等各类过渡形态类比」：**还没抵达终局形态，却已多次拆掉重建**。

**Steven Sinofsky** 提醒硅谷观众：大企业说「要先画出多年赌注地图」听上去「蠢」，但如果你职业轨迹是数家两年一换的创业公司，你不会背负「选一个要用四十年的应付账款系统」那种**长尾责任**——这种时间尺度差异，解释了为何「押错 horse」的恐惧真实存在。

## 集成墙：代理也会像人类一样被转接、被拒访

### 💡 核心洞察

> **Steven Sinofsky** 祭出 **Gilfoyle** 式工程师梗：理想极客**不想跟任何人说话、只写代码**；企业里也有人拿模型「自己玩」。但人人都会撞上**集成墙**——**千人员工、十年历史的公司，就是一堆待集成的遗产**；**AI 并不自动帮忙集成**。就算把 Agent 设计成「像用户登录」，仍会像打进客服却被反复转接：支付与预订不是同一个窗口、经理才有权看到的内容，**人类靠组织内网络补位**，**Agent 不知道如何拍肩膀问 Sally 或Bob**。

### ✍️ 深度解析

**Aaron Levie** 延伸访问控制：**不同人类角色背后是不同权限**；若 Agent「抄近路」绕过人工流程，安全与合规模型立刻报警。多数遗留环境**并非最权威的访问控制层**；人类日常靠「嘿 Sally 分享给我」「Bob 你系统里的数是多少」——若 Agent **只继承你的权限**，会在每道门前停住；又不像人会回头找人，于是**卡在错误数据、过时文档或非权威管道**上。

这与 **OpenAI / Codex 与大型系统集成商合作**的新闻相关：**Martin Casado** 反对冷嘲——**大型企业必然需要变更管理、系统实施与集成**，「要人来装『会自动化人的 Agent』」并不反讽，而是**几十年来企业技术变革的常态**；这类服务商会忙**几十年**。

**Aaron Levie** 给落地策略分歧：企业创业若只依赖「已无头化的 SaaS」会撞墙——更稳的**第一刀**往往是**先做信息聚合与揭示**：Agent 能不能先**搞清组织里发生了什么**；历史上 **Web 第一次极有价值**时是「给人看东西的入口」，企业内亦可先让 **learning across files**、跨域检索真正有效，再上「批准 / 驳回」一类行动按钮。**Steven Sinofsky** 暂时站在「先情报、后动作」一侧，但仍留话头给反方——于是进入下节。

## 「像人一样接治理」对「像软件一样接 API」——两种心智

### 💡 核心洞察

> **Steven Sinofsky** 提出与「集成论」并行的一条 **end-to-end argument**：**LLM 非确定性、能处理长尾 messiness**，这些原本也是人的特征——人类花四十年堆出界面、流程、访问控制。**若你把 Agent 当新员工：发邮箱、按人的方式要权限、遵循人类流程**，反而会**搭上**整套为人设计的治理，而非硬与「软件集成范式」较劲。

### ✍️ 深度解析

**Aaron Levie** 愿意部分同意，但强调**人有的免费上下文 Agent 没有**：组织里**该拍谁肩膀、未写进文档的暗知识**。**Agent** 的巨大优势是可**无限并行**；巨大劣势是**不知道找谁**。于是 **Martin Casado** 开玩笑说支持「**Agent onboarding**」—— orientation、CEO 文化演讲、各部门路演——直指：Agent **熵太高、太 unruly**，最终仍要借用**管人的那套**，而非只造「漂亮索引数据库」。

**CES 电梯段子**（**Martin Casado**）：矮机器人进电梯够不着按钮，同公司又卖「按电梯专用外设」——因为**电梯没有可接 API 的「无头版」**，没有「Wi‑Fi 代按楼层」——类比企业里大量系统本质是**为人类手点而造**。**Steven Sinofsky** 点题：世上两类系统——**给人**与**给软件**——而 **Agent 更像前者**，应尽可能**挂靠人类通道**。

## Salesforce 无头化、SaaS 末日论再踩一脚刹车

### 💡 核心洞察

> **Salesforce** 高调走向 **headless**，希望被各类 **Agent** 广泛使用；**Aaron Levie** 视其为**风向标**——Salesforce 往哪走，大量企业软件会跟；同时逼出**新计费模型**：小额 **API tax**？**Agent 席次**？**Martin Casado** 个人马上能列出 **5–10 个**若 CRM 真无头化会立刻爽用的客户情报场景（进城见谁、会议前扫一遍关系数据等）。

### ✍️ 深度解析

**Martin Casado** 强调 **Agent 也是实体（entity）**，需要**身份与许可证**——也许计价不同，但**就是另一张席**；访问边界不能高于委托它做事的人，否则「我去雇一个超能 Agent 看我本不该看的」会爆炸。

**Steven Sinofsky** 从 **IT 架构**补刀：你不能等 **LLM 吐出话**再事后裁决合不合规——输出不是 **SQL 行**那样可机械套规则，而是**句子里跳数字**，治理必须前置。由此 **SaaS apocalypse（SaaS 末日）** 叙事在他眼中「**比原来更蠢**」——会是**席次爆炸**；也许有人发明 **read-only 前几年廉价**等聪明计价，但**「蹭人类账号」从安全上就是坏实践**。

**Martin Casado** 再抛 **OpenClaw + Mac mini** 例子：很多人用 Mac mini 跑，**第一原因**是 **iMessage 没有无头接口**——为了**集成真实世界**；第二，**headless 浏览器**常被站点反爬策略识别而失效，于是需要**真 Safari**。结论：**「一切无头」假设整个互联网先无头化**，而真实训练数据来自**人点非无头 App**——模型只会越来越像**真用人方式**操作。

## API、浏览器 Computer Use、反爬与现实工作负载

### 💡 核心洞察

> 双方并非完全对立：**API 在可调用时更省**；但真实世界大量动作没有干净 API（**Zillow** 房价例子：**headless 被识别，真浏览器才行**）。长期可能针对 **MCP / 各 SaaS API** 做更严谨训练，但眼下采用路径往往是：**有 MCP/CLI 就走接口，没有就云上或本地弹出真实浏览器会话**。**Aaron Levie** 自信会「**用力敲 Salesforce API**」做只读查询；**Martin Casado** 则争辩**产品 API 与人的操作路径映射并不一致——站在电梯口见客户前几秒人最需要的信息流，不见得是现有开发者 API 那张脸**。**Box** 的实践：新版 **Box Agent** 可跨库 **fan‑out** 多条查询并自 **rerank**，**不应被人手点搜索的 SLA 限速**——与人形机器人仍需走进电梯按下按钮这类物理约束相对照，数字世界**可并行**但并非没有产品化与公平的访问设计问题。

## 规模、熵与 AI 编程：加速器也是麻烦制造机

### 💡 核心洞察

> **Martin Casado** 反问：若每名员工各跟一个 **Agent**，而每个 Agent **以人类五百倍频次**击打同一套 SaaS，**后端与网络会否先崩**？这不是 token 计价问题，而是**吞吐与现实架构**。**Martin Casado** 本人先以「缓存只读、我们经历过互联网」回应；随即 **Steven Sinofsky** 把话锋扭向更刺的点：**用 AI 写代码，代码会随时间**明显变差**——熵增，**引入的问题与解决的问题一样多**——组织级长期跑 Agent 的持续任务，我们是否真会治理？

### ✍️ 深度解析

**Steven Sinofsky**、**Aaron Levie** 共同指向「**大公司或其事业部每一分每一秒都感觉要散架**」的体验：靠**层层护栏**才不爆；这是为什么「一股脑 vibe coding」经验在巨企行不通——也呼应 **Gilfoyle** 式无协作天才无法在任何大型组织协调生存。

**Aaron Levie** 分享 **Box**：不对外吹 **10×**，内部强调护栏；**八成到九成代码由 AI 写某新功能**，发布仍被**完整安全评审**拦住以防注入风险——速度被下游流程限速；体感 **整体也许 2–3×**，且**代码审查与上线流水线**整条仍在重塑。更广泛知识工种（会计等）：AI 可先扫海量数据抓异常扩大**可见度**，但最后 **逐笔核对关键数字**仍高度依赖人——CEO / 董事会须在噪音里识别**哪类生产力真的划算**。

**Steven Sinofsky**「量 token KPI」插刀：**Aaron Levie** 转述某大公司员工让 Agent **跑无用任务**只为刷量——**you get whatever you measure**，假忙碌与可疑产物随之而生。

## 工作会不会被干掉？从历史类比到「更多软件＝更多事要做」

### 💡 核心洞察

> **Aaron Levie** 仍对就业**极度乐观**：凡声称「人完全出圈」的理解大多跑偏——过程仍需人**发起、审阅与纳入真实世界**。**Martin Casado** 拿出 **1990 年代《The End of Work》**：出版几乎撞车互联网井喷，「技术没带来生产率却将消灭就业」云云如今听来荒谬。**Steven Sinofsky**、**Aaron Levie** 串起：当年以为计算机会干掉会计，结果会计工作**更复杂、更综合，岗位反增**；**信息愈多，消化与行动的人手也愈多**——公司本质是在**对信息行动**。

### ✍️ 深度解析

**Martin Casado** 用 **Mark Andreessen**「**软件正在吞噬世界**」收束：未来不是不需要工程，而是工程人才分布到 **John Deere、Caterpillar、Eli Lilly** 等所有产业——**更多软件、更多自动化需要半专家级的人**来提示、审查与运维 Agent。

**Steven Sinofsky** 举起 **1981 年《时代》杂志**谈「计算机消灭办公纸质流程」；法律业从**禁止课堂用笔记本**到今日**离开 track changes 几乎无法做交易**——律师人口大增，业务是「计算机化」而非被计算机替代。

**Aaron Levie** 提醒：许多专业工作**大量上下文在真实世界**，不是「打开 Word 就能完成」；AI 增加初稿与信息处理，却可能**提高**你去找专家核实的次数。
