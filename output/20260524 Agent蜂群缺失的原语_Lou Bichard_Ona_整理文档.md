## 核心导读

> **全文论点**：Agent 蜂群基建的运行时与编排已解，**协调层**——接任务、验关 SDLC 阶段、再前进——才是行业重复自研却仍未标准化的缺失原语；**GitHub** 不是答案，下一项原语可能是任何编码 Agent 都能调用的 **CLI 网关**。

**重复造轮子。** **Stripe** 叫它 **Minions**，**Ramp** 叫它 **Inspect**——两家都为后台 Agent 舰队从零自研内部基建。**Lou Bichard** 认为这种事不该每家重做。

**三根支柱。** 基础设施可拆为：**运行时**（大体已解决）、**编排与触发**（已解决）、**协调**（尚未解决）。协调层负责 Agent 间传消息、确认是否通过生命周期某一阶段；**GitHub** 为人设计、噪声极大，难承载并行 **数百 PR**。

**Ona 与展望。** **Ona** 以 **VM 级隔离** 承载 **Fleet** 跨仓库自动化；协调还可能经**状态机**、**持久化执行**与类 **n8n** 工作流落地，但须回答门禁与合规——社区两周公开建造与 **CFP** 正在试探行业共识。

## 开场：会议尾声与「缺失原语」

### 核心洞察

> **行业重复自研舰队基建；真正缺的是让多 Agent 按 SDLC 阶段协作、验关、再前进的协调原语。**

### 深度解析

**Lou Bichard** 在 **AI Engineer** 大会尾声登台，笑称不知对听众是「可以开始周末」还是「舍不得会议结束」。本场主题：**Agent 蜂群（agent swarm）缺失的原语（missing primitive）**——在**编码 Agent** 语境下谈**子 Agent**、**蜂群**及其**底层基础设施**。

他自我介绍为 **Lou**，任职 **Ona** 现场 CTO。此前是首席工程师与平台工程师，做过产品管理，现偏客户现场。**Lou Bichard** 观察到：虽非人人已到这一步，但过去数月「**软件工厂**」占据他大量思考；会上也有多场类似演讲，试图把编码 Agent 铺到**整个软件开发生命周期（SDLC）**。

他还创建了 [backgroundagents.com](https://backgroundagents.com)，用可视化归纳后台 Agent 模式，反响不错，建议听众查阅。

### 对谈实录

**Lou Bichard**：「编排层可以说已经解决，触发也解决了，但缺的是协调。」

## 软件工厂：把人类逐步移出 SDLC 环路

### 核心洞察

> **软件工厂是承诺让人类逐步退出 SDLC 主动操作，使工作理论上自动从开发流向生产——而非单人同时开很多并行 Agent。**

### 深度解析

**Lou Bichard** 在幻灯片中给出他对**软件工厂**的定义，并刻意与另一种流行说法划界。

他的定义是：**承诺在 SDLC 内逐步把人类移出环路（out of the 循环）**，使人**不必主动与计算机交互**。

他见过有人把软件工厂理解为「**一个 IC 同时跑很多编码 Agent**」——**那不是他的定义**。他的版本是：**慢慢把人拿掉**，工作从开发**理论上以自动化方式**流入生产。但他强调：我们处在软件工厂的**极早期**。

### 对谈实录

**Lou Bichard**：「我的个人定义是：你在慢慢把人拿掉，然后工作从开发流向生产——理论上以自动化方式。但我们处在极早期。」

**Lou Bichard**：「我的软件工厂定义：逐步把人移出环路，使人**不必主动**与计算机交互——不是一个人同时开很多 Agent。」

## 规模化三种模式：Swarm、Fleet 与 Events

### 核心洞察

> **Swarm 收束到单一产出；Fleet 跨仓库扇出；Events 用 Webhook 等把人移出触发环路。**

### 深度解析

**Lou Bichard** 用 [backgroundagents.com](https://backgroundagents.com) 上的图示归纳**大规模运行编码 Agent** 的三种模式。

**Swarm（蜂群）**：从**单一意图**出发，扇出给多个 Agent，再**收束**到例如**一个 PR 或一个任务**——类似常见的**子 Agent** 流程。

**Fleet（舰队）**：在组织内**跨多个仓库**扇出 Agent——**Ona** 已具备该能力一段时间，行业偶有出现，**Lou Bichard** 认为未来组织会更多采用。

**Events（事件）**：要建软件工厂，必须定义**何时、如何触发** Agent 上线。大量能力已存在：**Webhook**、**PR 创建**、**Linear 工单创建** 等。

## 大厂内部实践：Stripe Minions 与 Ramp Inspect

### 核心洞察

> **Stripe 与 Ramp 各自从零造舰队基建；证明需求真实，也证明行业缺可复用协调层。**

### 深度解析

过去数月，多家大公司公开其**内部 Agent 基础设施**实现，对 **Lou Bichard** 团队很有参考价值。

**Stripe** 的内部系统叫 **Minions**，建立在既有基础设施之上，接入编码 Agent 后可在 Stripe 内驱动**数千个 PR**。

**Ramp** 在社交媒体上极为高调；其内部系统叫 **Inspect**，同样是运行**后台 Agent** 的基础设施。

**Lou Bichard** 指出：**Ona** 做类似事情也已有一段时间——这不是单家公司的怪癖，而是**重复出现的工程模式**。

## Ona 平台：开发环境、Fleet 与典型用例

### 核心洞察

> **Ona 用可编排的 Fleet 在触发器下跨千仓执行「简单但规模大」的 Agent 任务。**

### 深度解析

**Ona**（口播亦称 **Owner**）作为平台，约 **6 年**专注**开发环境基础设施**；近 **1–2 年**更深地接入 Agent 侧。

平台可**任意 spin up 开发环境**；额外能力是 **Fleet**：按**计划或触发器**自动化 Agent，在**多个仓库**间解决问题。

典型用例包括：

- **CVE 修复**
- **提升测试覆盖率**
- **在规模上强制执行某类变更**

以当前 LLM 能力，这些任务往往**本身不算难**；难在横跨**大量团队**或**数千个仓库**。

实现上类似**工作流搭建**：添加 **提示词、脚本** 等，驱动组织级变更。**Lou Bichard** 称这已在产品界面中可视化配置。

## Harness Engineering：上下文工程的延伸

### 核心洞察

> **Harness Engineering = 用仓库内一切可反馈手段约束 Agent，跑通后把「迷路点」写回仓库形成闭环。**

### 深度解析

**Lou Bichard** 特别致敬 **OpenAI** 与 **Ryan**（**Ryan Lopopolo**）的 **Harness Engineering** 博文——把流程尽可能**编码进仓库**、**上下文文件**、**agents.md**，以建成软件工厂。

会上不少人问「Harness Engineering 是什么」。**Lou Bichard** 将其视为 **上下文工程（context engineering）的延伸**：仓库里从 **skills**、**agents.md** 到**单元测试**——凡能给 Agent **反馈**的，都算编排层（harness）。

他幻灯片一角的可视化表达：**先做事 → 让 Agent 跑 → 发现 Agent 在哪里迷路 → 把知识编码回仓库/上下文 → 再让 Agent 在软件工厂里尽量顺畅流动**。

这与 **OpenAI** 的 **Symphony** 项目、**Harness Engineering** 博文一脉相承：**Lou Bichard** 自己也在做「**不手写一行代码**、尽可能自动化、让 Agent 自主驱动开发」的实验。

## 三根支柱：运行时、编排与协调

### 核心洞察

> **运行时与编排（含触发）基本够用；协调——Agent 如何互协作、接任务、验关——才是今天自建工厂的最大难点。**

### 深度解析

从**基础设施**视角，**Lou Bichard** 将软件工厂所需能力归纳为三层。

**第一层：运行时（runtime）**——Agent **在哪里跑**。他认为**大体已是 solved problem**。

**第二层：编排（orchestration）**——**规模化运行**：水平扩缩、**await** 式调度、**触发机制**（何时启动 Agent）。他也认为**基本解决**。

**第三层：协调（coordination）**——Agent **如何互动**、**互相接任务**、**如何协作**。若今天自建系统，**最大困难**在这里。

**运行时实现谱系**（由轻到重）：

- 独立**线程**
- **工作树** 隔离
- **容器** / **微 VM**
- **完整虚拟机**——**Ona** 称此类为 **dev environments**；「沙箱」一词已使边界模糊

**Lou Bichard** 主张：要跑**真正的开发任务**，应在 **VM** 内执行，理由包括：

- **安全**：容器**不是**牢不可破的隔离边界；要保护 Agent 运行时，容器有挑战
- **资源**：容器在 **Kubernetes Pod** 上**突发（bursty）**，有**吵闹邻居**与**算力争抢**

只有 **VM 级完全隔离**，才适合**正经、规模化**地跑开发类 Agent 任务。

### 对谈实录

**Lou Bichard**：「容器**不是**牢不可破的隔离边界……只有 VM 的完全隔离，才能正经做好这件事。」

## 现场演示：进程级子 Agent 与 VM 级舰队

### 核心洞察

> **同一任务可走「单 VM 内进程子 Agent」或「父 Agent 再 spin 多台 VM」两条 UX 路径，子 Agent 通过消息回传父 Agent 治理。**

### 深度解析

**Lou Bichard** 播放昨日在酒店录制的演示（防现场 Wi‑Fi 不稳），展示 **Ona** 界面中蜂群**实际长什么样**。

界面左侧列出进行中的任务；他启动了两个示例：

**任务一**：实现 **Symphony**——仓库内有**极详细 spec**。**Lou Bichard** 要求用**基于进程的 Agent（process-based agents）**：在**单个 VM** 内运行 Agent，由该 Agent **再 spawn 子 Agent**（仍在同一 VM 内）。

**任务二**：运行 **Fleet**——Agent 被授权在平台内**创建更多 VM**，理论上可接近**云厂商扩容上限**。

演示中，**VM 模式**的父 Agent 控制整体，**子 Agent** 在左侧陆续上线，各领**小块上下文**与**子任务**，通过**消息传递**回父 Agent。**Lou Bichard** 坦言：**UX 如何管理越来越复杂的子 Agent** 仍是挑战。

进度上，两侧任务各有 **7 个子项**；子 Agent 上线、开工，结束后 **VM 会终止并收束**，任务完成。

**第二种 UX（进程级）**：子 Agent 在**单一 Agent 窗口**底部以**栈**形式出现；点击可打开**新聊天窗口**（近乎新上下文）。即：

- **全隔离 VM**：横向扩展蜂群
- **进程级**：在单个 VM 内跑子 Agent

> ⚠️ **[视觉盲区]**：**Lou Bichard** 演示了界面录屏与左侧任务列表动效，字幕未给出具体 VM 数量上限、耗时或成功率等数值；细节以画面为准。

## 自建软件工厂：SDLC 微步骤与确定性

### 核心洞察

> **教科书式五步 SDLC 对 Agent 太粗；要拆成可训练、可确定性执行的微步骤，否则 Agent 会跳步。**

### 深度解析

**Lou Bichard** 不仅卖平台，也**亲自尝试造软件工厂**——类似 **OpenAI** 做 **Symphony** 与 Harness 博文的路径。

技术「**今天已经在了**」，但仍有多重挑战。教科书里的 SDLC 是**粗粒度五步**；**Agent 并不尊重这种划分**——例如 **Plan** 阶段内部就有大量**微步骤**。

若要让 Agent **按 SDLC 逐步推进**，必须把 SDLC **拆解为微步骤**，且尽量**确定性**地执行——贯穿规划、构建、测试、发布全流程。

**最难的部分是上下文（context）**——下文专节。

### 对谈实录

## 上下文管理与上下文腐烂

### 核心洞察

> **上下文被占满后 Agent 会迷路、跳步、讨好式省略测试——协调层必须配合细粒度 SDLC 状态而非靠人脑盯 GitHub。**

### 深度解析

**Lou Bichard** 称上下文窗口是 working with LLM **最难的部分之一**；前一位讲者也谈到 **context rot（上下文腐烂）**。

当上下文窗口被消耗，Agent **逐渐失去方向**，效能下降，且**常跳过步骤**。模型**相当讨好（sycophantic）**——例如要求写测试，可能**少写几个**以「完成任务」。

这与后文**协调层**问题相连：人若用 **GitHub PR 洪水** 判断该介入哪里，认知负荷极高。

### 对谈实录

## GitHub 与 Linear：为何不是 Agent 协调层

### 核心洞察

> **GitHub 不是 Agent 的协调层；Symphony 建在 Linear 上也有同样问题——把人用工具硬套给 Agent。**

### 深度解析

一旦具备运行时与编排，仍缺**协调**。**Lou Bichard** 对 **GitHub** 作为协调层的判断极为直白。

Agent 可以开 PR、人可审查、可解合并冲突、可修 CI——但对人类而言**极其嘈杂**，难以判断**应在何处介入某一 Agent**。**GitHub 是糟糕方案**。

**Symphony** 基于 **Linear**，但** suffer 同样问题**：我们在**非常奇怪地**复用现有人类工具给 Agent 用。

### 对谈实录

**Lou Bichard**：「若你用 GitHub——GitHub **不是** Agent 的协调层。会变得极其不堪重负。」

## 协调层展望：状态机、持久化执行与 CLI 网关

### 核心洞察

> **协调层或将由 SDLC 状态机 + 持久化执行 + 本地/CI 可调用的 CLI 网关打包而成，并覆盖合规门禁。**

### 深度解析

**Lou Bichard** 认为行业**正处在解决协调问题的门槛上**，可能路径包括：

- **状态机 / 工作流**：把 SDLC 建成**状态机**——业界早有成熟思路
- **持久化执行（durable execution）**：多家公司已探索**可恢复、可持久**的长流程执行
- **门禁与合规（gates and compliance）**：也必须纳入
- **CLI 构造**：打包成可在**本地开发环境**运行、也可在 **CI** 等远端环境调用的 **CLI**

他总结：**运行时** 已解决（沙箱、容器等多选）；**编排** 与**触发** 已解决；**协调** 仍缺；**安全** 也是推动更多自动化的另一块拼图（现场有 **Ona** 安全团队成员）。

### 对谈实录

## 问答：CLI 形态、协议与实验生态

### 核心洞察

> **CLI 网关让本地编码 Agent 查询「本 SDLC 阶段是否完成」；图工作流需收敛为 Agent 可调用的 CLI，协议标准仍开放。**

### 深度解析

现场约 **2.5 分钟** Q&A（字幕至约 **18:16**）。听众请 **Lou Bichard** 回到协调层问题幻灯片，并追问**更具体的解决方案**（含 **memory** 相关 elaboration——字幕未展开 memory 细节）。

**Lou Bichard** 复述问题：协调层问题与**更具体的解法**。

内部有多种**原型**。一种有效形态是 **CLI**：

- 也有**基于图（graph-based）**的开源方案，用类似 **Mermaid**、**n8n** 式工作流定义提示词与步骤
- 但最终应**打包为 CLI**：本地运行的 **Claude Code** 或任意本地 CLI Agent，应能通过**工具集成**调用它，询问：**「我是否已完成 SDLC 的这一部分？能否进入下一部分？」**

他愿在会后展示**更完整 spec**。

业界也在用不同方式摸索：

- **n8n** 式**工作流图**
- 他自己的 **CLI gateway** 原型
- 多种**临时 hack**；**OpenClaw** 相关方在 **ACP** 上构建了某工作流组件——口播 **[ASR不确定：ACPX]**，他一时想不起全名
