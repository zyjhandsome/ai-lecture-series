## 首次公开：YC 为何现在谈内部 AI

### 核心洞察

> 投资人与被投公司之间的「共生」在于：孵化器必须亲自把 Agent、Harness 与组织大脑做出来，而不是只写支票讲方法论。

### 深度解析

**Gary Tan** 开场点明：过去几年 **YC** 主要投资 **AI** 公司，并随批次迭代「如何打造 **AI 原生** 产品公司」的建议；外界常忽略的是，**YC** 自身也在用同一套东西改造一家**远早于 AI 时代**创立的机构——而 **Pete Koomen** 是这项内部转型的牵头人。这是 **Lightcone** 第一次系统对外讲内部基建。

**Gary Tan** 将 **Pete Koomen** 介绍为 **Optimizely** 创造者，并强调其负责 **YC**「字面意义上」的全部 **Agent** 基础设施与 **Harness** 用法。内部实践与对外辅导的 **AI** 公司形成闭环：**Pete Koomen** 同意叙事应从约一年前、财务侧项目真正启动的时刻讲起。

### 对谈实录

**Gary Tan**：「这就像一个共享的组织大脑——最接近让我们彼此连接大脑的东西。」

**Pete Koomen**：「当你对安全没那么担心时，你会想：天啊，这些东西强大到不可思议。」

**Gary Tan**：「组织内就是这样诞生超级智能的……对你做的每一件事都这么做。」

**Pete Koomen**：「随着我们更会做 AI 原生软件，会更像是 Agent 包裹确定性的软件工具，而不是确定性软件包裹 AI。」

**Pete Koomen**：「我一看到 AI 被说成用来替代人，就很不舒服……它是要让我们做成更多事。」

## 起点：财务团队与「用英文写工作流」

### 核心洞察

> 旧模式是「业务口述 → 工程固化」；新模式是让财务用**自然语言提示词**自己编码工作流，而不是写 **Ruby**。

### 深度解析

约一年多前，**Pete Koomen** 与几名工程师正为财务团队规划工具：记账分录、记录定价轮次等——**YC** 历来大量依赖自研软件，这在非 **AI** 时代已是优势。

当时内部循环是：财务向工程师解释复杂财务流程 → 工程师做**确定性、封装完备**的专用软件 → 交还财务。**Pete Koomen** 同时在个人环境感受 **Agent 编码工具** 的爆发（**Windsurf**、**Cursor** 已成熟，**Claude Code** 刚出现），与 **YC** 古典交付方式之间的裂缝越来越大。

最初动机明确：**Pete Koomen** 希望财务团队能**自己掌控软件**，把工程师从「必须听懂每一道财务工作流」的循环里解放出来——不是用 **Ruby**，而是用**英文 + 提示词**。

**Gary Tan** 插入主持引子：两三年前 **LLM** 已能写 **SQL**，**YC** 曾投资过多家「自然语言查数」创业公司，不少并未跑通；但在 **YC** 内部，**Pete Koomen** 第一版就**真的好用**，让无工程背景的财务同事能问**真问题**——**Pete Koomen** 坦言自己也曾意外。

随后系统从财务专用形态**重写为更通用的 Agent 循环**——今天市场上已随处可见，但 **Pete Koomen** 的第一个「魔法时刻」是：**Agent 循环 + 共享工具注册表**，而注册表里第一个真正解锁能力的工具，来自 **Jared**。

### 对谈实录

## 只读 SQL：打破过窄工具边界的转折点

### 核心洞察

> 过度担心安全与隐私，才是阻碍 Agent 发挥能力的常见枷锁；适度放开只读库访问，往往立刻见效。

### 深度解析

**Jared** 做了两个工具：对生产库跑**只读 SQL**，以及读取**模型文件**。**Pete Koomen** 回忆自己有点像「违规」——起初工具域极窄，他因不够用而沮丧，于是设想：若给 Agent **完整访问生产数据库**、任其「践踏」会怎样？他**悄悄**在深夜推上线。

结果：**极其有效**。这与后续 **OpenClaw** 等实践同构——真正拖慢世界的，常是对「可能出错」的过度焦虑；个人机器上的 **Claude Code** / **OpenClaw** 几乎什么都能做，而公司环境却把人关在窄盒子里。**Pete Koomen** 的目标之一是缩小「上班」与「在家」之间的能力差。

为何只读 **SQL** 如此简单却如此有用？因为 **YC** 自研软件全部落在**一个 Postgres** 上：被投公司表、创始人表、财务交易、内部 **CRM** 笔记等——许多公司外包给第三方 **SaaS** 的上下文，在 **YC** 是自建并入库的。配上 schema 说明后，Agent 可回答关于业务的**任意**问题。

**Gary Tan** 确认：第一次看到时确实是魔法时刻——不仅更容易答，还**大幅**增加了**敢问的问题数量与复杂度**；旧日用 **BI** 工具问「过去四期里投过太空相关公司的投资人」可能要**数小时写 SQL**，不重要就不问。这是**杰文斯悖论（Jevons Paradox）** 的又一例：跨团队来回成本下降后，总需求量上升。

## 单一数据库：YC 的上下文垄断优势

### 核心洞察

> 对组织 Agent 而言，「单一 schema 的上下文层」至少不亚于工程上的单体仓库（monorepo）对编码 Agent 的价值。

### 深度解析

**Pete Koomen** 将 **YC** 进入实验时的结构性优势归结为：**自研软件 + 单一 Postgres**。编码 Agent 在 **monorepo** 里更高效；**YC** 的 Agent 在「什么都在一张库」上得到同样结论。

对更老、上下文散落在 **SaaS** 孤岛的组织，**Pete Koomen** 建议优先建设：**尽量把内部重要上下文收进数据仓库或统一层**；再配**内部工具注册表**——下文详述。

## 杰文斯悖论：问题越多，而不是越少

### 核心洞察

### 深度解析

当获取答案的摩擦从「敲数据科学团队门、排队 backlog」降到近乎零，人们会问**更多、更难**的问题——这不是效率幻觉，而是组织学习曲线变陡。

**Gary Tan** 补刀：2026 年仍有多数公司活在旧世界，「有点不可思议」；但这也意味着改造空间巨大。

## 为 Agent 反范式：大表、GBrain 与检索栈

### 核心洞察

> Agent 需要的是为检索优化的「大表」，而不是把企业所有 MCP 端点原样暴露；CLI 往往比 MCP 更顺手，但前提仍是数据形态要对。

### 深度解析

**Gary Tan** 把历史类比拉到 **Google Bigtable**：用「一张大表 + MapReduce」替代复杂 join 的时代，正在 **Agent** 侧重演——他个人用 **OpenClaw** 接多系统，再**反范式（denormalize）** 成与自己相关的 schema，供 **OpenClaw** / **Hermes** 类 **Harness** 检索。

**Gary Tan** 的 **GBrain** 栈包含 **RAG**、图 **RAG**、混合 **RRF**、重排序等——给 Agent「灵魂」+ 懂你关心的数据后，能力会「绕角」预判问题。对「数据到处都是」的抱怨，他的答案是：**必须反范式，放进为 Agent 检索与理解优化的格式**；可包一层 **MCP**，但他仍倾向先做好 denormalization。

这与 **Karpathy** 式知识 **LLM Wiki**、**GBrain** 的「梦境循环」Skill 改进并行——并可把 transcript 写回内部 **CRM**，沉淀对人/公司的认知。

### 对谈实录

## 单人 Agent 时代与「多人 Harness」缺口

### 核心洞察

> **YC** 基建的探索价值，在于哪些原语能让**团队与全组织**复用个人 Agent 的超能力。

### 深度解析

过去一年半，**Pete Koomen** 认为仍处**单人 Agent 时代**：**Claude Code**、**Codex**、**Pi**、**OpenClaw**、**Hermes** 等，都是**单人单机**超级个体工具——好玩、极强，但**组织级多人 Harness** 尚未被很好解决。

对成立超过两年的「遗产组织」，**Pete Koomen** 归纳可聚焦的三件事：**统一上下文层**、**内部工具注册表**、以及（在 **YC** 已验证的）**单一数据库** 路径。许多工具可把个人 **Harness** 接到 **MCP** 与其他真相源，但 **monorepo** / 单库的经验表明：**上下文集中** 的边际收益极高。

### 对谈实录

## 350+ 工具与共享注册表：把 Agent 变成「能在 YC 上班」

### 核心洞察

> 工具注册表是组织特异性的注入点；从约 20 个工具滚到 350+，每一团队都能把自己的工作流「工具化」。

### 深度解析

早期极简：**Agent 循环 + 简单工具注册表 + 模型路由**。注册表承载 **YC** 特有能力——把通用模型变成「能在 **YC** 上班」的 Agent。

最初约 **20** 个工具，含神奇的只读 **SQL**。此后每次发现某块工作可被 Agent 改进，就**加工具**——**Pete Koomen** 现场核对：**已超过 350 个**，各团队持续贡献。

举例：管理 **office hours**、财务记账分录、活动运营等。工具集中后，既服务内部 **Agent**，也可给个人机器上的 **Claude Code** 调用。

若 **Pete Koomen** 去其他组织，他会优先复制：**统一上下文 + 内部工具注册表**——并直言受 **YC** 实践启发，才有 **OpenClaw** 侧的 **技能化** 等元技能思路。

### 对谈实录

## 技能化、DRY、MECE 与解析器原语

### 核心洞察

> **Claude Code** 的 Skill 注册表、**YC** 的工具注册表、**OpenClaw** 的 resolver，本质是同一原语；上面再叠 **DRY** 与 **MECE** 检查，可避免 Skill 爆炸。

### 深度解析

**Gary Tan** 在 **OpenClaw** / **Hermes** 上实践 **技能化**（元技能：把任意顺手操作固化成 **Skill**），并强调要接到**解析器（resolver）**——类似 **agents.mmd** 列出 Agent 能做什么，再链到 **Markdown** 入口。

**Gary Tan** 常用流程：做一件新的、喜欢的事 → 说「**技能化**」→ 变成工具/方法调用 → 再跑 **检查可解析项**，对照已有 **Skill**/**工具**，要求**不重复（DRY）**且**互斥穷尽（MECE，麦肯锡幻灯片术语）**——模型似乎「懂」这些词；**DRY + MECE** 的 resolver 表被他认为接近最优。

**Pete Koomen** 感慨：像应用计算机科学家见证 **Unix** 时代发现栈与堆——今天大家在 **Claude Code**、内部 **Harness**、**OpenClaw**、**Hermes** 里**并行 reinvent** 同一套有用概念。

**YC** 侧 **Skill** 是工具之上的薄抽象；全员共享少量 **Skill**，并已进入**自主自我改进循环**——与 **Karpathy** 的 **AutoResearch**、**Codex** 纳入的 **SLG goal** 等同族。

### 对谈实录

**Pete Koomen**：「今天已有超过 350 个[工具]。我刚查过。」

## 梦境循环：夜间自我改进与两句描述 Skill

### 核心洞察

> 组织超级智能的针尖案例：一个看似微小的「两句公司描述」Skill，因会议 transcript 的 meta-prompt 而在一个月内超越任何单个合伙人。

### 深度解析

**YC** 有一个通用 **Agent**，**每晚**读取员工当日 **Agent 对话**，寻找本可做得更好之处，以及若提前拥有哪些上下文会更高效——即 **OpenClaw** / **GBrain** 的 **梦境循环（梦境循环（dream cycle））**；也可把 transcript 写回内部库。

共享 **Skill** 之一：帮被投公司写 **two-sentence description（两句描述）**——用任何人都能懂的自然语言说明公司做什么、为何有趣；看似简单，资深创始人也常忘写，因脑中上下文过满。

**Gary Tan** 教学框架：第一句——**我是否知道这是啥**？不知道就无法提问，只剩「跟电脑有关吧，午饭几点」。第二句——**是否值得我花时间**？若屋里已有 20 家做 X，必须说清为何**值得一提**，否则听众脑子回到三明治。

**Tom**（**YC** 合伙人）先手写 **Skill**；近一两个月，其他合伙人在春季批次 **group office hours** 让每位创始人练两句描述并给反馈——这些**活在合伙人头脑里的诀窍**，进入**会议 transcript**，再交给 **Agent**：「Given 你读到的上下文，改进两句描述 **Skill**」——结果 **Skill** 明显变好；**Pete Koomen** 认为现已**强于**任何单人。

**Gary Tan** 将其连接到 **Jack Dorsey** 在 **Block** 的叙事：把公司做成围绕「让人彼此付款」的迷你 **AGI**——**YC** 的两句描述只是成千上万 founder 服务中的一例，但机制是通用的：**写 prompt → 多人用 → 产生 artifact（含 transcript）→ 每日自动 meta-prompt 改进 → 该 Skill 超越个人上限**。对组织里**每一件事**重复此过程，即「在公司内部造超级智能」——并不更复杂。

**Gary Tan** 警告：不愿相信此道的在位者会把上下文锁死（「不安全」）；创业者反而应趁窗口期行动。不愿开放上下文的大公司与资源丰富却 **AI** 迟钝的 在位者，可能被「时间扭曲」甩开。

## 组织超级智能如何复合：对一切重复上述过程

### 核心洞察

### 深度解析

**Pete Koomen** 与 **Gary Tan** 共识：**AI 原生组织**的关键不是 **Copilot**（**Gary Tan** 称这很 **2023–2024**），而是把 **AI** 当作**一切工作的建筑层**，并**记录所有 artifact**——会议录音火爆，不只因为「教练式复盘」，更因为可反哺邮件、沟通、规划等全链路产出。

**Gary Tan** 引用 **Dario** 文章：部分 blocker 非技术，而是社会文化——两年前录会议还尴尬，如今 **Zoom** 上近乎默认；若把录音框架为「用同事集体技能让自己更强」，力量巨大。

**Pete Koomen** 强调：canonical 两句描述 **Skill** 不只是生成文案，更是让每人接入 **Diana**、**Harj Taggar** 等人多年练出的沟通直觉——**共享组织大脑（shared organizational brain）**，最接近「把脑子连在一起」的隐喻；还可与 **Agent** 做练习、被 critique。

### 对谈实录

**Pete Koomen**：「关键之一，不只是把 AI 当 Copilot 用——而是把它当作一切事物的基础设施层，并且开始记录所有产物。」

## 记录 artifact 与共享组织大脑

### 核心洞察

> 默认全员可查看任何全职员工的 Agent 对话，是用透明换安全、用示范换培训的杠杆。

### 深度解析

**Gary Tan** 指出一个细微但关键的决定：**Agent 对话默认对全体全职员工可见**——当初争议很大（「大家岂不是什么都看得见？」），最终保留开放；收益包括：从 **Gary Tan** 等高用量者的创意用法中学习（「原来还能这样用」）。

每条对话还**广播到内部 Slack 频道**，任何人可订阅观摩——在**高信任环境**里，这形成社会性约束，使「给 Agent 大量上下文」与「隐私」取得折中；**Pete Koomen** 认为在 **YC** 相当有效。

**Gary Tan** 总结：真正 **Agentic**、追求 **1000x** 的组织，必须具备**相对平等主义**与**信任默认**——多数公司两者皆缺；创始人若要做这种组织，须把它写进核心文化。环境上，**初创小团队高信任**最合拍。

### 对谈实录

## 信任默认、平等主义与 Token 时间扭曲

### 核心洞察

### 深度解析

组织还须在 **Token** 上愿意每年花 **约 1 万–10 万美元**（**Gary Tan** 口播亦提到极端可到 **百万美元** 量级，但强调价格曲线会陡降）：现在花 **10 万–100 万**，两三年后可能变成 **1 万**、再后 **几百**——所有人都会这么做；当下是**一次性时间扭曲**，可 leapfrog 所有 **Fortune 500** 与迟钝初创。

**Gary Tan** 类比九十年代公司给员工买电脑：贵、脆，但有电脑者对无电脑者是超能力——**Pete Koomen** 用「抬高地板（raising the floor）」描述 **YC** 内部感受：新人过去可能要六个月 ramp；现在通过 **Agent** 自动获得公司上下文，并**学徒式**观察明星合伙人（如 **Pete Koomen** 辅导销售、**Gary Tan** 给具体建议）——类似个人编码 Agent 让羞于问的「蠢问题」敢问，组织级亦然，**问题总量与 ramp 速度**双升。

### 对谈实录

## 抬高新人地板：《无马车厢》与聊天界面

### 核心洞察

> 未来 **AI 原生**软件更像「Agent 包裹确定性工具」，而非「确定性软件包裹一点 AI」；聊天界面足够好，因为最接近人类语言与思维的外化。

### 深度解析

**Pete Koomen** 在基建成型后写下 viral 文 **《Horseless Carriages（无马车厢）》**：批判当时大量 **AI** 软件仍把 **AI** 当小功能塞进旧壳（如 **Gmail** 写信助手）——潜力在于把软件**控制权从开发者移给用户**；开发者把 prompt 上下文锁死是「安全主义」，他反感。

结论方向：**Agent 包工具** 而非 **工具包 Agent**。**YC** 用原语向内部员工暴露了这一面，但路还长。

关于「是否需要全新 **AI UI**」：**Pete Koomen** 与 **Gary Tan** 都已转向拥护**聊天**——信任 Agent 做更多决策后，不必厚重 UI 逐项审批；偶发需要再生成**即时视图**。**Just-in-time software（即时软件）**：单页 **JavaScript** 当场为你生成，或固化成随时调用的 **Skill** 文件。

**Gary Tan** 曾怀疑聊天是否终局；反思后认为聊天最接近**清晰智能（clear intelligence）**的表达，硬塞固定盒子反而约束。**Telegram** + 语音备忘录等多模态也被认为「挺好」。

**Pete Koomen** 观察：最好用的 **AI** 软件往往**代码量极小**，只加最少先验，让模型发光——可写数万行，但上手理解成本极低。

**Pi**（极简开源编码 **Agent**）被 **Gary Tan** 点名为 **OpenClaw** 底座之一：可用 **Pi** 改 **Pi**，自指、自扩展软件；**OpenClaw** 走同路。商业软件未来或默认带「用 Agent 扩展」能力。

### 对谈实录

## 即时软件、Gary's List 2.0 与中心化之争

### 核心洞察

### 深度解析

**Gary Tan** 自述：一月至二月用 **Claude Code** 为 **Gary's List** 写了约 **50 万行** **Rails**（首周博客，后续一个半月搭 **Agentic** 框架：自研深度研究、事实核查等）——按 **2013** 年 **Web 2.0** 方式建站；最近三天为 **GBrain** 又写约 **4 万行**，相当于 **Gary's List 2.0** 开源版，含检索、语音抽取、事实核查等，昨日交给团队作 **OpenClaw** 实例 + **Telegram** + 同一检索数据 + **MCP**，告别臃肿单体聊天 UI 与 **OpenClaw/Telegram** 已解决的问题。

**Rails** 半百万行 vs **TypeScript** 约 **1 万行** + **Markdown** 约 **2000 行**——后者更动态；第二段想加政治家传记，在 **OpenClaw** 里不必改 **Rails** 或复杂 **eval** 基础设施，**主编 Agent** 用 **eval Skill** 即可现场改。

**Gary Tan** 将 **AI** 分叉为**中心化** vs **去中心化**未来（**18–24 个月**内抉择，或拖 **五年**）：**1984** 式——少数「国王」掌控最强 **AI**、算力、太空数据中心，且**不让用户改 prompt**（**Gmail** 式锁死放大到整个数字生活），如同只有主机没有个人电脑；**白丸**路径则像 **Homebrew Computer Club** / **Apple I** 时刻——**OpenClaw**、**Hermes**、**GBrain** 让你跑自己的软件、改自己的 prompt、私有仓库、自选模型（含开源权重）。

**ChatGPT** 十亿用户但 **MCP** 收紧；**Claude** 稍开放仍有限；**Perplexity Computer** 被 **Gary Tan** 评为较好但仍不及 **OpenClaw**/**Hermes**。**Pete Koomen** 反对把 **AI** 框架为「取代人」——他经历的是**赋能**；从主机到 **PC** 到互联网发布，主线是**个体赋能**，**AI** 将消灭大量「苦工式」工作，但须**主动选择**开放组织与开放计算——观看者是未来建设者，「我们最好选对」。

**Gary Tan**：「我们现在就在 Apple I 时刻——正在定义底层原语。」
