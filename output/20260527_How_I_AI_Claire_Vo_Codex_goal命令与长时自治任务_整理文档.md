## 核心导读

> **全文论点**：**`/goal`** 描述的是**结束时世界应是什么样**，而非下一步指令——**Codex** 在「工作 → 验证 → 决定下一步」闭环里自治运行，直到拿出可核验证据；强 Goal 须写可测量结果与约束，弱 Goal 与单行修改不该动用这把重锤。

**告别「下一步呢」。** 编码工具里最高频的「**好的，下一步呢？**」——**Claire Vo** 认为 **`/goal`** 让 **Codex** 自己跑完闭环，首次即跑出 **5 小时 45 分钟** 自治编码。

**三类实战。** **ChatPRD** 把 **Sentry**「无效编辑操作」打到 **零**；约 **3,900** 封邮件清到 **68** 封待人工（**3 小时 52 分**、约 **600 万 token**）；批量整理 **Linear** 播客积压——**Vercel API** 日志与 **Linear** 任务整理展示同一机制跨域复用。

**强弱 Goal 标尺。** 强 Goal 写可测量结果、验证方式与约束；「重构这段代码」类模糊产出、单行修改，都不该动用 **`/goal`**——工作方式正从「堆任务给我」转向「给自己下复杂副本」。

## 为什么需要 Goal：回合制提示的瓶颈

### 核心洞察

> **Goal** 把「再试一次、继续测、直到真做完」从你的人肉循环里卸载出去，交给 harness 自己转。

### 深度解析

**Claire Vo** 开场点题：时间线上那些「AI 通宵跑任务」的帖子，核心往往就是 **Goal 式框架**——在 **Codex** 里即 **`/goal`**。本期会讲它是什么、怎么用，并给技术示例与**非开发者也能照搬**的用法。

她引用 **OpenAI** 开发者团队博文 *Using goals in Codex* 里的示意图：**Prompt** 是大家熟悉的回合制——你下指令，模型做完一次就停，等你再问「下一步」；若你总在微操、拍肩膀催进度，**`/goal`** 值得进工具箱。

**Claire Vo** 坦言自己用 AI 编码代理很多年，但在 **Codex** 推出 **`/goal`** 之前，很难让 AI **自我编排**到能连续跑多小时。她的问题域不算操作系统级或高深数学，相对 bounded，却仍需要长时 harness——此前做不到。

社区里 **Ralph 循环** 等说法，功能上与此同源：**一直做到 X 行为或 Y 结果可被验证，否则自我重提示**。

### 对谈实录

**Claire Vo**：「**Prompt** 是『做这件事』；**Goal** 是『若成功，世界应是什么样、如何证明』——我见过 **Codex** 为同一个 Goal 跑非常久。」

**Claire Vo**：「现场这个 **Vercel** Goal 文案很烂——但成功态很清楚：没有该面向用户的错误，该 warning 的已 warning。」

**Claire Vo**：「派给人类同事时，你不会站在背后逐步指挥；你给目标，他们离开一段时间，带着结果回来——我现在更像管理者，老实说，我不确定自己是否喜欢。」

## Prompt 与 Goal 循环：机制对比

### 核心洞察

> **Prompt** 描述要做什么；**Goal** 描述成功时长什么样、怎样证明成功。

### 深度解析

| 维度 | 回合制 Prompt | Goal 循环 |
|------|----------------|-----------|
| 交互节奏 | 一问一答，等你下一句 | 设定终点后自动「工作→检查→选下一步」 |
| 停止条件 | 单次任务结束即停 | 直到收集到**达标证据**才标记完成并通知你 |
| 典型痛点 | 「好的，下一步呢？」循环 | 适合「继续、再跑测试、直到真完成」类任务 |

**Claire Vo** 强调：网上讨论的 **Codex**、**Claude Code** 等「长时自治」，底层多是这类 **Goal** 框架，而非单靠神秘的新模型能力。

### 对谈实录

**Claire Vo**：「我在编码工具里说得最多的一句话，就是『好的，下一步呢？』——若你也在这种循环里，**`/goal`** 值得进工具箱。」

## Goal 生命周期：启动、查看、暂停、恢复与清除

### 核心洞察

> 长时自治不等于失控；生命周期命令让你随时踩刹车或改道。

### 深度解析

她归纳四条管理路径（口播为四种用法）：

1. **`/goal` + 目标描述后走开**——最常用。
2. 再次 **`/goal`**——查看当前 Goal。
3. **暂停 / 恢复** Goal。
4. **清除** Goal。

不必让 AI 跑满 6、12、24 小时；跑偏时可介入。OpenAI 博文与她自己的经验一致：**当你会反复在同一回合后说「继续、试下一个、再跑测试、直到真做完」时，就该用 Goal。**

**Claire Vo** 第一次成功案例总时长约 **5 小时 45 分钟**——她个人迄今最长单次运行。这建立在她问题规模「不够 OS 级」但足够烦人的前提下；对工程团队，意义在于**错误清零、技术债燃烧**类任务可一次性交给 harness。

### 对谈实录

**Claire Vo**：「第一次用 **Goal**，编码任务跑了大约 **5 小时 45 分钟**——比我以前任何东西都长。」

**Claire Vo**：「OpenAI 说：当你会在同一回合后反复说『继续、试下一个、再跑测试、直到真做完』——那就是 **Goal** 的用武之地。」

## 如何写好 Goal：结果而非产出，六个组件

### 核心洞察

> 强 Goal 几乎总是「结果导向 + 证据收尾线 + 护栏」，而不是「帮我改这页 UI」式产出描述。

### 深度解析

**Claire Vo** 点名产品经理与写**成功标准**的工程师：写 **Goal** 是独立技能集，接近写 **OKR**——要**可测量、可验证**。

OpenAI 博文给出的技术范例：**把结账页 P95 延迟降到阈值以下**——可反复加载页面测量；护栏如**测试套件保持全绿**；有明确可执行面，LLM 才容易成功。

**六个组件**（她逐条复述）：

1. **Outcome（结果）**：做完后什么应为真？
2. **Verification（验证）**：怎么测？测试套件、开浏览器、还是盯某个数字？
3. **Constraints（约束）**：工作中什么不能退化？（例：不能把结账页删掉换「零延迟」）
4. **Boundaries（边界）**：允许用哪些工具、改哪些文件？
5. **Iteration policy（迭代策略）**：下一步试什么、如何决策？
6. **Stop / escalate（停止与上报）**：何时承认卡住、向你汇报要什么？

推荐句式骨架（她归纳 OpenAI 模式）：

- 终态由**具体证据**验证；
- 必须保持的约束；
- 迭代间可用工具；
- 按 X/Y/Z 决定下一步；
- 若阻塞或无有效路径，**停止并报告**证据与所需帮助。

**P95 结账延迟** 的「加强版」Goal 示例：阈值以下 + 用 checkout benchmark 验证 + correctness suite 全绿 + 只动 checkout 子系统 + 每轮汇报改动、benchmark 与下一实验 + 无新思路则停并交 blocker 清单。

非技术项目同样可套此结构——后文邮件与 **Linear** 即证。

### 对谈实录

**Claire Vo**：「强 **Goal** 有六样东西：结果、验证、约束、边界、迭代策略、何时停下并找你。」

## 技术案例一：ChatPRD 与 Sentry 编辑错误清零

### 核心洞察

> 把 Goal 对准「错误日志归零」类终点，往往得到**统一框架修复**，而不是满仓库创可贴。

### 深度解析

**ChatPRD** 主 AI 写作循环里有一个**基于 diff 的局部 PRD 编辑器**：在文档内找「操作区间」再改——文档含表格、列表、加粗、引用、图片时，让模型精确定位节点极难，**Sentry** 上充斥 *invalid operation on the edit tool* 类错误。

团队曾「修一个冒一个」，像卡通里堵一个洞另一个喷头喷水。**4 月底至 5 月初** 错误曲线断崖式归零——因 **`/goal`** 系统性处理。

她给 **Codex** 的 Goal 大意（据口播还原）：

- 接入 **Sentry** 与编辑请求数据；
- 遍历 **Sentry** 中每一条 *invalid operation on the edit tool* 的 trace；
- **分类 → 修根因 → 重放**所有共享该问题的历史事件，直到该类问题及历史样例全部解决。

运行方式：抽一例 → 找根因 → 实现修复 → 扫剩余同类事件「烧掉」→ 再抽下一例，循环直至 **零错误**。耗时**数小时**；交付物是**更智能的编辑应用框架**，而非零散补丁——自该次 **Goal** 后**编辑错误保持为零**。

### 对谈实录

**[视觉盲区]**：**Claire Vo** 在片中展示 **Sentry** 曲线与 **ChatPRD** 界面；具体图表刻度、错误分类命名以口播与她的描述为准，文本未逐字读出全部 UI 标签。

**Claire Vo**：「我们像堵喷泉——修一个 **Sentry** 错误又冒另一个；用 **Goal** 后曲线归零，而且是系统性框架，不是满仓库零散补丁。」

## 技术案例二：直播烧掉 Vercel API 日志错误

### 核心洞察

> 「错误归零」可指向任何长期缠着团队的日志类别——**Goal** 负责分类、修复、验证直到烧尽。

### 深度解析

为体现「这就是 How I AI」，她在片中**现场**再开一个 Goal，对象是两週内 **Vercel** 日志里 **`API chat v2` 端点**相关错误（她称背后有重试机制，表面吓人）。

模型选择：**GPT-5.5 medium**（她认为非需深度推理的复杂题）。她承认口播 Goal 文案「很烂」但够用——成功态是：**无用户可见错误**，该降为 warning 的后端噪声已降级。

Goal 要点（口播）：

- 按错误类别遍历 **Vercel** 日志；
- 判是否用户可见；若是则找根因并**开分支 + PR**；若否则降为 warning；
- 处理完近两周日志后，汇报所有待审 **PR** 与无法修复项 / 需她介入项。

**Codex** 通过 **Vercel** 插件拉日志后生成 **3–5 步**计划：清点仓库 → 拉两周错误分组 → 分类 → 按类修复或降级 → 发布 **PR** 并报告。她判断类似任务以往约 **2–3 小时**；本片录制时 Goal 才跑约 **1 分钟**，**最终是否完全成功需看节目说明/后续跟进**（她明确会在 show notes 里交代）。

## 非技术案例一：3,900 封邮件自治清理

### 核心洞察

> 对普通人，Goal 清理收件箱的 ROI 可能比写代码还高——提示词可以很糙，结果仍可惊人。

### 深度解析

**Claire Vo** 称这是她**最喜欢的非编码用例**——希望观众至少带走这一条。

**Codex** 通过 **Gmail** 插件（**MCP**）读信。她昨日约 **3,900** 封未读；Goal 文案极简（她自嘲写得更糟）：

- 归类批量推广/垃圾邮件；
- 取消不必要的订阅；
- 需要判断时求助她。

运行 **3 小时 52 分**，约 **600 万 token**（她强调**不省 token**）。行为包括：逐封阅读、打标签（含「需人工判断」）、代点退订、汇总可手动退订链接列表。结束后她让系统自检：由约 **3,900** 降至 **68** 封待处理——「今日项目」从邮件山变成可扫清单。积压回复也由此清理。

她建议：**`/goal` 清理全部未读邮件**——即使 prompt 简陋，也值得试一次。

### 对谈实录

**Claire Vo**：「若你只学一件事：用 **Goal** 清未读邮件。我从约 **3,900** 封到 **68** 封，跑了近 **4 小时**，约 **600 万 token**——不便宜，但值。」

## 非技术案例二：Linear 播客任务卫生

### 核心洞察

> 任务卫生类 Goal：用清晰业务规则批量改状态，比人手点几百条更可持续。

### 深度解析

面向产品经理：**Linear** 任务池失控——部分因她把 **OpenClaw** 代理接入 **Linear**，代理创建了大量未必完成的任务。

Goal 大意：

- 清理 **How I AI** 播客团队在 **Linear** 的 issue；
- **已发布集数**下仍未完成的项 → 标为 **won't do**（她口播先说 *will not do*，系统中实际状态名为 **Canceled**）；
- 只保留**本周及未来集数**的开放工作，砍掉旧集永远不会做的 aspiration 任务。

她预期约 **30 分钟到 1 小时**（需高质量判断）。代理归纳规则：**保留当前周与未来集数工作；取消周一前未完成的旧集发布相关任务**等，并做 bulk update 与结果验证。

## 何时不要用 Goal 与强 Goal 的三条判据

### 核心洞察

### 深度解析

**Claire Vo** 再次引用 OpenAI 博文「不该用 Goal」的情形：

1. **极简单行修改**——用大炮打蚊子；Goal 应是 **outcome 不是 output**（「确保删掉某行」不适合）。
2. **终点模糊**——如「让客户满意」无法可靠测量、无确定完成条件。
3. **「重构这段代码」式任务**——她在做重构专项，但**刻意不用 Goal**，因完成线不清晰。

**强 Goal 三属性**（她复述）：

- **Durable objective**：目标在长时间内保持稳定；
- **Evidence-based finish line**：能用证据判定完成；
- **多轮调查路径**：到达终点需要若干轮尝试。

三者齐备再用 **`/goal`**。

### 对谈实录

**Claire Vo**：「别用 **Goal** 做单行编辑；别用终点模糊的『让客户满意』；『重构这段代码』也不适合——完成线说不清。」

## 工作方式转变：从盯梢到管理，以及她的复杂感受

### 核心洞察

> Goal 让她第一次真正「走开几小时」；工作体感更像派活给同事，而非坐在模型旁边逐步指挥。

### 深度解析

**四点收束**：

1. **盯梢日子大体上结束了**（她仍要盯分支，但不再逐步催下一步）。
2. **生活质量**：长期烦人的代码质量问题可用「错误归零」式 Goal 一次性推进——适合烧技术债、不稳定测试、难复现前端错误。
3. **产品经理必须升级**：团队越多用 **`/goal`**，越要会写带**技术级验证**的好 Goal——现有 OKR 技能要加码。
4. **角色焦虑**：与 **OpenClaw** 类似，她感到自己更多进入**管理者模式**而非**构建者模式**；**`/goal`** 刚出来时甚至有空手搓脚、找还能亲手写的活——**她不确定自己是否喜欢这种转变**。

结语：若在 **Codex** 外，也可在常用工具里实现类似「做到验证通过为止」的循环；把更大、更烦的问题交给 AI，回来时只做**审阅**。
