慢下来，才可能用好 AI：Dax Raad 谈 OpenCode、工程判断与开发者工具现实

Dax Raad 的核心反直觉判断是：AI 编码工具确实让写代码变容易，但并没有自动让团队更快、更会做产品。

The Pragmatic Engineer Podcast: Dax Raad on OpenCode, AI developer tools, open source, and engineering judgment 2026-05-28 ~42 分钟阅读

核心导读

Dax Raad 的核心反直觉判断是：AI 编码工具确实让写代码变容易，但并没有自动让团队更快、更会做产品；真正稀缺的仍是判断力、反馈回路、产品品味与工程约束。OpenCode 的增长故事不是「更聪明的代理赢了」，而是开源定位、模型供应商竞争、开发者工具消费品化、以及对质量的执念一起作用的结果。

AI 提高的是局部速度，不是组织判断。 Dax Raad 反复强调：OpenCode 团队自己高强度使用 AI，但他仍像以前一样费脑子。产品达到市场契合后，难题不是「能不能实现功能」，而是面对用户请求、竞品动作和内部点子时，能否克制地决定什么不该发布。AI 让「一事一功能」变得廉价，也让产品更容易长成拼贴怪物。

OpenCode 的战略不是偶然爆红。 Dax Raad 早早判断 AI 编码工具市场缺一个「开源默认选项」：模型供应商彼此竞争，开发者工具的底层默认项常由开源方案占据。Anthropic 阻止通过 OpenCode 使用 Claude Code 订阅后，OpenCode 借势与 OpenAI、GitHub Copilot 等模型或平台方合作，把一次封锁变成增长事件。

工程师的新工作，仍是老问题的新形态。 企业多花几千美元给每位工程师配 AI 工具，未必换来更多产出；有时只是同样工作量下工程师更轻松。AI 代理还会淡化写权宜补丁时的刺痛感，掩盖技术债。于是老式工程纪律反而回潮：测试、领域建模、清晰模式、反馈闭环、可安全改动的代码库，都是为了让「不够成熟的执行者」也能少犯错。

未来预测不如明天行动。 Dax Raad 对社交媒体上「某年龄段工程师将无敌」「某类公司必死」之类论断极度怀疑。他认为大变化会发生，但大多数预测只是焦虑中的自我安慰：每个人都在描述一个自己会赢的未来。更可靠的做法，是守住反馈回路、行业理解、质量信念与每天该做的下一件事。

1. AI 让写代码变容易，但没有让思考变轻

核心洞察

AI 降低了实现成本，却没有消除产品判断、组织取舍和工程设计的难度。

深度解析

Gergely Orosz 开场提出一个矛盾：Dax Raad 正在做最受欢迎的 AI 编码工具之一，OpenCode 被用来加速软件开发，但 Dax Raad 却说这还不足以带来更好的软件。

Dax Raad 的回答很坦白：OpenCode 是团队自己每天使用的产品，确实重要，确实让很多工作更容易；但所有旧问题仍在。他仍然像以前一样努力、一样挣扎、一样需要费尽心力决定做什么。

这种体验在产品达到市场契合后尤其明显。预产品市场契合阶段，团队的核心任务是判断到底该做什么，AI 只能帮助多试几个方向，却不能替代深思熟虑。Dax Raad 甚至更偏好「多想」而不是「多挥几杆」，因为很多错误方向在讨论和思考中就能被排除。

OpenCode 所在阶段已经越过产品市场契合，难题变成「如何兑现潜力」。这时四面八方都有诱惑：用户每天要求新功能，竞品每天推出新特性，内部也有无数显而易见的想法。若每个问题都直接交给 AI 生成一个功能，结果看似是发布了一千个功能，实际可能是做出一个难以维护、毫无整体感的怪物。

Dax Raad 的判断是：能发布十倍功能，不代表有十倍值得发布的好想法。功能一旦进入产品，就要长期支持；未来每一个新功能都要和它互动。因此 OpenCode 过去半年用 AI 改变了很多流程，也踩了很多坑，现在反而要回头判断旧世界的哪些纪律仍然有效。

2. 从 Minecraft 到开源：Dax Raad 的开发者工具底色

核心洞察

Dax Raad 对开发者工具的理解，来自早期社群、创业失败和多年开源实践，而不只是 AI 热潮。

深度解析

Dax Raad 的技术路径从少年编程开始。他的父亲是软件工程师，这让他更容易接触编程。高中毕业后，Dax Raad 就进入工作，创办过公司，也经历过小型收购式招聘，随后成为真实科技行业的一员。

早期 Minecraft 模组经历对 Dax Raad 很关键。他参与过 Minecraft 模组框架，自己做过服务器和模组。相比单纯玩游戏，Dax Raad 更喜欢创造「沙盒场景」：让上百名玩家在特定规则下互动，然后观察人的行为。这种兴趣后来延伸为他对工具、系统和用户行为的敏感度。

更重要的是，Minecraft 社群当时聚集在 IRC 上，其中有不少资深、优秀但并不追逐职业头衔的程序员。Dax Raad 认为自己在那几个月里学到很多，因为能近距离和这类人交流。

之后 Dax Raad 在 Ride Health 做过工程负责人，也经历了年轻创业团队的混乱。他回头看时认为，二十多岁的一群人一起创业，常被神化，但现实是很多人的安全感、冲突处理能力和自我认知都还没成熟。Ride Health 虽然没有成为理想的创业退出，但 Dax Raad 在那里遇见了后来成为妻子的产品负责人，这被他开玩笑称为「比创业退出更好」。

Open source 轨迹则来自 SST 和 OpenNext。Dax Raad 先在一家 B 轮公司做管理者，白天有大量会议，仍想写代码，于是开始参与开源。他加入 SST 团队后，OpenNext 成为一个重要项目：大量 AWS 用户需要把 Next.js 部署到 AWS，Vercel 当时自然更关注自家平台，生态里出现缺口。SST 团队填补这个缺口，并逐步促成 Cloudflare、Netlify、Microsoft、Google 等平台加入适配讨论，最终推动 Next.js 官方适配器方向变得更协作。

这段经历让 Dax Raad 形成了一个反复使用的策略：找到一个临时的「坏人」或瓶颈，联合其竞争者推动生态前进。OpenNext 面对的是 Vercel 之外平台的需求，OpenCode 后来面对的则是模型供应商之间的竞争。

3. OpenCode 的诞生：开源默认位与模型竞争的机会

核心洞察

OpenCode 抓住的不是单一功能缺口，而是 AI 编码工具市场中「开源默认选项」的战略空位。

深度解析

OpenCode 起步于 SST 团队刚刚实现收支平衡之后。2026 年 2 月，SST 曾经只剩一个月现金，但收入也在同一时间追上成本。公司暂时安全后，团队终于可以退一步问：既然可以做任何事，真正应该做什么？

Dax Raad 认为，AI 是开发者工具十年级别的大方向。每次出现大浪潮，都会有大量看起来荒谬的投资和产品，因为任何有潜力的方向都会吸引过量资本。若只因为大部分东西看起来不合理就完全旁观，也会错过其中真正有价值的部分。

SST 团队试过一些 AI 想法，很多还没正式发布就发现不成立。直到团队开始使用 Claude Code，它成为第一款真正留在他们工作流里的 AI 编码工具。Dax Raad 的反应不是「这东西很好」，而是「为什么不是我们做的？我们应该为没做出来而羞愧」。

随后团队从定位角度重新看市场：AI 编码代理很多，但没有一个清晰占据「开源选项」心智。开发者工具历史上，数据库、编译器、基础设施等类别常会出现开源默认项；与此同时，模型层竞争极其激烈，OpenAI、Anthropic、开源模型阵营都投入巨大，没有谁会轻易让对手独占入口。

OpenCode 的初始战略由此形成：先忽略市场短期噪音，明确占据开源位置，并让工具能和多个模型供应商合作。这个定位为后来增长埋下伏笔。

数据上，OpenCode 的增长非常快。团队最初只有三名联合创始人，后来加上一位朋友和一位设计师。2025 年 12 月，OpenCode 达到约 65 万 月活跃用户；2026 年 1 月跳到 250 万；节目对话时，Dax Raad 称上月约 650 万，当月可能接近 800 万，下一目标是 1000 万。

3

「AI 显然是这十年开发者工具里必须参与的事。大浪潮会吸引很多不合理投资，所以很容易看着它说：这些都很蠢，我不参与。但通常里面有几件事真的有价值。如果完全坐在场外，就会错过。」

4. Anthropic 封锁事件：把坏消息变成增长杠杆

核心洞察

Anthropic 的封锁原本可能重创 OpenCode，却因为沟通方式和竞争格局，变成 OpenCode 的公关胜利。

深度解析

2026 年 1 月的增长跃迁，一部分来自开发者假期后尝试新工具的季节性趋势，另一部分来自 Anthropic 阻止通过 OpenCode 使用 Claude Code 订阅。Dax Raad 认为 Anthropic 当然有权保护商业模式，但问题在于做法：晚上突然封锁，没有提前沟通，没有分阶段迁移，等于把用户不满压缩到一个爆发点。

OpenCode 团队并不惊讶。他们早就知道这一天会来，而且也判断使用 Anthropic 高价订阅的用户只是 OpenCode 用户中的一部分。社交媒体上很多人以为 OpenCode 会因此归零，但 Dax Raad 已经在同一时间联系 OpenAI：第二天所有人都会生 Anthropic 的气，OpenAI 可以通过正式支持 OpenCode 获得一次清晰的公关胜利。

OpenAI 次日确认合作，OpenCode 当天实现集成并宣布正式支持。此前 OpenCode 已经拿到 GitHub Copilot 等其他支持，只是尚未公布。Anthropic 的封锁无意中把 OpenCode 和 Anthropic 放进同一句话，这对小公司而言是巨大的曝光。

Dax Raad 把这称为他们很熟悉的生态策略：临时选定一个大公司作为反面角色，动员其竞争者围绕开放入口向前推进。这不是长期敌友关系，而是竞争如何发挥作用。未来如果 OpenAI 变成限制入口的一方，OpenCode 也可能围绕 OpenAI 组织其他竞争者。

这段也揭示了快速增长公司的风险：一个小动作会波及数百万人。Dax Raad 举了 OpenCode 自己的例子：一个 bug 让几乎所有深色终端用户打开 OpenCode 时看到浅色界面，像被突然闪光照到。过去这可能只影响一百人，如今是一周内影响上百万人。

5. 开发者工具也是消费品：为什么体验先于最聪明的代理

核心洞察

OpenCode 初期并不是最聪明的代理；它先赢在开源定位、低摩擦上手和像消费品一样的第一体验。

深度解析

Dax Raad 认为，开发者工具行业有一个常被忽视的事实：开发者工具本质上常常是面向消费者的产品（B2C，business-to-consumer）。很多工具最终不是靠企业自上而下采购成为标准，而是单个开发者先喜欢、偷偷带进团队，再逐步扩散到公司。

程序员做开发者工具时，往往过度关注底层机制，忽略首次体验。OpenCode 的策略相反：从用户打开工具的一瞬间，就要让人感觉它不一样、团队很有能力。为此，OpenCode 早期投入做了自研终端渲染框架，而不是直接使用 Ink 等现成方案。这个选择看起来不理性，却让工具第一眼就有差异。

OpenCode 初期的编码代理能力并不强，Dax Raad 坦言前五个月只是「够用」。但对大多数用户来说已经足够分辨不出明显差距；等 OpenCode 拿到足够使用份额，团队再回头增强代理本身。竞品普遍认为必须先做最聪明的代理才能赢，OpenCode 则先赢得心智和使用。

团队也做过错误技术选择。OpenCode 曾探索桌面应用，选过 Tauri，但 Dax Raad 认为当时没有足够认真对待桌面方向，技术判断也不够深，后来转向 Electron。他也认为，并非 800 万 月活用户都应该使用终端；很多人也许更适合图形界面。

另一个体现产品品味的选择，是取消自动在 GitHub 提交或 PR 上标注「由 OpenCode 生成」。早期 OpenCode 模仿 Claude Code 做过这种署名，后来用户要求可关闭。Dax Raad 觉得这像赌场用小技巧困住用户，太明显、太低级，于是干脆默认关闭，而不是只加一个开关。

6. 商业模式：推理服务、企业控制平面与 GPU 瓶颈

核心洞察

OpenCode 的收入来自两端：让用户顺畅使用模型的推理入口，以及企业大规模使用时不可避免的管理控制层。

深度解析

OpenCode 是开源工具，但背后公司需要赚钱。Dax Raad 描述了两条业务线。

第一条是 OpenCode Zen。OpenCode 刚发布时，用户必须连接 Anthropic 或 OpenAI 账户，而很多新用户连足够速率限制都拿不到，无法顺畅使用。团队为了降低上手摩擦，做了一个推理服务，让用户注册后可以访问多种模型和足够额度。它最初只是入门体验的一部分，却迅速增长。

随着开源模型能力上升，托管开源模型也变成难点。OpenCode Zen 逐渐成为聚合最佳模型与最佳推理服务的位置。Dax Raad 称，几个月前这条业务已经达到约 5000 万美元年化收入运行率。开源模型若托管得当，毛利可以很好；一些模型的标价与 OpenCode 自己租 GPU 的成本之间，甚至有约 80% 的差距。

第二条是企业控制平面。一个有上千名工程师的公司不可能只让所有人下载 OpenCode、各自填 API key。企业需要单点登录、权限、预算控制、速率限制、模型供应商配置等管理能力。OpenCode 做了这类管理产品，目前先以企业部署形式提供，也计划公开化。它本身也是开源的，但多数企业会为托管版本付费。

这也解释了赞助口播中提到的企业基础设施需求：当 AI 工具进入大公司，身份、权限、细粒度授权、控制平面等「无聊功能」会变成必需品。与此同时，检索增强生成（RAG）也从 2023 年的简单向量检索，演变为代理一次提示后调用向量索引、全文索引、文件系统、命令行、数据库、记忆系统和重排序流程的复杂搜索网络。

推理业务之所以可能很赚钱，是因为长期下限接近电力成本。训练和研发当然昂贵，但纯推理环节在当前价格下毛利极高。Dax Raad 认为 OpenAI、Anthropic 等巨头凭借最大规模 GPU 交易，当前某些推理业务毛利可能接近 90%，但这种利润率长期不可防守。

短期真正的瓶颈是 GPU。推理需求可能不是线性增长，而是指数式增长；GPU 生产、配套硬件、供应链和人力却更接近线性扩张。大科技公司每年投入数百亿美元，吸走供应链注意力，连 OpenCode 这样规模的公司也要提前预订、预付大量 GPU 容量。Dax Raad 预计紧张最终会被过度供应修正，但眼下确实紧。

7. 生产力神话：更快写代码不等于公司更快前进

核心洞察

AI 可能让个人少花力气完成同样工作，却未必让公司得到更多高质量成果。

深度解析

Dax Raad 对 AI 生产力叙事的怀疑，来自对真实软件组织的观察。软件工程行业非常大，几乎每家公司都雇工程师；但多数环境并不是高度激励、人人想改变世界的创业团队。很多人只是想做好工作、回家陪孩子、过合理生活。

如果给这些人一个按钮，让他们更快完成任务，他们很自然会用它完成同样工作量，然后把节省的时间留给自己。这并不道德错误，而是正常人性。问题在于，公司高层不能想当然地把个人节省的能量换算成组织产出增长。

更糟的是，在很多公司里，少数真正还在乎质量的人，会被大量低质量 AI 生成 PR 淹没。Dax Raad 说，OpenCode 团队招过一些人，他们之前就是公司里「仍然在乎的人」：其他人按按钮交差，他们则被迫审查、补救和兜底，最后精疲力尽离开。

成本也会迫使公司重新计算。当前很多企业处在炫耀式阶段：声称每位工程师每月花 1 万美元也值得，因为这是未来。但当公司有几千名工程师时，即使每人每月多 1000 或 2000 美元，也会冲垮预算。若不能清楚证明更多业务结果，公司最终可能只看到「同样产出，工程师更开心」。

Dax Raad 认为，这种结果对很多公司不够好。它们可能选择收紧工具，甚至让人回到手写代码。但另一边也存在人才市场压力：顶尖工程师若发现公司只允许落后工具，可能离开。高端人才会把工具自由度当作工作条件的一部分，就像有些人会因为讨厌 Jira 而离职。

8. 被 AI 静音的刺痛感：权宜补丁、清理债务与放慢节奏

核心洞察

AI 最大的工程风险之一，是让权宜补丁不再让人难受，于是团队更容易误以为自己在高速前进。

深度解析

Dax Raad 曾给 OpenCode 团队发过一封内部备忘录，承认团队面临三个被大语言模型放大的问题：发布了不值得发布的功能；原始设计不支持新需求时，团队更容易接受权宜补丁；清理和重构投入不足。

第一点对应产品克制。问题出现后，直接发布一个功能是最容易的反应，但这不等于正确。OpenCode 团队开始更明确地区分什么值得发布、什么需要等待更多清晰度。

第二点是最大挑战。过去工程师手写权宜补丁时，心里会有刺痛感：知道自己埋了地雷，知道以后可能伤到别人或自己。第二次补丁叠上第一次补丁时，这种不舒服会加剧，迫使人重新评估是否该重构系统。

AI 代理改变了这个反馈回路。它会替人写补丁，也会替人处理补丁带来的后续麻烦。问题仍在，地雷仍在，但那种「我正在做坏事」的直接感受被静音了。Dax Raad 认为，判断力因此被扭曲，团队更容易把本该重新设计系统的地方当成临时修补。

第三点是清理债务。讽刺的是，AI 让清理技术债比过去更容易。以前发现一个更好的模式，团队也许只在新代码中采用，因为清理旧代码太累；现在可以让代理把新模式应用到整个代码库，删除旧模式、迁移旧实现。这让「清理」更有可能成为高回报工作。

Dax Raad 还提出一个尖锐结论：他们承担这些问题后，并没有换来更快速度。团队感觉自己在快跑，但回头看并没有比竞品快，也没有明显慢。也就是说，AI 让速度感增强，却不一定让真实前进速度提高。于是「慢下来」不是保守，而是为了在基础打好后更有力地冲刺。

9. 反预测：技术焦虑中的年龄叙事与赢家幻觉

核心洞察

大多数 AI 时代预测，都是焦虑者在描述一个自己会赢的未来。

深度解析

Dax Raad 对社交媒体上的宏大预测尤其反感。节目中特别讨论了一条流行观点：「24 到 29 岁工程师将成为科技行业最有价值资产，因为他们兼具 AI 前原则和 AI 后速度。」很多人把它当作未来判断，Dax Raad 则直接称其为自我安慰式口号。

他的分析很简单：这类帖子通常都在说「像我这样的人拥有所有优势，不像我的人有所有劣势」。大变化时期，每个人都焦虑自己会处在什么位置。一个防御机制，就是自信宣布某个未来，在那个未来里自己是赢家。公司说「我们这种公司会赢，其他公司会死」；个人说「我的工作不会被替代，别人的会」；某年龄段说「我们这一代刚好最有优势」。

Dax Raad 并不否认年轻人有优势：精力多、包袱少、愿意尝试；经验较丰富的人也有优势：见过更多模式、判断更稳。这个结构一直存在，并不是 AI 独有。那条帖子特别选择 24 到 29 岁，而不是 18 到 25 岁，在 Dax Raad 看来恰好暴露了发帖人自己可能落在这个年龄段。

Gergely Orosz 补充，Ken Beck、Martin Fowler、Grady Booch 等资深软件人物也都认为这次变化速度比过去主机到微处理器等变迁更快。Dax Raad 承认变化巨大，但强调真正发生的结果往往反直觉。事后看会显得合理，事前却很难预测。

因此 Dax Raad 的策略是缩短视野：今天做什么，明天做什么。他一年前也不知道自己会做 OpenCode，一年后也不知道自己会做什么。与其不断预测，不如做眼前最合理的下一步。

10. 产品原则：最快触达一个好想法，解决五十个问题

核心洞察

好产品不是接到一个问题就发一个功能，而是抽象出一个能同时解决许多问题的方案。

深度解析

Dax Raad 的产品原则很朴素：一个产品里可能只有一个真正好的想法，要让用户尽快体验到它。听起来容易，但绝大多数产品从用户听说它到真正感受到价值，中间都意外塞进太多步骤。

OpenCode 为此会定期复盘首次体验。Dax Raad 有一个命令，可以启动新的 Docker 容器并运行 OpenCode，让他模拟全新用户第一次使用。他大约每两周做一次，总能发现团队无意中破坏了入门体验。

更高阶的产品工作，不是收到问题后发布即时解法，而是吸收所有问题，找出一个能解决 50 个不同问题的方案。这个能力需要经验、思考、用户访谈、团队讨论和代码库理解。Dax Raad 认为 AI 几乎没有帮助他提升这件事。

帮助他提升产品判断的，是真实反馈回路。设计糟糕时，会有人直接骂他；架构没想清楚时，他自己要在代码库里痛苦地做本应简单的改动。团队不能隔绝这种痛感。大公司里，产品和工程团队有时为了早点发布砍掉细节，痛苦却落在支持团队身上，支持团队手动修复、安抚用户，发布者只看到「功能上线了」。OpenCode 作为开源公司，尽量让做功能的人直接承受 GitHub issue、社交媒体反馈和用户抱怨。

这种反馈回路也解释了为什么要「面向整个市场」思考。做一个适合所有开发者环境、偏好、约束、流程的工具很难，旁观者会觉得团队在关注奇怪问题。但如果目标是整个市场，而不是少数早期爱好者，产品判断就会自然不同。

11. OpenCode 团队如何工作：公开反馈、推上山与动机管理

核心洞察

OpenCode 的组织方式依赖高上下文共享：团队理解市场和定位后，自主选择最值得推进的问题。

深度解析

节目对话时，OpenCode 团队已经从几个人增长到二十多人，且大多是在最近三个月加入。Dax Raad 说这正处于一种糟糕但常见的阶段：人比以前多，却因为大家还在适应，感觉反而比以前慢。

OpenCode 尽量公开构建。代码本来就是公开的，用户也非常活跃。团队内部把一个新方向分成两个阶段：第一阶段像把东西推上山，从零到有非常痛苦；第二阶段还远未完成，但大部分关键问题已被弄清，剩下是细节打磨。

团队会尽快把事情推到可演示状态。一旦用户大致理解某个功能，就会开始告诉团队哪里不好、还需要支持哪些环境、哪些流程缺失。做这个功能的工程师直接接收反馈，而不是由某个产品经理汇总后转述；他们自己看 GitHub issue、社交平台回复，并据此判断路线图。

创始人的主要工作是给团队充分上下文：市场如何变化，竞品在做什么，OpenCode 的定位是什么，公司真正关心什么。如果团队理解这些，就会自然形成优先级。Dax Raad 认为动机是长期游戏的核心：如果要跑很久，策略应是如何待在游戏里最久，而这需要每天做让人兴奋的事。

因此 OpenCode 允许团队成员抓自己最想解决的问题。Dax Raad 也承认，有时他公开不同意某人想做的方向，几周后发现对方是对的。这是他希望看到的团队判断力。

12. 品味与质量：小公司对抗巨头时还剩什么

核心洞察

品味不是口号，而是相信产品、代码和体验都必须好，并愿意做很多短期看似不理性的事。

深度解析

谈到「品味」时，Dax Raad 先承认这是个好概念，但也容易变成空话。好想法往往简单，却很难真正活出来。很多人说品味重要，但心里其实不相信产品必须好、代码必须好、体验必须好。他们会指向一些产品差、工程差但仍赚很多钱的公司，得出「这些都不重要」的结论。

一旦这个想法进入团队脑子，就很难做出好产品。Dax Raad 认为，真正相信工艺和质量的人并不多。优秀产品可能做到今天的 50% 好也仍能赚钱，但那些额外的质量会以难以直接归因的方式出现。如果团队在一个地方偷懒，懒散会像感染一样扩散到所有地方。

Dax Raad 也坦白，OpenCode 还远没达到他自己的质量标准。他仰慕的对象之一是 Mitchell Hashimoto：HashiCorp 的开源产品如 Terraform 成为默认工具，业务模型、宏观定位和细节执行都很强；后来 Mitchell Hashimoto 做 Ghostty 终端，也在架构和细节上体现出对体验的长期执念。

质量也是小公司对抗巨头时少数可倚靠的差异。AI 代理让产品更容易腐烂：大公司产品在更快变差，创业公司产品也可能一年后就开始失去完整性。OpenCode 早期面对 Claude Code 这样更大产品时，一个重要差异就是终端体验更好、更少闪烁、更有完成度。为此自研终端框架看似违反「不要造框架」的工程常识，但 Dax Raad 认为这是实现想要体验的必要选择。

13. AI 时代的工程领导：护栏、老模式与行业专家化

核心洞察

AI 时代工程领导的核心仍是老问题：让经验不足的执行者在清晰护栏中安全地产生价值。

深度解析

Dax Raad 听到很多工程领导的新叙事：如果工程师不再亲手写所有代码，那么工程师的职责是搭好护栏，让代理或非工程同事能安全发布。测试要好，代码库模式要清晰，约定要足够明确，避免代理写出疯狂实现。

他同意这个方向，但不认为它新。过去工程团队一直想解决同一个问题：如何让初级工程师安全提交代码，如何让代码库模式清晰，如何让测试保护系统，如何让经验不足的人发挥超过自身经验的价值。AI 代理只是让这个问题变得更密集，因为团队现在多了一群二十四小时工作的「不成熟执行者」。

这也让一些旧模式回潮。OpenCode 一直偏向轻量领域驱动设计，现在开始更重地使用这类看似企业化、啰嗦的模式。原因是 AI 代理需要更多护栏。过去这些模式让优秀程序员觉得像训练轮，太啰嗦、太烦；现在代理替人打字，冗长成本下降，模式带来的可靠性、模块化和安全性重新变得划算。

给资深工程师的建议上，Dax Raad 不愿给普遍处方，因为 OpenCode 是开源创业公司，处境很特殊。但他一直相信一个职业策略：软件工程是一种可迁移技能，如果再叠加对某个行业的深入理解，就会变成稀缺组合。一个懂农业、医疗或任何具体行业的优秀工程师，可能很快成为那个交叉领域最稀缺的人之一。

这种路径的好处是，工程师不像很多职业那样必须一生绑定一个行业。可以在医疗做十年，理解行业后换到另一个领域；核心是不要只把自己当成「接任务改界面」的人，而要利用工程工作进入任何行业并学习它。

节目最后，Dax Raad 还分享了个人工作设置：Framework 笔记本、Arch Linux、5K 显示器、平铺窗口管理器、远程高性能机器、多个 tmux 会话、Neovim 左侧、OpenCode 右侧。他不怎么读书，但通过爱读书的妻子吸收很多观点；早年影响他的包括 Nassim Nicholas Taleb 的《Skin in the Game》《Black Swan》，尤其是自下而上系统和涌现属性：伟大事物往往不是顶层设计出来，而是许多较小实体随机互动后出现。
