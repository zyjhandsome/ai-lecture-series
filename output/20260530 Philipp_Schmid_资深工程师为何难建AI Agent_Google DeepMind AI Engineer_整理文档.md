> **全文论点**：传统软件工程消除歧义，**Agent** 工程管理风险——资深工程师的隐式上下文与确定性习惯，反而可能拖慢 Agent 交付；五条原则：**别和模型打架、文本当状态、错误当输入、评测代替断言、为可删除而建造**。

**范式悖论。** **Philipp Schmid** 论断直白：**Agent** 只见 schema、工具与 docstring，看不见「接口为何这样设计、失败会发生什么」——越资深有时越难快速交付可用 Agent。

**指挥员到调度员。** **交通指挥员**控制红绿灯；**调度员**只说「从德国去伦敦」——路线由 **Agent** 在目标约束下自行规划，这是从确定性到概率性系统的迁移。

**五原则收束。** 别和模型打架、把文本当状态、把错误当输入、用评测代替断言、为可删除而建造——API 层「越薄越好」与 Harness 设计，是资深工程师把经验转化为 Agent 可消费上下文的实操路径。

## 传统软件与 Agent：两种建造循环

### 核心洞察

> 写软件时你是「指挥交通」，建 **Agent** 时你是「派单调度」——定义目标，不定义每一步。

### 深度解析

**Philipp Schmid** 开场说明：接下来约 **10** 分钟，讨论工程师为何在搭建 **Agent** 时挣扎——他在 **Google** 内外部几乎每天都看到这一现象，并带来 **五** 个与传统软件截然不同的例子。

**传统软件**路径他概括为：写规格/产品需求文档（PRD）→ 写代码 → 有时写测试保证正确 → 部署 → 用户使用。流程相对线性、可预期。

**Agent**路径则是：写清你希望 **Agent** 做什么的指令 → 运行 → 观察行为 → 可能改提示词、可能改工具 → 再运行——形成一个**迭代提升可靠性**的闭环。这与过去「写完即交付」的心智差异很大。

他用类比收束：传统软件里工程师像**交通指挥员**——控制红绿灯、限速、可用道路，本质上决定车怎么开。 **Agent** 时代更像**调度员**：你说「我从德国要去伦敦」，可以坐火车、坐飞机、开车，甚至他半开玩笑提到「从水下开过去」——你定义**目标**，不定义达成目标的**精确步骤**。每个人都见过编码 **Agent** 中间过程很怪、但最后结果对了——这正是他想要的方向。

**Philipp Schmid**：「以前写软件：写 spec、PRD，写代码，有时写测试……部署，用户用起来。建 Agent 则是：跑起来，观察它做什么……这样迭代循环。」

**Philipp Schmid**：「传统软件里我们更像交通指挥——现在有了 Agent，我们更像调度员。」

**Philipp Schmid**：「文本成了新的状态——我们不再真正运转于那些清晰的结构化数据概念里。」

**Philipp Schmid**：「我们要从单元测试转向评测（evals）——测的是某件事能成功的频率。」

**Philipp Schmid**：「Agent 看不到代码……只能看到函数 schema 和 docstring……我们需要为 Agent 使用而写的工具和方法。」

## 范式一：文本是新的状态

### 核心洞察

> 自然语言承载语义与偏好，布尔标志装不下「同一次批准里顺带改需求」这类真实交互。

### 深度解析

过去应用状态常映射到布尔值、枚举与可检查的标志位。**Philipp Schmid** 以 **Deep Research** 类 **Agent** 为例：传统软件可能是「批准计划 / 拒绝计划」的二元分支，难以捕获**语义层**的细微差别。

有了大语言模型，系统能理解语义。例如市场研究任务：用户既可以**批准**初稿计划，也可以在同一轮里**附加**「聚焦美国市场、忽略加州」「再补充某类信息」——而不必走「拒绝 → 再填表 → 重新生成计划 → 继续」的多步刚性流程。

**记忆与个性化**同样难用结构化字段穷尽。他举自身习惯：身在欧洲，日常用**摄氏度**；但若做饭时想用**华氏度**，靠用户档案里的 `useCelsius` / `isEurope` 等标志很难**动态**响应当下意图——真正驱动行为的是**文本与上下文**（图像、视频、音频亦可，但核心已不再是「一切皆可布尔化」的清晰数据结构）。

这也解释了他后文强调的：**保留语义**，别把自然语言指令当成布尔开关来硬掰。

## 范式二：交出控制权，别复刻客服工单机

### 核心洞察

> 旧客服是「意图分类 + 固定剧本」；**Agent** 应在理解含义后动态协商，而不是把每条岔路都预编码进状态机。

### 深度解析

**Philipp Schmid** 提醒一种常见陷阱，来自传统客服自动化：用户说「我想取消订阅」→ 分类模型打上「流失意图」→ 进入预定流程（挽留销售、确认取消等）。路径是**静态**的，缺少对情境的**动态**反应。

若 **Agent** 能理解含义，可能在取消流程中提供额外选项，用户改变主意，**意图本身**随之变化——要建模所有分支与状态组合几乎不可能。过去那些**有状态、强确定性的工作流**难以覆盖每一种独特性与中途转向。

因此需要**把一部分控制权交给大语言模型**，接受我们不再处于「纯确定性环境」里事事可预测。这与「和模型打架、用硬编码步骤把方差码掉」相对——后文在总结里他明确反对强行规定「第一步做这个、第二步做那个」的僵化编排。

## 范式三：错误是输入，不是重启开关

### 核心洞察

> 长跑 **Agent** 失败后「从头再来」既烧钱又丢上下文；错误应像用户消息一样回灌模型。

### 深度解析

**Philipp Schmid** 的第三点是：**错误只是输入（errors are just inputs）**。流程中某步失败时，应像处理普通用户输入一样对待，而非默认崩溃重来。

他提到 **Go** 语言里函数返回值可以是值或错误，二者在类型层面被同等对待——**Agent** 设计需要类似心态。过去 **HTTP** 请求便宜：搜索或商品查询失败，重发请求、重做工作，成本可接受。

现在 **Agent** 可能跑 **5** 分钟、**15** 分钟；中途某环断裂若**整段重启**，要重复大量算力，还可能丢失已有上下文。不能「从头开始整个流程」，而应理解错误、把错误信息**反馈给模型**，配合绕行策略与额外检查，让流程**继续前进**。

这与「失败即抛异常、进程退出」的传统后端习惯形成碰撞——对长时 **Agent**，把错误当**学习信号**比当**终止符**更经济。

### 对谈实录

**Philipp Schmid**：「错误只是输入——不能一出错就把整个流程从头再来。」

## 范式四：从单元测试走向评测（Evals）

### 核心洞察

> 同一输入不保证同一路径与同输出；要问的是「十次里几次成」，并用评判者衡量主观质量。

### 深度解析

第四点：**从单元测试转向评测（evals）**。传统软件会写集成测试、单元测试、冒烟测试等，默认**输入 A + 代码 B → 恒得输出 C**。**Agent** 是**非确定性**的：相同输入未必相同步骤、相同结果。

因此测试重心变为：**某件事成功的频率是多少**。客服 **Agent** 若同一提示词十次只成一次，就不该上生产——会变得极其**不稳定（flaky）**。需要跑评测、看**通过率**，并与传统软件的「断言」思维区分。

结果还常带**主观性**：写研究报告与写客户反馈场景，「好结果」的定义不同。需要**定性评判**——**大模型作评判者（LLM-as-judge）** 或人类专家是可行路径。

同时必须**追踪（trace）** **Agent** 在做什么，但评测锚点应在**产出**：也许对用户甲多研究四步、多耗一些 token，对用户乙路径更短——**最终 outcome** 才是成功与否的度量。

## 范式五：Agent 会进化，API 不会——工具要「Agent 就绪」

### 核心洞察

> `deleteItem` 对建 API 的人一目了然，对 **Agent** 只是 schema 里一行；资深工程师的隐式上下文不会自动注入模型。

### 深度解析

第五点：**Agent 会进化，API 不会（agents evolve and APIs don't）**。做过后端的工程师见过大量「自解释」的 API 端点——例如产品微服务里的 **`deleteItem`**，带一个 **ID**，对亲手搭建系统的人含义不言而喻。

**Agent** 看不见代码库与多年领域经验，只能看见**函数 schema、docstring、工具定义**。第一眼它并不「知道」`deleteItem` 会删什么、**ID** 指什么、失败会怎样——资深开发者常默认不必写 docstring，对 **Agent** 却是致命信息缺口。

因此需要建造**真正为 Agent 使用而写的工具**：**语义化接口、自文档化（self-documenting）**，而不是假设「只有写过这套 API 的人才会调用」。这也是用户摘要里强调的重点：**资深工程师最难迁移的，正是他们脑中那些从未写进 schema 的隐式上下文。**

讲者还呼应行业经验：**The Bitter Lesson（苦涩教训）**——软件愈发**可丢弃**；我们会用更好的模型、更好的 **Agent** **反复重建**同一类系统，事物会变。故最后一项原则叫 **为删除而建造（build to delete）**：别在注定被换代的实现上过度浇筑混凝土。

更细的代码级示例见其博客：[philschmid.de/why-engineers-struggle-building-agents](https://www.philschmid.de/why-engineers-struggle-building-agents)。

### 对谈实录

**Philipp Schmid**：「为删除而建造——软件是可抛弃的。我们会用更好的模型、更好的 Agent，一遍又一遍重建同样的东西。」

## 收束：信任但要验证，以及「为删除而建造」

### 核心洞察

> 给模型信任，也要用评测验证；在歧义中做弹性系统，而不是把概率性压成确定性假象。

### 深度解析

**Philipp Schmid** 用五条行动纲领收束全场（与上文五范式一一呼应）：

1. **信任，但也要验证（give trust, but also verify）**——做调度员，不当事事微操的控制器。
2. **别和模型打架（stop fighting the model）**——不要强迫模型走进唯一僵化的「步骤一、步骤二」剧本。
3. **保留语义**——一切皆可成为上下文；应用不再总能用明确定义的数据结构表达。
4. **为恢复而设计**——模型与 **Agent** 不完美，长时运行必然出现怪异情况，系统要能从中恢复。
5. **评测，不要只断言**——在「多少次运行算够好才能给用户」之间找平衡；**Agent** 不是 **100%** 可靠。
6. **为删除而建造**——接受同一 **Agent** 会随更强模型反复重建；软件是可消耗的。
