## 开场：快速上线胜过数月规划

### 核心洞察

> 在 AI 降低构建成本的今天，**数月闭门规划再发布**往往是拖延恐惧的借口，而非更好的策略。

### 深度解析

**Josh Pigford** 开门见山：每次发布都令人心慌——即便做了 **25 年**、上线过**数百款**产品，仍会想「**要是零人在意怎么办？**」

他描述典型工作流：**Claude Opus** 承担大部分规划与实现；完成后用 **GPT** 做审查轮次（片头冷开场口播为「**GPT 3.5**」，正片详述为 **GPT-5.5**，见下文 **[编者注]**），**几乎总是**能找出 **Opus** 漏掉的 **3–5 个 bug**；再跑 **`/but-for-real`** Skill——本质是「**欺负**」模型：「你**几乎肯定**搞砸了一些东西，回去再查。」

**Peter Yang** 追问：通常做多久就发布？**Josh Pigford** 答：有的 **24 小时内**甚至**当天**上线；「花**几个月**做东西再给别人用，我觉得是**很坏的主意**。」

**Peter Yang** 在引子中设定：**Josh Pigford** 以 **400 万美元**出售上一家公司 **Baremetrics**，现**至少五款** AI 产品并行，并将展示完整 AI 技能栈。

### 对谈实录

**Josh Pigford**：「发布新产品总是让人发怵。我干了 25 年，上线过上百个产品，每一次心里都会想：天哪，要是根本没人关心怎么办？」

**Josh Pigford**：「我主要用 Opus 处理大部分工作……然后会用 GPT 5.5 做一遍审查，它几乎总能找出 Opus 漏掉的三到五个 bug。」

**Josh Pigford**：「我觉得在产品给别人用之前花好几个月打磨，是个很坏的想法。」

## /build Skill：研究、规格、分阶段实现与进度文件

### 核心洞察

> **`/build`** 把「调研文档 + 可测分阶段 PR + 跨 工作树 进度文件」产品化，让每次新功能都走同一套可验证流水线。

### 深度解析

**Josh Pigford** 倒序演示 **`/build`**（**GitHub** 开源，托管于 [initialcommit.co](https://initialcommit.co/)）：YouTube 章节概括为 **Research → Plan → Track → Implement**——对应 **调研文档 → 实现规格（分 phase）→ 进度文件（进度文件）→ 按 phase 构建**。

#### 第一步：Research 文档

以 **Botblock → Reply Social** 集成为例：合并旧 **Botblock** 代码库与 **Reply Social** 代码库，产出**高层**调研——API、取舍（Chrome 扩展哪些可弃）、系统如何交互等。**Josh Pigford** 不必通读全文，但会**扫一眼目标**是否跑偏。

#### 第二步：Implementation 文档与阶段

Implementation 命令把功能拆成多个**阶段**——例中 **Botblock** 为 **4 个阶段**，对应 **4 个 PR / 4 个 Git 分支**，避免一次巨型提交。

他配置 Skill：**每个阶段必须用户可亲手测**；复杂功能可达 **30+ 阶段**。构建过程中 Agent 会**开浏览器自测**，但最终 **Josh Pigford** 必须亲手感到「**手感对了**」——慢、缺交互、与想象不符都要迭代。

#### 第三步：`build phase N`（按阶段构建）

对某一阶段再下钻：**真实网页搜索**（竞品如何实现、最新文档——可能用 **Context7**）、**UI.sh skill** 做设计轮次（颜色、组件等）。完成后更新**进度文件**：本阶段做了什么、**已做决策**、系统「学到什么」——供**下一阶段 工作树** 在**零会话上下文**时读取，避免重复犯错与**上下文腐烂**。

**Peter Yang** 归纳：Research ≈ 调研；Implementation ≈ 产品+技术规格；再分 **3–4 个阶段**落地。

#### 与一次性需求

对 **Clearly** 的 GitHub Issue（如 Mermaid **zoom**）：挂 **research skill**（从 **`/build`** 抽出的子能力），附 Issue → 自动调研、给计划 → **Josh Pigford** 批准后开建。竞品调研会**真实浏览网站**，必要时**截图**。

**Peter Yang** 分享：因 **Substack** 无 API，做网站 **Substack off** 时也用类似「看别人怎么做 + **Codex** 自己琢磨出来」路径。

### 深度解析

**Josh Pigford** 答：部分因**上下文**，更因**回滚**——搞砸时可回到检查点；每次**全新会话**也减少**幻觉**。

在 **Conductor** 内可设 **审查模型**（默认模型可改），点 **Review** 按钮触发——YouTube 章节标题写作 **`/adversarial-code-review` skill**，但 **Josh Pigford** 在正片将其描述为 **Conductor 内置的对抗式审查流程**（非与 **`/build`** 同级的独立开源 Skill 文件）。**Peter Yang** 问为何用 **Conductor** 而非只用 **Codex/Claude**——**Josh Pigford** 强调：**多模型切换** + **自动化 工作树 管理**（唯一端口、**10 个**并行实例、复制 **env**、iOS 进程等 setup 命令）。

**Peter Yang** 主持侧对比：**Codex** 与 **Claude** 现已很强，但 **Conductor** 的 工作树 / 换模型是差异化能力——**Josh Pigford** 同意。

**工作树 = 可发货存档点**：隔离上下文、便于回滚，且每个 phase 就是 **100% 一个 PR**。

## /learnings 与 /but-for-real：让 AI 更新 CLAUDE.md 并「真查一遍」

### 核心洞察

> 人类 PR 审查的「第二双眼」被拆成 **GPT 对抗审查 + but-for-real 自我施压 + learnings 写回 CLAUDE.md** 三层。

### 深度解析

### CLAUDE.md（片中 ASR 作「cloud file」）模板

每个项目启动时，按**统一模板**自动生成 **CLAUDE.md**（非手写；**Peter Yang** 亦问及 **agents.md** 式项目规范，片中实际展示为 **CLAUDE.md**）：产品是什么、**用户画像**、营销语气、monorepo 结构（Web + iOS）、**该用的命令与测试方式**（含 **Conductor** 端口变量等）。**Rails** 最佳实践与 **Josh Pigford** 个人偏好（如用 **Agent Browser** 打开页面测试）都写进去。

### /learnings Skill（开源）

某**阶段上线之后**：扫描该 工作树 全部改动 + **会话里反复纠正**的内容（「不对、再试、还不对」），蒸馏成**可写入 CLAUDE.md 的规则**，避免下轮重复犯错。**CLAUDE.md 持续进化**。

### /but-for-real Skill（开源）

在「计划 → 实现」之后单独运行；与 **GPT 审查**互补。语气是**施压**：「你**几乎肯定**搞砸了，回去再查」——通常再抓 **3–5** 个 bug。**Peter Yang** 称这是他见过 **Josh Pigford** 最喜欢的 Skill；**Josh Pigford** 说效果好到「惊人」， frustrated 时有情绪出口——「对机器有时得**凶一点**」；自己平时很客气，崩溃时会**全大写**，「它在乎的。」

Skill 本身用 AI 迭代；**Chops** 内置迭代器，可指导「再狠一点」整份重写。

## 现场演示：如何设计非同质化 AI 审美的网站（以 Rumored 为例）

### 核心洞察

> **品牌先在 Illustrator 里长出来，代码后上；营销文案在产品功能稳定后再生成**——反「先挂落地页收邮箱」套路。

### 深度解析

**Peter Yang** 问：落地页文案、配色、字体是一步还是分步？

**Josh Pigford** 流程：

1. **必须先有产品名**
2. **Logo**：**99%** 在 **Adobe Illustrator** 手绘（钢笔工具、试上千字体）——「还没找到很好的捷径」
3. **配色**同样在 Illustrator 探索（亮/暗/纹理、网站图标场景等）

**Rumored** 案例：概念是 **LLM 断章取义/错误引用**你 → 视觉锚定为 **引号（quotation marks）**；他遍历字体看哪种**引号形状**最有性格，再定**深橙**主色与纹理。

4. 导出 **SVG** 与色板 → **再写代码**
5. **营销页通常在产品建完之后**：功能真实形态要迭代**数天**；稳了后再让 AI 读**全功能集** + 品牌约束生成文案。他想要「**紧迫、略带锋芒的科技感**」，不要软绵绵的典型 SaaS 审美。

**Peter Yang** 原本想问最小可行产品 / 需求验证——发现 **Josh Pigford** **先建完整产品**再谈营销；**Josh Pigford**：现在用 AI **建整个东西很便宜**，且上线前落地页 + 邮箱常是**拖延恐惧**的分心物；「**咬牙做出来，推出去，看发生什么。**」

## 上线、获客、定价与关停：独建者的商业底线

### 核心洞察

> 独建商业线不是「有没有产品市场契合叙事」，而是**有没有人付钱覆盖 API/服务器**——否则关停并退款，不替用户永续补贴。

### 深度解析

### 获客与「没人关心」

主渠道：**社交媒体**——**Josh Pigford** 在 X 约 **6 万**粉丝，「至少能**启动**」。**仍会**上线后零人在意；更大问题是 ADHD 式**很快转向下一项**，旧产品**无人持续宣传**——自认需在「多聊已存在的东西」上改进。

### 定价与产品市场契合
