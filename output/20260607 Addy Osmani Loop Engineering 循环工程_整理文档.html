别再亲自提示代理：Addy Osmani 论循环工程（Loop Engineering）

循环工程（loop engineering） 的本质是用系统设计取代自己当「提示词操作员」——你定义递归目标，让 AI 迭代直至完成。

Loop Engineering 2026-06-07 ~12 分钟阅读

核心导读

循环工程（loop engineering） 的本质是用系统设计取代自己当「提示词操作员」——你定义递归目标，让 AI 迭代直至完成；Codex 与 Claude Code 已内置自动化、工作树、Skills、连接器、子代理与外部状态六类原语，工具形态趋同后争论该用哪款反而次要；但循环越顺，验证责任、理解债务与认知投降三类风险反而更尖锐——杠杆点从写提示词移到了设计循环，工程师身份不能丢。

范式转移。 约两年里，从编码代理拿结果靠「写好提示词 + 喂够上下文 + 一轮轮手动接棒」。Peter Steinberger 与 Anthropic Claude Code 负责人 Boris Cherny 的共识是：别亲自提示了，去设计让代理互相提示的循环。新做法是搭小系统——自动发现工作、分发、检查、记账、决定下一步——由系统去戳代理，而非人手逐轮对话。

六块积木。 循环需要五件东西加一处记忆：自动化（定时发现与分诊）、Git 工作树（并行隔离）、Skills（项目知识外化）、插件/连接器（接入工单、数据库、Slack 等）、子代理（构思与验证分离）；第六是落在磁盘或 Linear 上的状态——模型跨轮次遗忘，仓库与 Markdown 文件不会。

收束与警示。 完整循环可晨间分诊 CI 失败、开隔离工作树修 bug、子代理审查、连接器开 PR 并更新工单；你设计一次，不再逐步提示。Addy Osmani 自承仍处早期、对 token 成本保持怀疑；若不亲自审代码、全靠循环修产品，质量会螺旋下沉。同一套循环，有人用来加速已深度理解的工作，有人用来逃避理解——循环分辨不了，工程师必须分辨。

从提示词到循环：编码代理工作方式的新范式

核心洞察

手动逐轮提示正在退场；工程师的新杠杆是设计会自我驱动的循环，而非写更好的单条提示词。

深度解析

Addy Osmani 开篇定义：循环工程是用系统设计取代自己当「提示代理的人」。此处的「循环」可理解为递归目标——定义目的后，AI 迭代直至完成。他认为这可能是与编码代理协作的未来，但仍处早期，须对 token 成本保持警惕（「token 富裕」与「token 拮据」两种用法模式差异极大）。

Peter Steinberger 近期表示：「你不该再亲自提示编码代理了，而该设计循环去提示你的代理。」Boris Cherny 同理：「我不再亲自提示 Claude；我跑着循环去提示 Claude、决定下一步，我的工作是写循环。」

过去约两年的主流路径清晰：写好提示词、共享上下文、输入—阅读—再输入；代理是工具，人全程握持，一轮接一轮。这一模式「差不多结束了」——至少部分从业者如此判断。

新路径是构建小系统：发现工作 → 分发 → 检查 → 记录完成项 → 决定下一步，让系统去驱动代理。Addy Osmani 此前写过近亲概念代理编排层工程——为单个代理打造运行环境；还有工厂模型——构建软件的那套系统。循环工程坐在编排层再上一层：编排层加上定时器、拉起小助手、自我喂养。

令他意外的是，这已不太是「自己写 bash 脚本维护一辈子」的工具问题。一年前想要循环，得堆一堆 bash 并永久维护；如今组件直接随产品出货。Peter Steinberger 的清单几乎一一对应 Codex app，再几乎同样对应 Claude Code。一旦看出形状相同，就不必争论工具——设计换座席仍能运转的循环即可。

循环的六块积木：Codex 与 Claude Code 的对照

核心洞察

循环 = 五类能力原语 + 一处外部记忆；两大产品在能力上已对齐，差异主要在命名与入口。

深度解析

Addy Osmani 先列清单，再映射到产品：

1. 自动化（Automations）：按日程自动发现与分诊 2. Git 工作树（Worktrees）：并行代理互不踩文件 3. Skills：写下代理否则会瞎猜的项目知识 4. 插件与连接器（Plugins / Connectors）：接入已有工具链 5. 子代理（Sub-agents）：一个出主意，另一个检查 6. 状态/记忆（State）：Markdown、Linear 看板等，存「已完成」与「待办」

第六项听起来「蠢到不值一提」，却是所有长运行代理的同一诀窍——Addy Osmani 在「长运行代理」一文中展开：模型在运行之间遗忘一切，记忆必须在磁盘上而非上下文里。「代理会忘，仓库不会。」

两大产品现已具备上述五类能力（加状态）。对照表如下：

| 原语 | 在循环中的职责 | Codex app | Claude Code | |------|----------------|-----------|-------------| | 自动化 | 定时发现 + 分诊 | Automations 标签页：选项目、提示词、频率、环境；结果进 Triage 收件箱；/goal 用于跑到完成为止 | 定时任务与 cron、/loop、/goal、钩子（hooks）、GitHub Actions | | Git 工作树 | 隔离并行特性 | 每线程内置工作树 | git worktree、--worktree、子代理上设 isolation: worktree | | Skills | 固化项目知识 | Agent Skills（SKILL.md），$name 或隐式调用 | Agent Skills（SKILL.md） | | 插件/连接器 | 连接工具 | 连接器（MCP）+ 插件分发 | MCP 服务器 + 插件 | | 子代理 | 构思与验证 | .codex/agents/ 下 TOML 定义 | .claude/agents/ 下 Task 子代理、agent teams | | 状态 | 跟踪进度 | Markdown 或经连接器接 Linear | Markdown（AGENTS.md、进度文件）或经 MCP 接 Linear |

名称略有不同，能力实质相同。Addy Osmani 强调细节决定循环是「撑得住」还是「悄悄到处漏」——下文逐项展开。

自动化：循环的心跳

核心洞察

没有按节律触发的自动化，就只是一次性运行；/loop 与 /goal 则是会话内的近亲原语。

深度解析

自动化把「跑过一次」变成真正的循环。

Codex app：在 Automations 标签页创建——选项目、提示词、频率、本地检出或后台工作树。有发现的运行进入 Triage 收件箱；无发现则自动归档。OpenAI 内部用于日常工单分诊、汇总 CI 失败、写提交简报、猎杀上周新引入的 bug。自动化可调用 Skill，用 $skill-name 替代把巨型指令墙贴进无人维护的日程。

Claude Code 路径不同、目标相同：用 /loop 按间隔重跑提示或命令；可排 cron；用钩子在代理生命周期节点触发 shell；或推上 GitHub Actions，关笔记本后继续跑。定义自主任务、给节律、把发现推给你——不必自己四处巡检。

会话内还有第二原语，更接近全文主题：

- /loop：按节律重跑 - /goal：直到你写的条件实际为真才停；每轮后由另一个小模型判断是否完成——写代码的代理不负责给自己打分。例如「test/auth 下全部测试通过且 lint 干净」，然后走开。Codex 同名 /goal：跨轮次工作直至可验证停止条件成立，支持暂停、恢复与清除。

自动化负责浮出水面的工作；循环其余部分负责动手处理。

Git 工作树：并行代理的隔离

核心洞察

多代理并行时，文件碰撞是首要失败模式；工作树用独立检出目录隔离编辑。

深度解析

一旦同时跑多个代理，文件就开始冲突——两个代理改同一文件，等同两个工程师没沟通就抢同一行。Git 工作树提供独立工作目录、独立分支、共享同一仓库历史，一个代理的编辑物理上碰不到另一个的检出。

Codex 内置工作树，多线程可同时打同一仓库。Claude Code 用 git worktree、--worktree 开独立会话，或在子代理上设 isolation: worktree，每个助手得新鲜检出，结束后自清理。

Addy Osmani 在「编排税（orchestration tax）」一文中写过人的一侧：工作树消除机械碰撞，但人仍是天花板——审查带宽决定实际能跑几个并行，而非工具本身。

Skills：把项目知识写进磁盘

核心洞察

Skill 把每轮冷启动的「意图猜测」写成一次性的外部知识，循环才能复利而非从零重推。

深度解析

Skill 让你不必像金鱼一样每个会话重复解释项目上下文。两工具格式相同：文件夹内 SKILL.md 承载指令与元数据，可选脚本、参考、资源。

Codex：$ 或 /skills 调用，或任务匹配描述时自动触发——因此紧凑、平实的描述胜过花哨文案。Claude Code 同理；Addy Osmani 曾在「agent skills」一文详述模式。

意图债务（intent debt） 视角：代理每会话冷启动，意图空洞会被自信地填满。Skill 把意图写在外面——规范、构建步骤、「我们因某次事故不那样做」——写一次，每轮读取。无 Skill，循环每周期从零重推整个项目；有 Skill，知识复利。

须分清：Skill 是创作格式，插件是分发方式。跨仓库共享或打包多个 Skill 时，打成插件——Codex 与 Claude Code 皆然。

插件与连接器：循环接入真实工具链

核心洞察

只能看文件系统的循环很小；基于 MCP 的连接器让循环在真实环境里开 PR、更工单、发 Slack。

深度解析

连接器基于模型上下文协议（MCP），让代理读工单系统、查数据库、调用预发布环境 API、往 Slack 丢消息。Codex 与 Claude Code 都讲 MCP，为一边写的连接器通常另一边也能用。插件把连接器与 Skills 打包，队友一键安装整套，不必凭记忆重建。

差别在于：代理说「这是修复方案」vs 循环自己开 PR、链 Linear 工单、CI 绿后 ping 频道。连接器是循环能在真实环境里行动、而非只描述「若能的话我会…」的原因。

子代理：写者与审查者分离

核心洞察

循环无人值守时，可信赖的审查子代理，是敢走开的唯一理由。

深度解析

循环里最有用的结构，是把写与查拆开。写代码的模型批改自己作业时过于宽松；不同指令、有时不同模型的第二个代理，能抓住第一个「说服自己」的漏洞。

Codex：仅在请求时拉起子代理，并行运行后折叠进单一回答。在 .codex/agents/ 用 TOML 自定义——名称、描述、指令，可选模型与推理力度；安全审查可用强模型高力度，探索可用快速只读代理。

Claude Code：.claude/agents/ 子代理与 agent teams 传接工作。常见分工：一个探索、一个实现、一个对照规格验证。

Addy Osmani 曾在「代码代理管弦乐（code agent orchestra）」与「对抗式代码审查」中论证此点。在循环里尤其关键：循环在你不看时运行，你真正信任的验证者才是敢走开的理由。子代理消耗更多 token（各自模型与工具调用），应把预算花在「第二意见值得付费」之处。

Claude Code 的 /goal 底层同理：新鲜模型判断循环是否结束，而非干活的那个——制造者与检查者分离应用到停止条件本身。

一个完整循环的实战形态

核心洞察

单线程可变成控制面板：晨间分诊 → 隔离修复 → 审查 → 连接器交付 → 状态文件续跑。

深度解析

Addy Osmani 常用形态如下：

1. 自动化每晨在仓库上运行；提示词调用分诊 Skill，读昨日 CI 失败、开放 issue、近期提交，把发现写入 Markdown 或 Linear。 2. 对每个值得处理的发现：开隔离工作树，子代理 A 起草修复，子代理 B 对照项目 Skills 与现有测试审查草稿。 3. 连接器开 PR、更新工单；循环处理不了的进 Triage 收件箱等人处理。 4. 状态文件是脊柱——记尝试过什么、什么通过、什么仍开放；次日晨跑从今日停处续上。

实际发生的事：设计一次，没有逐步提示任何一步——这正是 Peter Steinberger 观点的落地；且 Codex 与 Claude Code 用的是同一套积木。

循环解决不了的三件事

核心洞察

循环改变工作方式，不删除工程师；越顺的三类风险——验证、理解、认知——反而越尖锐。

深度解析

1. 验证仍在人身上

无人值守的循环也是无人值守地犯错。把验证子代理与制造者分开，是为让循环的「完成了」有点意义——即便如此，「完成」是主张而非证明。Addy Osmani 反复引用「AI 时代的代码审查」同一句：你的职责是交付你确认能用的代码。

2. 理解会腐烂（理解债务，comprehension debt）

循环越快交付你没写的代码，代码存在与你真正掌握之间的鸿沟越大。理解债务；顺滑的循环若不读产出，只会加速堆积。

3. 舒适姿态最危险（认知投降，cognitive surrender）

循环自跑时，极易停止有主见、照单全收。Addy Osmani 称之为认知投降。用判断力设计循环是解药；为逃避思考而设计循环是加速器——同一动作，相反结果。

搭建循环，但留住工程师身份

核心洞察

循环设计比提示工程更难，因杠杆点移动了；同一循环可加速深度理解的工作，也可纵容逃避理解。

深度解析

Addy Osmani 认为这是工作方式演化的预览。但若他不亲自审代码、全靠自动化循环修产品，产品质量会受损，可能陷入螺旋下沉、越挖越深。

他仍鼓励搭建循环，同时提醒直接提示代理依然有效——关键是平衡。

两人可搭完全相同的循环，结果却相反：一人用来加速已深度理解的工作；一人用来逃避理解。循环分辨不了，人能分辨。

这使循环设计比提示工程更难，而非更易。Boris Cherny 的意思不是工作变轻松，而是杠杆点移动了。

收束句：搭建循环——但要像打算继续做工程师的人去搭，而不只是按启动键的人。

Addy Osmani 文末推广其 O'Reilly 著作《Beyond Vibe Coding》，涵盖规格、编排层、评测、上下文与用 AI 交付生产级软件等主题。
