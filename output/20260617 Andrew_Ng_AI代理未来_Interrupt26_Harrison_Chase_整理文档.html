Interrupt 26 · Andrew Ng × Harrison Chase 2026-06-17 ~19 分钟阅读

核心导读

编码代理正以「每三个月变一次前沿能力」的速度改写个人与团队交付方式，但把软件造快 10–100 倍并不会消灭瓶颈，只会把约束挪到产品管理、法务、营销与数据架构；企业若只做单点提效而缺少自上而下重设计工作流，就换不来 AI 承诺的转型——增长型用例、保留供应商选择权、以及面向代理的非结构化数据重构，才是下一阶段的硬功课。

快慢错位的一年。 炒作与末日叙事（含「工作末日」）都超出 Andrew Ng 预期；真正快于预期的是编码代理——他六个月前几乎只用 Claude Code，如今已混用 OpenAI Codex、Gemini CLI 与 OpenCode，甚至在手机上写代码。代理式工作流也开始进入企业，节奏「感觉不错」。

小团队、宽护栏、多角色一人扛。 软件交付变快后，产品管理瓶颈恶化成「全面瓶颈」：营销跟不上功能解释、法务跟不上日级发布、设计同样滞后。Andrew Ng 倾向 1–10 人的高上下文通才小队，在宽护栏内狂奔，甚至用 AI 起草服务条款再交律师定稿——工程背景者最常见，但产品经理、营销、运营也在学编码参与。

积木、文档与教育。 未来工程师像拼 LEGO：掌握 检索增强生成（RAG）、代理框架、评测、护栏（guardrails） 与 UI/身份/数据库等积木，组合式快速组装；新积木常超出模型知识截止日期——Context Hub（chub） 为代理提供最新 API 文档。DeepLearning.AI 与 LangChain 长期合作短课；CodeDream.ai 用「对话式一对一视频 + 可点击交互 JavaScript」试探教育形态，但 Andrew Ng 认为教育大转型仍被过度炒作。

企业：自下而上不够，要重画贷款流程。 「千花齐放」式自下而上创新大多只产出点状提效；以银行贷款为例，只自动化审批中间一小时是增量，「10 分钟获批」 产品才需要营销、路由、数据基建与尽调全链路重设计。ROI 难有一刀切公式；他更兴奋于可衡量且真正改变业务的「挥棒打本垒打」项目组合——原型成本已暴跌，但资源分配仍常需自上而下。前线部署工程师（FDE） 有价值但炒作大于现实；LangSmith 等供应商中立观测层与一年期合同保留选择权同样关键。开源/开放权重模型 落后前沿约六至九个月却足够实用；非结构化数据 的治理、权限与碎片化，将催生数千万至数亿美元级重构项目——MongoDB 等 NoSQL 则因模式灵活而契合代理时代的快速原型。

过去一年的 AI 节奏：炒作、末日叙事与编码代理

核心洞察

炒作与「工作末日」叙事都超预期；真正快于预期的是编码代理前沿与手机端工作流。

深度解析

Harrison Chase 开场点出：距去年 Interrupt 一年间 AI 变化极多；去年 Andrew Ng 炉边对谈是大会回放中最受喜欢、观看最多的一场。Andrew Ng 被问「什么比预期更快、什么更慢」时，先划两条线：炒作超出预期；末日叙事（含工作末日，他认为不会发生）也比预期更有市场——这是「慢于期望的理性共识」一侧。

积极侧：编码代理 起飞速度超出他原先猜测。虽「AI 每三个月变天」并不完全成立，对编码代理而言接近真实——前沿能力竞争激烈。约六个月前他几乎全用 Claude Code；如今仍大量 Claude Code，也越来越多用 OpenAI Codex，并混用 Gemini CLI 与 OpenCode（因其开源）。工具组合切换极快；他亦未料到一年前会在手机上写这么多代码——办公室 Mac Mini 等工作流都在变。

代理式工作流 也开始进入企业，「感觉相当不错」。

编码代理重塑软件工程：产品管理瓶颈与全能小队

核心洞察

软件快 10–100 倍后，瓶颈从 PM 扩散到营销、法务、设计——未来团队可能是 2 人扛 5 种职能。

深度解析

Andrew Ng 约一年前提出 产品管理瓶颈（product management bottleneck）：若构建变快，决定做什么、定范围、收客户反馈 的产品管理工作成约束。过去一年该瓶颈「以好的方式变得更糟」——因为交付确实更快了。

但现象更广：当写软件快 10 倍或 100 倍，几乎一切 都成了瓶颈。他的团队已见 营销瓶颈——功能太多，营销难以弄清新功能是什么、如何对外讲。法务同理：以前三个月做一个需合规的产品，等法务一周尚可；现在一天造完却等法务一周，即成 法务合规瓶颈。设计亦然。

他常思考未来软件团队如何组织——尚无定论。趋势是极小规模编制：1–10 名工程师 的团队，成员多为高上下文、高授权通才，在很宽的护栏内狂奔交付，甚至写传统上不属于工程的 营销文案。若一个团队需要工程、产品、服务条款、营销文案、设计五种职能，却只有两名人类，由抽屉原理（鸽巢原理） 每人必兼多职。

好消息：AI 不会让他变成好营销人，但「没那么烂」——高上下文、深度技术、又能用前沿工具兼一点其他角色的通才，流程明显加速。工程师已用 AI 起草 服务条款 初稿，再交律师润色后发布。

「我写产品管理瓶颈时观察到：构建变快，决定做什么就成瓶颈——过去一年这瓶颈以好的方式恶化了。但快 10 或 100 倍时，几乎一切都成瓶颈——营销、法务、设计……我在设 1–10 人的高上下文通才小队，宽护栏内狂奔，甚至写营销文案。两人团队要代表五种职能，每人必兼多职。AI 没让我变成好营销，但没那么烂——这些流程让团队快很多。」

通才从哪来：工程底色与跨职能扩张

核心洞察

工程背景者最常见，但产品、营销、运营也能学编码加入——方向不限，工程理解前沿略占优势。

深度解析

Andrew Ng 密切合作者多为深厚工程与技术背景，又是高授权、略通才、可跨入他岗并借 AI 补技能缺口的人。成功路径不限方向：产品经理加强编码后参与这类团队；营销人员有效学编码；运营人员开始造更多产品——任何背景都可能，但当前人数上工程出身最多，因 AI 编码与理解前沿技术天然略占优。

进入 AI 软件工程：积木式组合与 Context Hub

核心洞察

掌握足够「积木」并组合式组装，比从零写一切更快；新 API 需 Context Hub 给代理最新文档。

深度解析

Andrew Ng 用 LEGO 积木 比喻：若只有白色砖只能搭无聊东西；黑白黄棕绿加曲线件，组合空间指数级增长。计算机科学一直有优秀积木；代理式编码 时代 AI 与非 AI 积木都在激增——检索增强生成（RAG）、代理框架、评测（evals）、护栏（guardrails），以及 UI 组件、身份机制、前后端持久化等。

精通足够多积木的开发者，可组合式极快造软件。DeepLearning.AI 提供大量短课帮人们掌握各供应商积木；另一挑战是用编码代理快速组装积木成目标软件。

痛点：许多积木太新，代理不知如何调用——例如 Nano Banana 在不少领先模型 知识截止日期之后 发布，代理甚至不知存在、更不知 API 语法。Context Hub（chub） 是他与 Rohit Prasad 推动的项目：像「给 AI 代理用的 Stack Overflow」，让代理获取最新 API、SDK、积木文档，并可反馈文档以改进。Andrew Ng 个人对一些 API 语法嫌烦，用 Context Hub 加载最新文档后让代理代劳调用，显著加速自己的编码。

Harrison Chase 插话：LangChain 也有名叫 Context Hub 的产品，但是不同东西——Andrew Ng 的 chub 对编码代理更有用，应去用那个。另：LangChain 是 DeepLearning.AI 课程合作方中仅次于 OpenAI 的第二家；许多人通过 DeepLearning 课程认识 LangChain。

「我脑中有积木模型——检索增强生成、框架、评测、护栏，还有 UI、身份、数据库……精通足够积木就能组合式快速搭建，像 LEGO：颜色越多组合指数增长。DeepLearning.AI 有很多短课；还要用编码代理组装。问题是新积木代理不会用——Nano Banana 在知识截止后才发布，代理不知道。Context Hub 是 Rohit 和我做的：代理版 Stack Overflow，拿最新文档还能反馈改进。LangChain 也有 Context Hub 名字，但是不同东西——他的对编码代理更有用。」

DeepLearning.AI、LangChain 与教育实验

核心洞察

该学什么在变（积木+PM 通识），但教育「大转型」仍被过度炒作——CodeDream.ai 在试对话式交互课。

深度解析

DeepLearning.AI 在训练内容与交付方式两侧实验。开发者要学：编码代理、积木、一点 产品管理 等通识技能——「学什么」已显著变化；Coursera 与 DeepLearning 已提供大量相关训练。

「怎么学」的变革则尚未真正到来——他们几周前预览上线 CodeDream.ai：愿景不是上网课，而是来对话。体验是模拟与 Andrew Ng 一对一视频：可后仰听他讲 Context Hub 与编码代理，也可随时打断 AI 提问。他个人大量实验用 JavaScript 替代视频与幻灯片——演示区可点击、可输入自己的提示词或查询，静态视频变交互区（已上线：不是录屏而是 JavaScript 共享，可交互运行）。

教育大转型被过度炒作；变革在路上，但今天上网课仍希望有更好形态——比十年前课程更互动（如当天刚上线 AMD Sharon Jo 的 Transformers 新课，可视化与可玩组件增多），更大跃迁他仍在深思。

企业 AI 落地：自下而上失灵与贷款审批案例

核心洞察

「千花齐放」自下而上创新大多只产点状提效；贷款案例说明须自上而下重设计全链路才换增长。

深度解析

AI Aspire（Andrew Ng 与 Chris Tann 共创的 AI 顾问公司）常对话 Fortune 50/500、G2000 级企业。一致主题：大家都投过自下而上创新、「千花齐放」，大多未见回报——CEO 与董事会问 AI 的 ROI 在哪。他仍主张继续自下而上，但这类创新常产出点状方案、带来增量效率——好事，却不是 AI 承诺的更广转型。

银行贷款承销 例：流程五步——营销产品、收申请、审批准、最终尽调、执行放款。多团队发现中间审批可用 AI：人审一小时变 AI，值得做。但若整条承销流程不变、只省一小时人工，只是小增量效率。

部分银行转向：不做这点效率，而重想工作流，营销 「10 分钟获批」 贷款——不必等一周才有人空出一小时审件，申请可立刻送 AI 决策。落地难在需更广范围 的人重设计：要营销新产品、申请立刻路由审批、营销与数据基建介入、AI 做初判、尽调与执行也要扩容。自下而上 产创意，须配 自上而下 推动改全链路步骤才能促增长。

对谈实录

1

Harrison Chase

「企业采用 AI 比您预期快还是慢？正确做法是什么？」

2

「AI Aspire 跟 Fortune 50/500 谈策略。一致的是：大家都投自下而上、千花齐放，大多未见回报——CEO 问 ROI。仍要继续，但常是点状提效，不是更广转型。银行贷款五步骤——若只自动化中间审批一小时，流程不变，只是小增量。有的银行改做『10 分钟获批』——要重设计营销、路由、数据基建、尽调扩容。自下而上很有价值，但要配自上而下改全链路才能促增长。」

省成本还是促增长：ROI 与本垒打组合

核心洞察

省钱有顶、增长无顶；本垒打项目可测但更需创意——300 条想法表格里筛少数大注常靠自上而下。

深度解析

许多企业谈 降本——可以做，但 Andrew Ng 更推动 更有想象力的增长：钱只能省那么多，增长几乎无上限。10 分钟贷款（银行真实推进）、客服/联络中心——不只省钱，自动化后可更快服务更多客户、体验更好从而促增长；汽车餐厅语音点餐 自动化亦带来更愉悦体验与增长。更多案例在不同行业涌现，部分因权限未公开。

ROI 怎么量？ 他坦言「我也给不出一刀切答案」——企业太多元，像量整个业务，难有一刀切公式。最让他兴奋的项目应测、值得测；也有些 挥棒打本垒打 式项目创造巨大价值——不是争论磁盘驱动 2% 增长减 1% 实施成本，而是一目了然在转型（上市公司仍须量，但性质不同）。

反直觉：驱动增量有时比驱动转型更难——老板要明年 2% 改善，大家只能更努力一点；若要 20% 或 50% 增长，不能全员多干 50%，必须更有创意。

AI Aspire 经验：企业常寄来数百条想法 的电子表格——一家金融机构 300+ 条，请帮忙筛该投哪些。分析极难；面对这么多想法，要技术可行性 + 业务影响双重分析，缩到少数值得重仓的下注。挥棒打本垒打 型多为 自上而下；他建议组合几笔深思熟虑的大注，而非一次孤注一掷。

代理式编码 让实验与原型成本暴跌——他们不断做原型；但 10 万美元预算 也无法全做，重仓仍需资源分配，故常需略 自上而下 推动。

前线部署工程师、供应商锁定与 LangSmith

核心洞察

FDE 有价值但炒作被放大；多数公司仍是内部工程师为主、少量 FDE——合同与工具层要保留模型选择权。

深度解析

前线部署工程师（Forward Deployed Engineer，FDE） 在硅谷热议中很火（Aaron Levie 一两天前亦上台分享）。Andrew Ng 认为 FDE 是好主意，多数公司未来仍是大量内部 AI 工程师 + 较小嵌入式 FDE 团队——比例上内部远多于 FDE；炒作 略大于现实，但仍是好事，应多创造 FDE 就业。

构建代理工作流 很难：懂业务、面向客户、可靠性、可观测性、评测、对客户反驳不可行方案、与利益相关方定自动化范围、变更管理——需深度技术判断；嵌入式 FDE 能加速。

另一难题：能否供应商中立的 FDE？ AI 领先模型 变化极快——他不知一年后领先模型或领先 编码代理 是谁；不确定期 选择权极有价值。许多供应商给 20–30% 折扣 但要 三年合同——他个人几乎从不签超过一年，不论折扣，以保留一年后选最好供应商的权利。

企业亦问：若让某家供应商的少数 FDE 嵌入并深度绑定单一模型，一两年后选择权损失多少？ 这是他个人常用 LangSmith 的原因——Harrison Chase 把它做得易用；这类 供应商中立 工具对观测、长期保留选择权很有价值。供应商很好，要合作，但长期保留选择权也重要。

深度解析

先修数据战略再建代理：AI Aspire 见大型企业最常见痛点之一是 重思数据架构。过去 10–20 年精力在结构化数据（表、关系型、电子表格）——仍重要；但 AI 能处理非结构化（文本、图像、PDF、音频、视频），在正确时间地点把非结构化数据组织好交给代理 突然比过去有价值得多。

市场很多供应商谈非结构化，但他找不到从政治上满意的单一好方案——AI Fund / AI Aspire 内部在做「疯狂实验」自建重构；若成功会多谈。他预见：像当年结构化 数据架构 大工程一样，未来数年许多企业会有 数千万至数亿美元级 项目，让数据更 AI 就绪 / 代理就绪。

现有架构为何未就绪？碎片化、治理、数据散落、无共识模式、数据在某人笔记本、权限为人类设计非为代理——代理继承我的权限吗？治理与可观测性怎么管？金融服务业大量 PDF 合规保留 20 年无人读——现在 AI 能看，整理极有价值。

Harrison Chase 提到 CJ 稍后讲链接技术；Andrew Ng 顺带编码小经验：他个人大量用 MongoDB——关系库虽好，但快速原型时 重设计数据库模式 太烦；虽 AI 做迁移 几乎从不 删库，「几乎从不但不是从不」仍烦。NoSQL 可写入任意数据、读时再定模式，迭代快；不一定撑最大生产负载，超大企业终用关系库，但 NoSQL 比多数人想更可扩展，支撑代理时代节奏。
