## 二十五年工程师生涯：从 Vim 到 IDE 再到 Agent

### 核心洞察

> 工程工具的阶跃——Vim、IDE、在线发布、Agent——一再改写「什么叫硬核」；Fiona 的犬食基因从「用 VS 编辑器写 VS 编辑器」延续至今。

### 深度解析

Fiona 在 **IBM** 实习时做 **DB2** 操作系统服务，栈越底层越「硬核」是当年的直觉；工具主要是 **Vim** 与终端调试，几乎不用 IDE。2000 年互联网泡沫破裂后，微软 offer 对毕业生弥足珍贵——她甚至不知道 **Visual Studio** 是什么，曾问经理这是不是「更好的画图程序」。

加入微软是她第一次用 IDE：断点、多线程调试，阶跃式震撼。她在 **Visual Studio 编辑器**团队，用 VS 编辑器开发 VS 编辑器——**犬食（dogfooding）** 从此成为职业信条：先为自己和队友做出愉悦体验。那时没有社交媒体式即时用户反馈，但 VS 团队人人都是重度用户，内部反馈极快。

另一转折是发行方式：早年软件刻 **CD**，制造 deadline 极硬，规划因此很重；转向在线发布后，发布节奏改变。她在演讲中把这条线与今天对照：**Claude Code** 与 **Cowork** 时代，编码不再是瓶颈，验证与野心才是新焦点。

个人起点同样曲折：高中本想当视觉艺术家，九年级才有电脑；爱上编程是因为「像艺术一样把想法做出来」。为付学费当过银行柜员，周末打工、暑假全职，贯穿大学与 dot-com 寒冬——「恐惧时问：我能控制什么？」成为日后劝工程师的底色。

### 对谈实录

**Lenny Rachitsky**：「人们可能忘了，不久前 **100%** 代码都是人写的；现在快变成 AI 写。**Boris Cherny** 说过 coding is salt（编码是盐）——你们昨天发的图，工程师季度合入代码约为 2021–2025 同期的 **8 倍**，曲线平平然后冲天。」

**Fiona Fung**：「从 IBM 到微软就是一大跳。微软是我第一次用 IDE——断点、多线程调试，阶跃式变化。我最喜欢的是我在 VS **编辑器**团队，用 VS 编辑器写 VS 编辑器，犬食之爱就从那儿来的。」

**Fiona Fung**：「我们当年还用 CD 发行软件，deadline 非常硬——必须赶上制造。工程时间极其珍贵，你会做很多规划。现在编码不是瓶颈了；**Claude Code** 和 **Cowork** 团队里设计师、PM 都在合代码，吞吐这么高，我们怎么想**验证**？这是我看到的新转变。」

## 编码不再是瓶颈：8 倍产出与验证新时代

### 核心洞察

> 吞吐暴涨后，稀缺从「谁能写代码」变成「谁能证明代码正确、体验好、有业务结果」。

### 深度解析

Anthropic 公开数据：工程师平均每季度合入代码量约为 2021–2025 同期的 **8 倍**——曲线长期平稳后陡升。Fiona 在 **Code with Claude** 活动上的核心判断：**编码不再是瓶颈**，它抬高了个人能力的上限；理论上一切都可能，问题变成**你有多野心**。

与「8 倍代码」并行的组织变化：**Cowork** 团队跨职能合代码；吞吐越高，人工 code review 越不可能兜住一切——去年甚至没有 **Claude Code Reviews**，人类审查曾是明显瓶颈。对策是把「什么才是好」写成可检查框架：**规格、内容设计标准**检入仓库并与代码同步更新，让 **Claude Code Review** 对照框架自动校验——类似测试驱动开发（TDD）的演化：测试生成曾是「先吃西兰花」的税，现在模型可先写测试、再修 bug、再让测试变绿。

Fiona 回忆自己第一次在 **Claude Code** 上做 TDD：「帮我先写测试，确保失败，再修，再变绿」——曾经痛苦的纪律，因自动化而重新可行。

### 对谈实录

**Fiona Fung**：「**编码不再是瓶颈**——它抬高了任何人能做到的上限。理论上一切皆可能，现在是你能有多野心。」

**Fiona Fung**：「说实话，去年我们连 **Claude Code Reviews** 都没有——人类 reviewer 曾是巨大瓶颈。把『好是什么样』写进 repo，**Claude Code Review** 对照框架检查——这就像 TDD 的演化：以前写测试是税，现在模型帮你写。」

**Lenny Rachitsky**：「这不公平——它先帮你把测试写好。」

**Fiona Fung**：「[笑] 对。」

## 2026 年 AI 原生软件团队长什么样

### 核心洞察

> 人人趋向「建设者」；管理者用常驻 **Claude Code** 会话与 **routine** 在 8 倍吞吐下保持可见性与对话质量。

### 深度解析

Fiona 对「2026 年 AI 原生（AI-pilled）软件团队」的概括：**角色在爆炸，人人开始是建设者（builder）**。

她的管理技法之一：在所有代码仓库部署 **Claude Code** 远程会话实例，可读全库工作、接入 Slack、对照团队追踪的一切指标。每月与下属「一起看回放」——共享屏幕开 **Claude Code** 会话，问：本月重点是什么？交付了什么？市场反馈如何？反馈渠道里有什么？过去这类会话可能只用来生成 PR；现在用来**开启关于影响力与质量的对话**。

晨间仪式也曾是：咖啡 + 扫反馈频道，看能否自己上手修缺口。约一两个月前 **routine** 上线后彻底改变：设定每日定时任务，自动汇总反馈主题，甚至准备好可审 PR。反馈来源极杂：内部频道、邮件、合作方、朋友私信、LinkedIn——「需要 **Claude** 帮我顶住信息量」。

**Cowork** 对她个人同样魔法：差旅报销等厌恶事务自动化；她把产品带给小企业朋友 onboarding，既发现 onboarding bug，也看到意想不到的用法（在「垃圾抽屉」文件夹里找菜单、做本地竞品定价分析）。

### 对谈实录

**Lenny Rachitsky**：「所以你用 **Claude** 不只是让人交付，而是交付得更好、理解有没有影响力？」

**Fiona Fung**：「对。shipping 之外：市场上表现如何？有没有引入 bug？我有句话：**犯新错误（make new mistakes）**——可以犯错，但要犯新的，这样总在学。追求零错误往往意味着太慢、太谨慎。」

**Fiona Fung**：「我以前是起床、咖啡、看反馈频道。一两个月前有了 **routine**，现在自动化了——醒来就有摘要，有时还有 PR 等我 review。」

**Lenny Rachitsky**：「以前一季列个 bullet：这个功能、那个功能；现在 8 倍产出，这是你跟上一切的少数办法之一。」

## 招聘画像：产品嗅觉建设者与系统深度专家

### 核心洞察

> 两类人：**有产品嗅觉的创意建设者** + **硬核系统专家**——模型越强，后者在「信任但验证」处越不可替代。

### 深度解析

Fiona 在 **Code with Claude** 演讲中的招聘框架：

1. **创意建设者 + 产品嗅觉**：「梦想家」——对产品有热情，有想法就动手，看反馈、打磨、端到端负责体验。
2. **深度系统专家**：她刚加入 **Claude Code** 时发现产品通才很强，但缺分布式系统背景——「信任但验证」：模型很强，仍有许多领域需要深度专业知识把关。

这与「野心」叙事一致：一位工程师本非移动端专家，却用 **Claude** 把功能扩展到移动面；另一位 **10 倍工程师** 的故事——以前听说某功能「太难」会拒绝，现在「让 **Claude Code** 做就行」。心态从「解锁小事」变成「理论上一切皆可能，看你敢想多大」。

### 对谈实录

**Fiona Fung**：「一类是**产品嗅觉**的建设者——有激情、有想法就 build，看反馈、迭代、打磨体验，端到端负责产品。另一类是**深度系统专家**——我加入时发现缺系统背景的人；模型再好，验证仍需要专业知识。」

**Lenny Rachitsky**：「这让我想到**野心**——一位 10 倍工程师说，以前别人说『我们该做这个』他会想『太难』，现在想『太可能了，问 **Claude Code**』。一切理论上可能，问题是你有多野心。」

**Fiona Fung**：「完全共鸣。昨天和同事聊：他不是移动端工程师，但我们需要移动面——**Claude** 成了搭档。它抬高了任何人能做到的天花板。」

## 适应与分化：谁适应良好、谁在挣扎

### 核心洞察

> 适应者：成长心态 + 好奇「向内探」；挣扎者：常夹杂恐惧与「一切失控」感——对策是找一件可控的行动。

### 深度解析

Fiona 区分适应良好与挣扎中的工程师：

**成长心态**在 AI 之前就有价值；她从微软到 Meta 第一年才真正理解「永远在学习」——过去让你成功的方式，未来可能不再适用；改变成功公式令人害怕。

对**挫败感**，她常看到**恐惧**：进化上恐惧合理，但建议「向内探」——「我能做什么？什么在我控制范围内？」把「发生在我身上」改成「为我发生」。

个人故事：高中想当艺术家、付不起学费的恐惧；主动申请加拿大国家银行高中实习当柜员——讨厌会计课却去数钱，暑假和周末打工供学费，dot-com 寒冬续两年——「做一件 scary 的事」与「若不恐惧你会做什么？」是她喜爱的两句。

**AI 鸿沟**：精通 AI 者与抗拒者差距拉大。Fiona 在加拿大长大（香港出生、祖母带大），小企业.yarn 店、编织圈给她社区感；她用 **Cowork** 帮小企业主（双餐厅朋友用文件夹找菜单、做定价分析），并推动 **Claude for Small Business** 打包插件。

传播建议：从**对你改变生活的用例**开场——AI 是工具；「知识就是力量」。主动 awkward 地跟朋友演示 **Cowork** 能做什么，缩小鸿沟。

### 对谈实录

**Fiona Fung**：「**成长心态**极其重要——微软到 Meta 第一年我才真正理解。过去成功的方式可能不再服务你；让人害怕，因为你在要求改变成功公式。」

**Fiona Fung**：「挫败里我常看到**恐惧**。我的建议：向内探——『我能做什么？什么在我控制内？』别觉得一切发生在身上；找一件可控的行动。」

**Lenny Rachitsky**：「你恐惧的洞穴里藏着你要找的宝藏（the cave you fear contains the treasure you seek）。」

**Fiona Fung**：「我爱这句。我要偷你的 quote。[笑] 职业举动里， scary 的事往往是好选择——别跳崖。」

**Fiona Fung**：「若听众身边有人还没接触 AI，从**真正改变你生活的用例**聊起。AI 是工具；知识就是力量。帮小企业、帮家人——别让鸿沟越来越大。」

## 缩小 AI 鸿沟与 Anthropic 的潜藏需求哲学

### 核心洞察

> **潜藏需求（latent demand）**：用户用脚投票的意外用法，往往比战略会议室里想的更大机会。

### 深度解析

Lenny 归纳 Anthropic 屡次抢先：**编码**（**Claude Code**）、**知识工作**（**Cowork**）、**模型个性**对体验与智能的重要性。Fiona 未在其他 lab 工作过，但分享内部方法：

- **Coding** 天然优势：Anthropic 工程师自己是第一批客户，反馈极快。
- **潜藏需求**：**Cowork** 来自「非程序员也在用 **Claude Code**」——能否把体验做得更好？
- 跨产品经验：Marketplace、VR 等都在教同一课——用户会以你没预料的方式使用产品；迭代、贴近反馈、把「绕弯路」变成更顺滑的官方路径。

小企业朋友问各种插件，团队注意到可打包——**Claude for Small Business** 在 **Cowork** 里一键切换。Fiona 强调这不是她个人功劳，是观察潜藏需求的其他团队。

### 对谈实录

**Lenny Rachitsky**：「**潜藏需求**这词在本播客出现很多次—— essentially 盯紧你没预期或正在涌现的行为，然后大力投入？」

**Fiona Fung**：「对。当人们为了让某事成立而『跳 hoops』，你能不能把它做成更顺滑、更好的体验？」

**Fiona Fung**：「**Cowork** 来自很多人不是程序员却在用 **Claude Code**。小企业朋友问插件——现在我们有了 **Claude for Small Business**，在 **Cowork** 里一个小开关全打包。」

## 异步 routine 与工程下一前沿

### 核心洞察

> 下一前沿是**异步**：**routine** 把「写提示词开 Agent」再抽象一层——定时任务生成提示词、启动 Agent、醒来审 PR。

### 深度解析

Fiona 认为工程运作正转向**异步**——「Agent 舰队」与 **routine** 相关。过去同步写提示词；后来异步启动多个提示词；现在 **routine** 可定时运行：「每天早上看这些反馈频道，归纳主题，若有可抛光修复就启动 Agent」——醒来 PR 待审，而非一堆原始信息自己再想「然后呢」。

抽象层级不断上移。作为管理者，可把日常检查（项目进展、谁卡住、抛光修复）写成 **routine**；验证足够好时甚至「go for it」——给 Agent 更多自主权自动合并。

**上下文切换**是尚未破解的难题：20 个 Agent 并行意味着 endless check-in；Fiona 自己 kick off 很多异步任务后，反而要重新**屏蔽专注时间**来「追上」所有异步产出——以前屏蔽时间是为了写代码专注，现在是为了处理 Agent 产出。她坦言还没解决。

### 对谈实录

**Lenny Rachitsky**：「**routine** 是什么？异步是什么意思？」

**Fiona Fung**：「我以前起床咖啡看 Slack。现在设 **routine** 每天定时跑：看反馈频道、归纳主题，看到 bug 就 spawn 修复——我醒来有 PR 可 review，而不是还在想『拿这些信息怎么办』。」

**Fiona Fung**：「抽象层级一直上移。我们会更多走向这种异步工作方式。」

**Lenny Rachitsky**：「验证够好时你甚至说 go for it——给一点自由让它多做。」

**Fiona Fung**：「对。[笑]」

**Fiona Fung**：「**routine** 和异步带来**上下文切换**负载——我还没破解。我现在反而要重新 block focus time 来追上 kick off 的异步工作。」

## 能动性、问责与 ROI：别用动作冒充进展

### 核心洞察

> **Claude Code** 文化：**高能动性（high agency）** 配 **高问责（high accountability）**；测产出要看结果，别用 token 或行数替代进展。

### 深度解析

Tyler Cowen 等讨论的**主动性 / 能动性**，在 Fiona 团队是显式价值观：「有个问题——人人有想法怎么解」；放手开干的同时要问：**问责是什么？假设是什么？**

**ROI 与工程师生产力**是永恒难题。Boris 在 Meta 曾负责工程师生产力度量。Fiona 的经历：行数 → 显著行数 → 时间到 PR 落地——每个指标都会被博弈（搬库、框架升级少行数但产出不变）。她的原则：

- 别**为动作而动作**——token 刷量像当年的行数刷量。
- 先问：我们要解决什么问题？怎么衡量？聚焦结果而非工具使用量。
- 对领导者：做**倾听之旅**，尤其听资深工程师——他们能帮你规模化最佳实践；指标仪表盘有用，但对话常比仪表盘更能激发想法。

这与她用 **Claude Code** 看 PR 再对话的流程一致：不是盯数字，而是谈影响力、事故主题。

Marketplace 早期案例：盯「卖家数量」会误判——某区域卖家少但**头部卖家**强，买家仍能找到想要的东西；指标从卖家数改为能反映头部卖家的口径。环境变得快，指标本身也可能过时。

### 对谈实录

**Fiona Fung**：「我们非常重视能动性。有问题时团队人人有解法想法——高能动性；我们说**高能动性也是高问责**：有自由放手干，也要说清楚问责和你要验证的**假设**。」

**Lenny Rachitsky**：「有能动性——大家都在做——但你到底**做出了什么**？[笑]」

**Fiona Fung**：「别**为动作牺牲进展**（don't forsake motion for progress）。测工具使用量不等于测重要结果。先放大：我们要解决什么问题？怎么衡量？别像 token maxing——像当年的行数。」

**Fiona Fung**：「Marketplace 早期盯卖家数——一区域卖家少但头部卖家强，买家仍找到货。我们改了指标。指标服务结果；环境变快，指标也要调。」

## 质量框架：坏体验与难过体验、评测与脏话仪表盘

### 核心洞察

> **坏（bad）vs 难过（sad）** 体验分级 + 规格进仓库 + 主动质量监控，比人肉 review 更能跟上 8 倍速度。

### 深度解析

在速度与质量平衡上，Fiona 强调**主动质量（proactive quality）**：

- 关键体验设指标、看趋势。
- **坏 vs 难过**框架：**bad** = 不可恢复的恶劣错误；**sad** = 可恢复但恼人的痛点——难过堆叠可能变坏。各团队高能动性定义自己表面的 bad/sad（CLI 崩溃率 vs 界面闪烁等），便于拉高视角看整体体验主题，而非一堆难以解读的裸性能数。

投资方向：**更早发现质量问题的检测**、测试、监控；**评测（evals）** 难，因为不只有准确性还有体验愉悦度。

趣味指标：约 2024 年 9 月，团队工程师提议追踪用户反馈里的**脏话**——「脏话仪表盘」反映挫败感，与 eval 互补。

Code review 自动化：把「好是什么样」的声明检入 repo；内容设计 recently 更新带 scale 的标准——spec 与代码频繁同步，**Claude Code Review** 对照检查。

### 对谈实录

**Fiona Fung**：「我提出**坏体验 vs 难过体验**：**坏**是非常糟、不可恢复的错误；**难过**是痛点但可恢复——难过堆起来可能变坏。各团队定义自己的坏/难过，比一堆仪表盘数字更容易看出体验主题。」

**Fiona Fung**：「CLI 例子：崩溃是 bad；闪烁可能是 sad。我们给团队能动性定义什么算 bad、什么算 sad、目标是什么。」

**Lenny Rachitsky**：「你们有个仪表盘追踪脏话——用户骂得多凶。[笑]」

**Fiona Fung**：「去年九月刚加入时的有趣对话——追踪脏话是个好主意。评测也难——回到体验是否愉悦、少挫败。脏话仪表盘挺好玩。」

## 管理者即建设者：犬食、结对午餐与孤独感

### 核心洞察

> 经理先当 IC、持续犬食；Agent 时代结对「平行玩耍」午餐与黑客松，对抗孤独与流程优化成瘾。

### 深度解析

Fiona 的独特管理实践：

**经理先 IC**：新经理招聘岗位改为先以 IC 身份深入代码库/产品，再承担支持他人的重责——建立默契，避免一上来就「经理工具箱」。她本人 PR 可能很小，但每天用 **Cowork** 与 **Claude Code** 保持触感；从 Meta **500** 人组织到 Anthropic 曾短期 IC——Meta 面试是经理岗但第一季也做 IC，学 Meta 工程师怎么交付。

**Claude 作为入职伙伴**：第一周本想请工程师喝咖啡，转而问 **Claude** 代码库问题、自动化测试、请它帮设计手动测试用例——重获「我能提交 PR」的信心。许多多年经理的朋友说「多亏了 **Claude** 又在写代码了」。

**技能萎缩讨论**：团队内有过争论——**Boris** 早期手写代码，新人是否还需理解架构？Fiona 立场：仍要「双击」依赖层——信任但验证；依赖变了你要能感知。也许有一天不重要，但以当前速度仍重要。

**孤独感**：全员与 Agent 协作久了可能孤独——**Claude Code** 团队近期办 **pairwise programming lunch**（结对午餐）：各人带自己的 **Claude Code** / **Cowork** 流程，旁观学习——工具链变得太快，每人用法不同。另重视 **hackathon**、共处 maker time。

失去的「心流」：工程师怀念戴耳机啃硬问题、编译通过的快感；现在喜悦更多来自产品结果。Fiona 与团队承认：最难的部分曾是享受，现在部分人在迁移情感寄托。

### 对谈实录

**Lenny Rachitsky**：「经理必须先当 IC、持续兼职 IC——球员兼教练。为什么在今天格外重要？」

**Fiona Fung**：「倾听之旅里听到：感谢能动性，但优先级怎么定？也有反馈说 review 层太多。我意识到：领导者若先当 IC、深入代码/产品，再承担支持他人的重责，会和团队建立极好默契。我的 PR 不大，但每天用产品保持触感——指标再好，不每天呼吸产品也会失去手感。」

**Fiona Fung**：「第一周我本要去请工程师喝咖啡，转而问 **Claude**——极好的入职伙伴。它帮我设计手动测试，我又有信心提交 PR 了。很多经理朋友说：『多亏了 **Claude** 我又在写代码了。』」

**Fiona Fung**：「**Claude Code** 团队后来觉得和 Agent 工作太多会**孤独**——我们办了 **pairwise programming lunch**。每人用 **Claude Code** / **Cowork** 的方式都不同，看别人工作总能学到。」

**Lenny Rachitsky**：「像小孩的 parallel play——坐一起各做各的，但看怎么 build 很有价值。」

## 角色融合与经理新基线

### 核心洞察

> 编码相邻角色剧变：PM 不再被工程带宽卡住；工程师更产品化，非工程师更会写代码——验证成共同难题。

### 深度解析

**第二变革角色**（除工程外）：**PM**——有想法不必等工程师；**Claude Code** 团队 PM 也会撸袖子交付功能。所有**编码相邻角色**都在变；下一机会在**设计、数据科学**等工作流的更多自动化。

数据科学朋友的新常态：业务方用 AI 做「不怎么样的分析」甩给 DS 说「帮我看看对不对」——一半不对；工作变成大量审 AI 分析。

**工程经理新基线**（相对几年前）：

- 绝大多数 commit **Claude** 辅助。
- Slack 反馈频道 + 指标 dashboard 是日常。
- 工程师持续练**产品嗅觉**，成为强产品思维工程师。
- 跨职能阻塞减少——模型补足工程师不具备的能力（设计、数据等）。

双向模糊：工程师更 PM；人人更工程师。角色≈「你花时间最多的那件事的百分比」。

**犬食执念**：同事与 Lenny 访谈 prep 都提到 Fiona 对犬食的最大化——VS 起源；离开 Marketplace 后仍偶尔自己卖旧 MacBook，遭遇新诈骗套路；VR 团队时她总能复现奇怪的 floor height 问题。轶事 vs 数据：她常靠个人体验最有效帮助团队；智利 Marketplace 之旅发现 LTE 慢导致信息流加载差——数据说不行时实地轶事破局。Jeff Bezos 式「有数据和轶事时信轶事」在此印证。

### 对谈实录

**Lenny Rachitsky**：「Anthropic 里工程变革最大；第二是谁？」

**Fiona Fung**：「所有**编码相邻角色**。PM 不再被工程带宽卡住——我们 PM 也会交付功能。更多学科合代码时，**验证**更重要。下一步机会：设计、数据科学等工作流的自动化。」

**Fiona Fung**：「现在大多数 commit 是 **Claude** 辅助。工程师要练**产品嗅觉**。跨职能阻塞变少——模型补足你没有的能力。工程师更产品化，非工程师更会写代码——都在模糊。」

**Fiona Fung**：「若你很难犬食产品，就去见客户。每次客户拜访都学到新东西——Marketplace 进智利，LTE 比美国慢，feed 几乎加载不了——巨大增长障碍。」

## 未解难题、新人培养与失眠之夜

### 核心洞察

> 演讲末三个开放题仍热：**移动双端编制**、**自动化 review 边界**、**角色模糊下的生产率公平**；新难题是**上下文切换**；失眠之首是**文化**而非产品。

### 深度解析

Fiona 在 **Code with Claude** 演讲结尾的三个未解题（至今仍活跃）：

1. **还要独立的 iOS / Android 组织吗？** 仍需深度专家，但规模可能缩小——人在 flex 跨端；平衡「够深的专长」与编制。
2. **自动化 review 推多远？** 内容设计检查等——端到端体验验证仍难；eval 不只有准确性还有体验。
3. **角色模糊下如何确保人人同等高效？** 仍在探索。

新增：**上下文切换负载**随 routine 与异步工作上升。

**工程师未来与招聘**：表面矛盾——AI 似乎减少工程师需求，但 Anthropic、OpenAI 等疯狂招人。Fiona 的开放焦虑：**下一代如何培养？** 她与 Lenny 的路径（多年打字写代码）与新毕业生截然不同——可能需要更像 **fellowship / apprenticeship** 的长期学徒，而非 3 个月实习小项目。仍要「双击」底层，但不一定靠多年手写代码；也许模型够好后某些深度不再必要，但双击依赖层仍可能带来产品改进机会。

 punch cards 时代的前经理如今用 **Claude Code** 狂 build——抽象层一再上移；重要性与培养方式都会变。

**失眠之夜**：不是产品或工程难题（那些有仪表盘、假设）——是**团队文化**。文化活着、会变，体现在如何相待。扩张中维持：**多元视角**、公开诚实辩论、「**一支队伍**」心态——冲线时回头看有没有人掉队，一起完成。她的噩梦是经理说「一切都好」而实则着火（医生猫喝咖啡房间着火梗图）——故强调经理坦诚「什么不好」才能一起解决。

Lenny 分享 Airbnb 文化：创始人 obsession + Sheryl Sandberg 火边聊——「快速增长下的文化问题是你想有的问题」好过不增长。

### 对谈实录

**Lenny Rachitsky**：「演讲里三个开放题：还要分 iOS/Android 组织？自动化 review 推多远？角色模糊下怎么保证人人 productive？」

**Fiona Fung**：「iOS/Android：仍要深度专家，但可能不需要那么大 mobile org——人在 flex。自动化 review：内容设计 scale 等——**验证**仍是巨大机会。角色模糊：更多机会，仍在想。」

**Fiona Fung**：「加上 **routine** 异步后，**上下文切换**负载在涨——我还没破解。」

**Lenny Rachitsky**：「AI 会让工程师变少，但你们在疯狂招人——工程师未来怎么看？」

**Fiona Fung**：「大开放问题：**下一代怎么成长？** 你我路径与现在毕业生太不同——也许要 fellowship/apprenticeship，而不只是三个月实习。仍要双击依赖层——也许有一天不重要，但以当前速度仍重要。」

**Lenny Rachitsky**：「什么让你失眠？」

**Fiona Fung**：「**团队文化**。文化活着、会变——多元视角、公开辩论、**一支队伍**心态：冲线时回头看队友。产品难题有仪表盘；文化是人性——扩张时文化漂移、有没有一起对话纠正，这让我失眠。噩梦是经理说一切都好——像医生猫喝咖啡房间着火。」

## JIT 规划与杀流程的文化许可

### 核心洞察

> **即时规划（JIT）**：月度轻量优先级表 + 周检；显式许可**杀掉**不再服务的流程——六个月 roadmap 也可能三个月就过时。

### 深度解析

收尾建议：**杀流程（kill processes）**——挑一件你厌恶、嘈杂、昂贵或手动的流程，问它是否仍服务目的。Fiona 加入 **Claude Code** 时做「超轻量」六个月 roadmap—— exercise 对齐有用，但三个月后几乎没再引用，因变化太快；转向 **JIT planning**：

- 约 **一个月**轻量规划（非六个月）；大项目可跨月，但对齐用简单电子表格，甚至无长文档。
- 试验：月度优先级 + **每周**快速确认「本月优先级仍对吗？」
- 仍在想如何更多自动化表格更新，避免成为负担。

每六个月全团队仍会聚在一起启动**大方向主题**，但具体优先级随环境快速调整——甚至想用 **Claude** 自动化月度电子表格。

**Claude Code** 团队文化另一条：**显式许可终止不再服务的流程**。

闪电轮补充：Fiona 推荐 Margaret Atwood、村上春树；每年读《小王子》；电影常备 *Amélie*、*Spirited Away*、*Nausicaä of the Valley of the Wind*（童年领导力启蒙）；产品 Sweet Sisters 有机洗护；工作 motto「keep it simple」；人生 motto「in a world where you can be anything, be kind」；开会背景常有编织声（**Boris** 说重要会议能听见 Fiona 针织 click-clack）——针织与编程都是 0/1，栈与队列；梦想开祖母名字的纱线店，**Cowork** 管报销。

她请听众：给 **Claude Code** / **Cowork** 反馈（什么好什么不好）、分享潜藏需求用例、帮身边小企业或朋友上手 AI。

### 对谈实录

**Fiona Fung**：「我们文化里很重要：**显式许可杀掉不再服务的流程**。挑一件你讨厌或很手动的流程——还服务目的吗？我加入时做六个月 roadmap，以为超轻量——三个月后几乎没再看，变化太快。」

**Lenny Rachitsky**：「现在怎么做规划？」

**Fiona Fung**：「**JIT planning**——六个月太长，现在约**一个月**轻量对齐，常常只是电子表格，甚至没长文档。我们在试：月度优先级 + 每周快速核对是否仍是本月优先级。六个月仍会全团队启动大方向主题，但环境变得太快。」

**Fiona Fung**：「昨天还在聊：怎么自动化表格更新？永远问：能自动化得更好吗？」

**Lenny Rachitsky**：「**Boris** 说重要会议常能听到你针织的 click-clack。」

**Fiona Fung**：「[笑] 刚织了这件上衣。祖母八岁教我针织——每次织都想她。针织和编程：一针正一针反，0 和 1。梦想开祖母名字的纱线店，造社区——**Cowork** 帮忙开票报销。」
