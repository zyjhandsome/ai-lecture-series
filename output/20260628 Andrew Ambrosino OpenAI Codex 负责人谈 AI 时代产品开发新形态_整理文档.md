## 核心导读

> **全文论点**：**Andrew Ambrosino** 认为 AI 已**彻底倒置**产品开发流程——**实现不再昂贵，品味、策展与媒介选择才是瓶颈**；Codex 团队用**区域联防**式产品协作、**角色重叠而非角色取消**、以及「**模型就绪后再发布**」的耐心，把 Codex 从工程师工具扩成跨职能的**工作中枢（home base）**；而人类仍须在**新颖范式、设计抽象层与「该用什么格式表达」**上掌舵。

**流程倒置与 90 份原型。** 在 OpenAI，人人高能动性、无限算力/token 下，同一需求常出现 **90 个**互不协调的实现探索——这与旧时代「先写文档、原型便宜、实现昂贵」完全相反。文档并未死亡，原型也并非万能；关键是**为要传达的观点选对媒介**，并避免 polished 原型被误读为「已可上线」。**首笔标记（primal mark）** 效应说明：过早的原型会锚定讨论，而非打开更大 solution space。

**品味、设计与角色。** 「品味」不只是审美——还包括系统契合、主题归属、信息呈现，以及动画语义是否匹配交互含义。**Andrew Ambrosino** 解释 AI 设计滞后于编码：设计更难自动评分、实验室早期更押注能加速研究的编码能力、设计还依赖文化新颖性与代码—视觉抽象层。Codex 团队角色边界更模糊（设计写代码、产品写代码），但**彻底取消产品等专业角色是危险误区**——每个学科仍有可习得的最佳实践。

**时机、野心与 Codex 愿景。** 他极确信：若 **2025 年 11 月** 就发布 **2026 年 2 月** 形态的 Codex 应用，**市场会失败**——差别主要在模型能力；Operator、Atlas、应用内浏览器等是同一能力线的多次「形状试验」。Codex 未来不是把一切 SaaS 塞进一个矩形，而是**统筹 ChatGPT、Codex 与既有工具**的 home base：必要时打开 **Excel** 插件而非内置表格，必要时为 **Premiere Pro** 写扩展而非自研视频编辑器。失败角里，**Andrew Ambrosino** 坦言创业多年「 essentially 拆卖公司」后才迎来技能、热情与市场时点的对齐——内部 **2000 条**「这很蠢」的 Slack 线程，是外部产品质量的预演。

## 1. 产品开发流程倒置：实现变便宜，品味与策展变贵

### 核心洞察

> AI 让「从零搭功能」变容易，产品流程**倒过来**：贵的是品味与从 90 份探索里**策展**出该做什么。

### 深度解析

**Lenny Rachitsky** 开场锚定主题：**Andrew Ambrosino** 希望听众带走的核心，是 **AI 如何改变产品工作的形态**。在 OpenAI 这类最前沿团队，变化不是「大家做完全不同的岗位」，而是**流程方向反转**。

**Andrew Ambrosino** 的观察：与前沿模型对话，**从零开始几乎可以搭出任何功能**——这不一定是软件最难的部分，但极其酷，并催生出「人人都在造东西」的环境。给全员无限 token 后，OpenAI 内部人人**高能动性（agentic）**、点子多，**每个人都在做一切**。

对比传统产品流程（即便过了瀑布）：研究 → 构思 → 或许有原型 → 但核心假设仍是**实现昂贵**，所以要靠文档、研究、原型在前端**去风险**——因为原型和设计比写代码便宜。如今这一假设**彻底失效**。某个急需的功能，他确信此刻有 **90 个**互不协调的团队在各自实现、各自试错。

短结论：**不是角色消失或技能蒸发**，而是**整个流程倒过来**——实现不再昂贵，**说得更直白些，贵的是品味**；更准确说是**策展**：从 90 次尝试里判断哪些好、该合并进哪条功能线、框架怎么定、切换控件该几个档位。**Andrew Ambrosino** 在开场片段还提到：OpenAI **近 100%** 员工（非仅工程师）每周使用 Codex；他的公开目标是让 Codex 成为打开后**毫无犹豫**的自然选择，像人们开浏览器标签一样。

**Lenny Rachitsky** 口播数据（嘉宾未否定）：自 **2026 年 1 月** Codex 使用量 **6 倍**增长，周活超 **500 万**；内部近 **100%** 员工每周使用。

### 对谈实录

**Lenny Rachitsky**：「准备这期时我问你：希望听众带走什么？你说最重要的是 AI 如何改变产品工作的形态。在 OpenAI 这种最前沿团队，产品团队现在长什么样，和两三年前比呢？」

**Andrew Ambrosino**：「当下做这类产品的领导者，最难的是流程倒置。很多人谈过：现在谁都能造任何东西。我相信，跟这些模型——我们的或别人的——从零开始，你基本能搭出任何想要的功能。这不一定是软件最难的部分，但真的很酷，也造成人人都在造的环境。」

「给无限 token 后，OpenAI 人人高能动、点子多，人人造一切。回看我们跑了很多年的流程，方向相反：实现昂贵，所以要先用文档、研究、原型去风险——原型和设计更便宜。这完全变了。我敢说有 90 个互不协调的团队，在实现、在试同一个我们急需的功能。」

「所以短答是：倒过来了。不是角色根本不同，也不是技能消失——是实现不再昂贵，说直白些，是品味；是策展：90 次尝试里什么好？该合并进哪里？怎么定框架？切换控件几个档位？全在这些。」

## 2. 文档还是原型：媒介选择与首笔标记陷阱

### 核心洞察

> 「PRD 已死、原型当道」是过度简化；**实现变便宜后，选对媒介**比站队更重要。

### 深度解析

**Lenny Rachitsky** 归纳：过去写文档讲要做什么；现在有人有想法就直接做**小原型**，全公司可能冒出 **90 个**可看的方向供挑选——**Andrew Ambrosino** 确认这类情况**很多**，但不认同「PRD 已死、原型至上」。

实现成本在各媒介上都降低后，**非工程师**尤其容易跳过文档直接「我给你看意思」；**工程师**则反过来可能写**大量不值得读**的文档——问题不在写文档的人，而在**实现充裕时，必须为论点选对格式**：

- 产品方向仍模糊、需要**清晰度** → 可能仍是**文档**
- 需要让人上手、**压测交互模式** → **原型**

旧时代媒介自带**进程信号**：看到像已上线产品的东西，默认已去风险、设计看过、业务目标成立。现在**媒介与成熟度脱钩**——因为以前要做到「看起来像成品」的原型很难要到资源，现在**窗口被掀翻**。打磨精细的探索稿可能被误当成「视觉已可上线」，但研究、用户需求、业务模型并未就绪——**不要过度锚定**在本意是探索、却看起来像成品的东西上。

**Lenny Rachitsky** 引入 **primal mark（首笔标记）**：画家第一笔会锚定后续所有反应；有时**原型是错的第一个标记**，让人只回应原型而非更大想法。**Andrew Ambrosino** 同意：文档在特定场景仍有用，品味也体现在**用什么媒介、如何呈现、如何达成目标**。

### 对谈实录

**Lenny Rachitsky**：「所以过去写文档，现在直接做原型，全公司可能 90 个方向可以挑——是这个意思吗？」

**Andrew Ambrosino**：「这类情况很多。你也听过很多产品负责人说 PRD 死了、原型来了——我完全不认同。有趣的是：实现各媒介上都便宜后，非工程师特别想跳过 PRD，直接『看我意思』；工程师又特别想写一堆不值得读的文档——不是羞辱写文档的人，而是实现充裕时，必须为论点选对格式。方向模糊要清晰度 → 可能是文档；要让人试、压测交互 → 原型。有意思的是：选对媒介现在超级重要。」

**Lenny Rachitsky**：「有位嘉宾提过首笔标记：设计师第一笔会锚定整个作品；有时原型是错的第一个标记，你就只回应它，而不是更大想法。所以不是『别写了，全做原型』？」

**Andrew Ambrosino**：「对。旧世界媒介自带进程信号——像已上线产品就意味着后期阶段、去风险了。现在脱钩了，因为以前要到那一步很难要到资源。我们要说：可以有原型、可以有文档——但清楚这是在干什么。别过度锚定在探索稿上：视觉像成品，但模型、用户、业务都不对。又是品味：做什么、怎么呈现、用什么媒介——这在每个领域都是最重要的事。」

## 3. 品味究竟是什么：审美、系统思维与呈现判断

### 核心洞察

> 品味不是「我审美好」——是**系统契合、方向主题、呈现方式**与交互语义是否一致的综合判断。

### 深度解析

**Andrew Ambrosino** 接 **Linear** 产品负责人 **Nan Yu** 的推文：人们过度强调品味的**审美**面——**Paul Graham** 品味极好却穿 cargo shorts。品味**以上皆有**：

- **审美面**：动画是否过 snappy、不符合语义
- **系统思维**：如何 fit 进整体系统
- **方向与主题**：我们往哪走、这属于哪条主题线
- **呈现方式**：怎么把信息交给别人

他自己可能**过度聚焦**交互动画语义，但更大的品味问题是：**既然什么都能造，这东西应该是什么、怎么到那**。

在 **AI 更强时人脑还在哪有价值**的脉络里，**品味**是核心答案之一——与 **Lenny Rachitsky** 观察「AI 设计输出总像 Cloud/Codex 设计、很少 nailed it」相连。

### 对谈实录

**Lenny Rachitsky**：「品味这个词被说烂了。它到底是『决定投哪个方向』，还是『有了东西判断能不能发布』？具体长什么样？大家爱说『我品味好』。」

**Andrew Ambrosino**：「昨天 Linear 产品负责人 Nan Yu 有推文——人们过度强调品味的审美部分；Paul Graham 品味极好，却穿工装短裤。要拆开看。你问的都算：有审美面——动画太急促、不符合语义，这很重要，我可能关注过多；也有系统思维、主题归属、怎么呈现。更大问题是：既然什么都能造，这应该是什么、怎么到那——我觉得那才是真正的品味问题。」

## 4. AI 为何仍不擅长设计：评分、投资与抽象层

### 核心洞察

> 设计比代码难自动评分；实验室早期押编码飞轮；设计还要**文化新颖性**与**代码—视觉抽象层**——后者仍超出当前模型。

### 深度解析

**Andrew Ambrosino**（非研究岗，自述可能被研究同事骂）给出**实用原因**与**更难问题**：

**实用面**

1. **设计更难 grade**：代码有 compile、是否跑对等清晰信号；设计好坏的反馈环依赖人类**品味**，训练「好设计/坏设计」循环更 tedious
2. **实验室投资优先级**：早期编码模型能写**正确代码**会直接加速 AI 研究；设计重要，但不在同一**飞轮**里——这些实用障碍会消失，模型会变好

**更 murky 的问题**

1. **文化性**：去年一度「每个新网站都是 **Linear** 网站复刻」——模型若每次输出 Linear 站，不是挑战；设计比软件工程更要**随机性与新颖性**；工程反而希望 over-index **已知模式**
2. **抽象层**：早期 Codex app 他大量写代码/督导代码时发现——角落里的控件与下方另一控件应在 codebase **共享 x/y/z 抽象**；这不是「模型要更会做视觉设计」，而是**更深的设计抽象**：明天公司 rebrand，浅层是改 **263 个组件**；深层是「两个看起来不同的 list 其实 convey 同一交互模式」——**当前技术仍够不到**

**Lenny Rachitsky** 接话：Codex 桌面形态（非 terminal、非 IDE、chat + 看 code）本身像**新范式**，像人类在**创造力**上仍领先——**Andrew Ambrosino** 同意「给人脑鼓掌，暂时」。

### 对谈实录

**Lenny Rachitsky**：「AI 设计输出 rarely nailed it，总像 **Codex design**。为什么顶尖模型设计还是不行？会到『 holy moly 完成了』那天吗？」

**Andrew Ambrosino**：「有实用原因，也有更难的问题——我不在研究岗，说完可能被 yell。」

「设计比软件**更难 grade**；训练『好/坏设计』循环比『代码能不能 compile』更 tedious，因为**人类品味**是反馈机制的一部分。实验室 historically 投资能让**AI 研究加速**的事——早期编码模型写**正确代码** clearly 加速研究；设计重要，但不在同一 flywheel。这些实用原因会消失，模型会变好。」

「更 murky 的：**设计有文化性**——去年每个新网站都像 **Linear** 复刻；模型若每次输出 Linear 站，不是挑战。设计比工程更需要**新颖性**；工程 almost 想 over-index 已知模式。」

「还有**抽象层**：角落控件和下面某物应在 codebase **共享抽象**——不是视觉设计师问题，是更深的设计：rebrand 时浅层改 263 组件，深层是两个不同 list 共享交互语义。那**仍有点 out of reach**。」

**Lenny Rachitsky**：「Codex 这种全新范式—— chat thing that codes——像人类在**创造力**上还有价值。」

**Andrew Ambrosino**：「完全同意。给人脑鼓掌——至少目前如此。」

## 5. 设计流程之死：与 Jenny Wen 的分歧、共识与 Codex 实践

### 核心洞察

> 设计「正统流程」本就令人不适；AI 时代**流程阶段标注**（我们在哪一阶段）比工具清单更重要。

### 深度解析

**Lenny Rachitsky** 提到 **Jenny Wen**（**Claude Code** 与设计协作产品负责人）论点：设计流程已死，没时间设计，只能边做边引导方向。**Andrew Ambrosino** 与她在很多点**同意**，且他在 AI 之前就不喜欢传统设计正统流程。

他引用 UX 圈「案例研究工厂」文章（中期冲浪文化时代）：设计师被教流程高于结果——走过发散/收敛、用户研究框架，就**保证质量**；东西不好、没人用，只要走过流程就算「对」。框架本身偏学院派，但 AI 实现速度暴露其弱点：流程假设**只能承担构建一次**，所以要穷尽探索问题/方案空间；**Figma**、**Origami** 等把交互原型提前，形成「高管：能不能先做个原型」的梗——这**真的**进了设计正统流程。

现在可以把**全部实现**拉进流程——精细原型像可发布版本，全公司看到就想「能发布吗？」，但恰当回应应是：我们还在**早期设计探索**，这是 **90 人并行探索**，不是可上线候选。

「设计流程死了」**既真又假**：若绑死在**具体工具与日课**，确实过时；若把**流程阶段标注**（我们现在处于哪阶段）扔掉，则**比以往更重要**。

**Codex 团队设计实践**

- 角色坍缩比其他部门更多（技术产品，设计师会 engineer 语言，PM 写代码；**Alexander Embiricos** 有 CS 硕士，Andrew 没有）

## 6. 角色坍缩与边界：重叠增长，但不能取消专业

### 核心洞察

> 欢迎「这不是你的 lane」消失，但**取消角色概念**会丢掉整门学科的最佳实践。

### 深度解析

**Lenny Rachitsky** 问长期是否人人都成 **member of technical staff**、职能是否还存在。**Andrew Ambrosino** 指出这传统来自研究型公司（**Xerox** 等），OpenAI 更常见，但**并非人人最终都无职能**。

他**害怕**的是公司极端跟风「取消产品岗、人人皆构建者」——会危险地消除「**专业领域有可习得的最佳实践**」的观念；产品这门学科积累的真实流程、试错会被抛弃，变成「哦我写了点代码」。

**平衡点**

- **欢迎消失**：「这不是你的赛道」式边界
- **必须保留**：不是人人深度/广度都能做一切——**管理者不会消失**；每个学科有 **技能成分**——工程师也常低估其他角色（「工程是技能，别的都是凭感觉」），但「会用 Excel」≠ 能进财务团队
- **新好处**：换角色、学最佳实践、不把效能绑死在**特定工具**上更容易——Andrew 长期觉得自己「不该做工程师」因为不爱汇编、背 TypeScript 语法；那种 **门槛式 gatekeeping** 正在瓦解
- 但公众**夸大**一切

### 对谈实录

**Lenny Rachitsky**：「OpenAI 叫 technical staff 成员——职能还会存在吗？还是人人皆构建者，全面坍缩？」

**Andrew Ambrosino**：「我有些害怕的。有些公司爱极端跟风。危险在于消除角色概念会消除『专业领域有可习得的最佳实践』。听过太多『取消产品岗』——我觉得糟糕透顶——人人皆构建者，然后整个产品学科被抛弃，因为『哦我写了代码』。」

「『这不是你的赛道』我欢迎那部分消失；但要平衡：不是人人能做一切——广度与深度；管理者不会消失。每个学科有技能——工程师也常犯：『工程是技能，别的都是凭感觉』——不，会用 Excel ≠ 财务团队。」

「也有好处：换角色更容易、学最佳实践更容易、效能更少绑在特定工具。我长期觉得不该做工程师，因为不爱汇编、背 TS 语法——那种门槛式 gatekeeping 在瓦解。但人们夸大所有这些。」

## 7. 区域联防：人人能造时的产品经理新打法

### 核心洞察

> 两个产品人贴太近常是坏信号；要像**区域联防**铺开覆盖，用品味从混沌中引导方向。

### 深度解析

**Andrew Ambrosino** 与 **Alexander Embiricos** 常用 **zone defense** 类比：

- 两个产品人**贴太近** → 往往**坏信号**
- 产品要做 **力导向式** 协调：混沌里想法满天飞、自上而下年度计划不适用，需要 **品味把关者** 从构思阶段引导到产品应是什么 → 要 **全公司覆盖**
- 拉开距离：谁擅长什么、创造空间，**全覆盖**
- 再填补缺口：招 **有产品思维的工程师**，避免一堆代码需要整队评审才 coherent——人人要有这些技能，但**深耕方向**要变

这与 **Lenny Rachitsky** 观察的线索一致：最有价值的是 **高能动性 + 高品味**，能把想法**一路护送到完成**、对做出卓越成果近乎痴迷。

招聘看：**学科驾驭力** + 在无限 token 世界里分辨信号/噪声，拒绝粗制滥造。

### 对谈实录

**Lenny Rachitsky**：「你提区域联防——产品人怎么运作？」

**Andrew Ambrosino**：「我和 Alexander 聊很多这个类比：两个产品人贴太近常是坏信号。要做力导向式协调：混沌里想法到处飞，自上而下年计划不行——需要品味把关者从构思引导到产品应是什么 → 要全公司覆盖。拉开：谁最擅长什么，创造空间，全覆盖，再填缺口——招有产品思维的工程师，别一堆代码要整队评审才 coherent。人人要有技能，但深耕方向要变。」

**Lenny Rachitsky**：「最有价值的是高能动性 + 高品味，把想法护送到完成——是你招人标准吗？」

**Andrew Ambrosino**：「核心标准。也关乎个人贡献者与管理：不是管理消失，也不是人人纯个人贡献者——人人某种程度上两者兼具。个人贡献者不是逐字打字，是管理智能体、管理正在发生的工作；管理者是不同粒度的同样事。我找学科驾驭力 + 品味：无限 token，不能只做粗制滥造——要分辨信号/噪声。」

## 8. 团队结构、规划节奏与个人贡献者与管理边界

共性：**能动性与品味**；很多前创始人或在大型公司做「创始人型事务」的人。OpenAI 允许团队很大，**大多是个人贡献者（IC）**——他认为这是好事。

**Andrew Ambrosino**：「你记得我怎么答？10 到几千——半玩笑但真实：模型研究、浏览器使用、人格、前端、用户……全是这个产品。但我们不是每天收几千人随意 PR。核心团队：两位数工程师，设计约一半，少数产品——产品层本身像区域联防。共性：能动性与品味；很多前创始人。大多是个人贡献者，团队可以很大——我觉得好。」

**Lenny Rachitsky**：「路线图难规划。你们怎么规划？多远？表格还是 MD？」

## 9. 模型节奏决定成败：十一月版 Codex 为何必败

### 核心洞察

> **同一产品形状**，晚几个月发布因模型不同而**市场结果完全不同**——November 版必败，February 版才行。

### 深度解析

**Andrew Ambrosino** 极有信心：**2026 年 2 月**发布的 Codex app，若 **2025 年 11 月**就 ready，**绝对会在市场失败**——**唯一差别是模型**（11 月 vs 2 月）。同一形态，市场结果完全不同。

这与播客常见线索一致：**先构建尚不奏效、等模型追上的东西**；以及**野心**——敢承担更大事，但有时**对市场过于超前**。

**案例链**

- 应用内浏览器：曾有可用版本；回溯 **Atlas** 里智能体、更早 **ChatGPT Operator**——酷想法，**未成**；但 Operator → Atlas → Codex → ChatGPT 可画一条**本质相同功能**线，**换不同智能度重新发布**改变结果
- 勿固执：「不奏效所以是坏功能」→ 可能是**尚未就绪**
- 原始 Codex 发布（Codex web）：给模型任务、它离线执行、回来交成品——**对当时而言过于 AGI 化**；模型写代码还行但**产品形态太早**。**Claude Code** 本地、更少挂钩云端、更智能体化、会问你问题——**当时模型能力点**上效果好得多
- 旧时代市场与用户教育会告诉产品形状；现在可能**同一形态发布六次**才奏效

**野心张力**：产品存在后，人易找小毛病、优化（应该）；也需要**自下而上探索**——Codex 曾冲击 ChatGPT，未来也会被后续努力冲击；一个团队难同时做颠覆与维护质量——要设计允许两者并存的流程。

### 对谈实录

**Lenny Rachitsky**：「Codex 故事太棒——11 月 ready 的 2 月版会在市场绝对失败，差别只是模型？」

**Andrew Ambrosino**：「非常有信心。2 月发布的，若 11 月就就绪，绝对市场失败——唯一差别：11 月到 2 月之间的模型。完全相同的形态，结果完全不同。」

「应用内浏览器：曾有可用版本；Atlas 里智能体、ChatGPT 里的 Operator——很酷，没成。但 Operator → Atlas → Codex → ChatGPT 本质上是同一功能，换不同智能度重新发布会改变结果。别固执：不奏效不等于坏功能，可能还没到时候。」

「最初 Codex：Codex web——给任务，离线做，回来交成品——对当时过于 AGI 化。写代码还行但产品形态太早。Claude Code 本地、更少云端、更智能体化、问你问题——当时模型能力点上效果好得多。我常想这教训。」

**Lenny Rachitsky**：「先构建尚不奏效、等模型——是你们的做法吗？」

**Andrew Ambrosino**：「很多是这样。但要清楚处于设计流程哪一阶段——『写了代码就该发』不对；那是可对照未来模型测试的工件。研究侧想最 ambitious；产品侧若过于 AGI 化会 hurt。」

## 10. 野心、Loops 与自主开发：尚未抵达的「自动增长」

### 核心洞察

> 「代码百分百 AI 写」已不是好问题；新问题是**监督 vs 无人监督**——离「听 Slack 自动改 app」还远。

### 深度解析

### 对谈实录

「很多探索：自主软件、Harness 工程、 overnight 垃圾回收清 codebase——模型通病：增加复杂度。Research 请更擅长删代码。完全 autopilot：human 侧教哪些功能该做/忽略/合并；codebase 侧也是问题。」

**Lenny Rachitsky**：「会到赢市场那种吗？」

## 11. Andrew 如何用 Codex 运转工作：简报、自动化与计算机操控

### 核心洞察

> 用法随**职责长法**变：从「用 Codex 写 Codex」到 Slack 简报、发布自动化与垃圾邮件过滤——**对话即可配置**。

**2026 年 5 月左右** 协调发布（应用内浏览器、计算机操控、artifact 创建）——「Codex for almost everyone」、**氛围编码（vibe coding）** 进入主流；他用 Notion 文档 + 自动化从 PR、Slack 拉更新填状态追踪——当时感觉在 bleeding edge，现在较 commonplace。

- 设 Google Cloud **Pub/Sub** 等：界面 annoying → 让 Codex **computer use** 代点——「没 connector？我直接 start clicking」

**设计边界**：in-app browser vs Chrome extension vs computer use——connector 何时用、何时浏览器、何时接管桌面，靠 feel out；Twitter 有 thread 讲三种用法。

**Personal workflows 主题**：人人不同，主题浮现后应成 **first-class experience**；**memory** 不应人人自建 Obsidian/Notion mind palace——应有 generic memory feature；但 job-specific process 可能 stay custom——持续 sift **什么该成 primitive**。

### 对谈实录

**Lenny Rachitsky**：「怎么 setup？听起来 amazing。」

**Andrew Ambrosino**：「还在探索阶段。现在我做自动化/定时任务——仍定义分类。聊天机器人形态的问题：我有时间做探索来 setup；你不开发这个，不应自己摸索全部。至少：不会就问 app——不够，但是至少。」

**Andrew Ambrosino**：「对。趋势是各产品都在加。Setup：app 里说设自动化看 Slack；没 Slack 连接器会问能否添加——点 yes。」

## 12. 浏览器、连接器与 Codex 作为工作中枢

### 核心洞察

> Codex 应是 **home base**——统筹工作、必要时打开 Excel/Premiere，而非把一切 SaaS 塞进一个矩形。

### 深度解析

**愿景脉络**

- 曾有 **Codex CLI**，后做桌面应用——不确定但信念强：非 IDE，**尺寸恰到好处的界面**，聊天 + 看代码，**不让手改代码**
- **2026 年 1–2 月** 内部试用：工程/研究**产品市场契合（PMF）清晰**；同时市场/传播/财务/法务也在用——尽管 UI **对非工程师极不友好**（展示代码、要批准跑 rg 等）
- 公司曾把 Codex 能力加到 **ChatGPT 桌面版**、**Atlas**——最恼人的问题：**没人愿意离开 Codex 应用**去「为其他角色做的应用」
- 教训：开发者工具 vs 通用知识工作**非二元**——做 Excel 的人不想看 git 信息，但可从行为推断工作类型，**从简单长到复杂**；可有**模式**让入口清晰

**Home base 定义**（非「超级应用」——有人说了 super app 他得整天听）

- 跟踪跨平台待办；有些事全在 app 做，有些**打开其他应用**
- **Excel 例子**：应用内有电子表格编辑器，但 OpenAI 财务建模可能不够 → 直连桌面 **Microsoft Excel** 插件，做完关 Excel
- 不是「画矩形把一切塞里面」——**开始工作、结束工作、自动化工作**，用所需工具

**浏览器形态迭代**

- **Electron** 限制：应用内浏览器曾仅用于开发测前端；换 **Owl stack**（支撑 **Atlas** 浏览器）→ 多标签、企业安全可登录网站
- 张力：**仅智能体用的浏览器** vs **让人当主浏览器**——权衡大；快捷键映射 **VS Code** vs **Chrome** vs **Linear** 等繁琐但必要
- 要服务从未构建过的基础用户到 **Peter**（**OpenClaw** 作者）式终端坚持者

**Dan Shipper** 预测：在 Codex 里跑 **Notion**/**Linear**/**Salesforce**——**Andrew Ambrosino** 说 Dan 每天 Slack 问他进展。两路径并行：

1. **Codex/ChatGPT 无缝对接已有专业工具**（连接器、计算机操控、扩展）——不必做更好视频编辑器，但要**能用手上的**
2. **Dan 式**：在 Codex 里打开 web 应用做额外操作——几乎相反，**同时大量推进两者**

**Premiere Pro 故事**（发布视频）：内部 DX 摄影师 **Brent** 用 Codex 剪片——Codex 不是视频编辑器，但知道他用 **Premiere Pro**，可编辑底层文件；不够则**自建 Premiere 扩展**让 Codex 对话改标记点——「相当疯狂」。模式：**专业工具做专业事；我们负责连接与交接**。

## 13. Premiere 扩展故事、失败角与职业自省

### 核心洞察

> 外部光鲜背后有 **10–15 年失败**与无数小失败；别绑死在旧流程，要绑在**你能独特交付的结果**上。

### 深度解析

**失败角**

- 长期创业公司创始人，公司** essentially 拆卖**；强监管行业苦战，持续失败
- 下一创业公司做 AI 工具，同样 locked-down 行业，一次次不成
- 或许**第一次**不觉得在失败——技能、热情、市场时点对齐
- Codex × ChatGPT 合并：无数小失败——Slack 扔形态，**2000 条**消息线程骂蠢；OpenAI 内部**毫不留情**，所以外部产品质量高
- 在此之前**失败了 10–15 年**——仍惊讶现在顺利

**闪电轮摘录**

- 书：育儿阶段，推荐 **The Gruffalo**；最爱童书 **The Big Orange Splot**（反 HOA、能动性——「去他的，全押上」涂房子，邻居逐个跟进）→ **Lenny**：能动性，你可以直接做

- 喜爱产品：每天都在重新发现自家产品；**Linear** 曾是 favorite 软件（规划「理论上」用 Linear）
- 人生格言：非格言型人，但同事说他常讲的话就是格言
- 产品/设计/工程哪个最难：**都很不同**；铁三角未来会有流动性，对高能动性的人 refreshing，但有些东西不应消失

**录后花絮**（制作人）：用 Codex 做播客剪辑——切成三段 break、理解停顿；产品非为剪辑设计，但是能写代码的空白聊天机器人 → **好奇心 + 明确目标** 即可试；**Andrew Ambrosino** 补充：别绑死具体流程，绑**你能独特交付的结果**；别绑 Figma 自动布局技能——AI 会更强；AI 成功需要高度**自我觉察**；OpenAI 的人 self-selecting 会自己找下一步——不是大多数人；反复学习常是 bummer——Andrew 讨厌重复，故 love 每天不同方式做工作（「为自己的工作找到产品市场契合」）
