从弦论到软件工厂：Matan Grinberg 谈守恒量思维、代币预算与模型无关战略

Matan Grinberg 用物理学的「守恒量与对称性」思维反驳「AI 将消灭一切工作」的叙事——世界问题只会越来越多，给工程师更高杠杆才能多解决软件本可解决却未解决之事。

Matan Grinberg on Noether's Theorem, Factory's $1.5B Vision, and Why Open Models Threaten Closed AI 2026-06-30 ~31 分钟阅读

核心导读

Matan Grinberg 用物理学的「守恒量与对称性」思维反驳「AI 将消灭一切工作」的叙事——世界问题只会越来越多，给工程师更高杠杆才能多解决软件本可解决却未解决之事；企业 AI 的下一战场不是选哪家闭源模型，而是代币（token）与人力的资源配置；Factory 押注模型无关的「软件工厂」——工程师不再写软件，而是调校组装线，让 Droids 在合适环节路由 Opus、开放权重模型或微调模型；闭源巨头的真正威胁是优质开放模型带来的定价压力，而非彼此内卷；离开弦论的身份危机与「沙漠岁月」塑造了敢于反驳、直面盲区的 CEO 气质，以及「共建（Build Together）」的跨职能文化。

守恒量与问题观。 Emmy Noether 定理：连续对称性对应守恒量——空无一物的宇宙里，位移与时间流逝不变，便导出能量与动量守恒。Matan Grinberg 将此映射为商业决策的基线：在剧变的 AI 编码赛道，先问什么不变——问题总量、人类解决问题的需求、软件可承载的比例。Mario 追问就业置换；Matan Grinberg 承认局部岗位会减，但全球未解问题远多于已解，且自由竞争会惩罚过早裁员、奖励继续押注工程师杠杆的垂直行业。

软件工厂与模型路由。 Factory 使命自始未变：为软件工程带来自主性。Droids 覆盖编码、测试、审查、合规等全生命周期；企业客户现更关心整条「软件工厂」——类比 Tesla 机械臂产线，工程师做系统设计：哪条臂用 Opus、哪条用开放模型、哪条为 COBOL 遗留代码微调。Factory Router、Droid Computer（远程沙箱）、Automated Security 等均由客户瓶颈驱动。Matan Grinberg 预言成功状态下，工程师对所用模型应像对烤面包机背后的电网一样无感。

生态、护栏与个人弧光。 xAI 拟收购 Cursor 对 Factory 是利好——市场需要不绑单一模型的独立软件工厂。Matan Grinberg 用实证反驳「模型+执行层一体必然更优」：Droids 用 Opus 可胜过 Claude Code 里的 Opus，因多模型暴露让执行层避免过拟合单一模型。Anthropic Fable 5 发布触发的数据留存与性能降级政策，强化开放模型与多供应商竞争的必要性。个人线：八年级几何「重修」触发报复式自学、Juan Maldacena 极隐晦的入门考验、Berkeley 助教经历引爆身份危机、IMDb Top 250 与经典文学补课、Sean McGuire 与 Enno Reyes 塑造的 Factory 文化与未来预测。

一、诺特定理与守恒量：剧变时代的第一性原理

核心洞察

对称性背后藏着守恒量——这是 Matan Grinberg 在 AI 乱局里仍能站稳的第一性原理基线。

深度解析

Mario 刻意不从当周 AI 新闻开场，而先问 Emmy Noether（转写误作 Noita）第一定律。Matan Grinberg 解释 Noether 定理：宇宙中每一种连续对称性，都对应一个守恒量。抽象地说：若某种变换不改变物理情境——例如在真空中换位置、或等待一小时——则相应有动量、能量守恒。数学上，不同「玩具宇宙」可构造不同守恒量，导出不同物理理论。

Matan Grinberg 高中三年级首次接触该定理；其魅力在于，连五岁孩子都能直觉理解「黑漆漆的虚空里过一小时，世界不变」，而背后却是能量守恒的源头。硅谷常谈「第一性原理」；在物理里，面对任意复杂问题，先问：什么是对称的？什么是守恒的？ 由此获得立足点，再分析对称性被打破之处——例如附近出现太阳，空间平移对称性破坏，但时间平移仍守恒。

Mario 将此法映射到 Factory 战略：Matan Grinberg 认同——在 AI 与代码生成这一经济最剧变区，训练自己识别「守恒项」至关重要。他反感「所有工作将消失」「永久底层阶级」等话术：既不准确，也伤害仍将在人类未来扮演关键角色者的心理。守恒思维可拆解此类断言。

对谈实录

1

Mario

「Emmy Noether 的第一定律——你显然是大粉丝。对我这意味着什么？」

2

Matan Grinberg

「Noether 定理说：每一种连续对称性，都有对应的守恒量。若事物在某方面对称——空无一物的宇宙里换位置、或等一小时——看起来都一样；背后却是能量与动量守恒。五岁孩子都能想象：黑漆漆的虚空里过一小时，什么都不变——那就是能量守恒的来源。」

3

Mario

「你如今身处经济最剧变之处——AI、代码生成。先问什么守恒，是否也体现在你们战略里？」

4

Matan Grinberg

「非常重要。很多人喊『所有工作都会没』『永久底层阶级』——这不准确，也伤害那些在未来仍至关重要的人。你其实可以用守恒和对称性来拆解这些说法。」

二、问题只会越来越多：反驳「工作消失论」

核心洞察

未解问题远多于已解，软件杠杆上升意味着多解决问题，而非净消灭工作。

深度解析

Matan Grinberg 的守恒式论证分三步：

1. 世界问题很多，大量尚未解决。 2. 其中很大比例本可用软件解决，但目前没有。 3. 问题总量随技术与生活质量提升而增长——人类「升级」所面对的问题类型（如心理健康在狩猎采集时代几乎不是议题）。

Mario 追问：生活质量提高是否意味着问题在减少？Matan Grinberg 认为人类是在问题阶梯上攀升——今日心理健康是重要议题，采集时代只需找野猪。Mario 承认缺乏可靠统计数据；Matan Grinberg 则强调：无论总量增删，未被解决的问题远多于正在被解决的。

Factory 给每位工程师更高杠杆，使其能攻克更多问题；某一具体问题所需人数可能下降（曾经一千人，现在一百人），但全局存在海量其他可解而未解之题。DMV（美国车辆管理局）软件糟透并非必然——资金与人才错配造成，未来可以更好。

Mario 归纳：Matan Grinberg 认为前沿模型与开放权重模型带来裁员与杠杆，但问题之丰沛使人类仍极其重要。Matan Grinberg 确认，并补充自由市场动态：竞争者若不裁员、反而给更好体验，可能抢走「抢先裁员」者的份额。垂直行业差异大——有的软件工程非核心能力，人数会减少；有的行业每位工程师与工具都增加杠杆，不应停止招聘。

Mario 问是否依赖「锯齿状智能」与算力昂贵使人力投资回报率仍佳。Matan Grinberg 同意，并衔接到资源配置问题（见下节）。

对谈实录

1

Matan Grinberg

「先承认：世界问题很多，很多没人在解。其中很大比例本可用软件解决，但没有。问题数量总体上还会增长——生活质量越高，人们期待越高，想解决的事越多。」

2

Mario

「所以你的观点是：会有大量岗位置换，但问题如此之多，人类仍极其重要？」

3

Matan Grinberg

「对。还会涌现新动态——自由市场有竞争。你裁员，对手不裁、体验更好，你可能丢份额。有的垂直行业软件不是核心能力，工程师会减少；有的行业每位工程师都是杠杆，你可能根本不想停招。」

三、代币、算力与人头：CEO 的新资源配置题

核心洞察

每增加一美元，CEO/CTO/CIO 要在人头、用户洞察与代币预算之间做精算——十二个月后绝不会再「每人每月一千美元代币」式盲配。

深度解析

Matan Grinberg 认为每家公司都将面临资源配置问题：增量美元投向更多员工？更多市场/用户数据？还是更多算力——此处算力即人均代币预算。

当前处于有趣过渡期：AI 工具用量激增，企业不知投资回报率（ROI），于是预防性设限——例如每人每月 1000 美元代币。Matan Grinberg 断言：十二个月内，组织绝不会再如此盲目均分代币；将出现「这一枚代币投在哪才能推动业务」的精算——销售还是市场？工程内部是平台、客服还是账单？

现状资源配置「蠢到极点」：该多招人还是多给代币？「不知道，先这样吧」——毫无数据支撑。Matan Grinberg 预判即将进入能用清晰指标说话的阶段：某部门应多投代币，另一部门代币边际效应低、应多招人——销售要面对面、建人脉，代币可能帮不大。

Mario 表示此前未细想归因与分配如何成为可能——这与 Factory 所做的事直接相关。

对谈实录

1

Mario

「这版未来是否建立在『人类在某些事上仍更强、模型仍算力受限』之上？」

2

Matan Grinberg

「是。每家公司都会有资源配置问题——CIO、CTO、CEO 对每个增量美元：多招人？多买市场/用户理解？还是多算力——也就是人均代币预算。」

四、Factory 使命：建造软件工厂而非手写软件

核心洞察

软件工程的未来：你不造产品，你造造产品的工厂——工程师调校产线，而非亲手拧每一颗螺丝。

深度解析

Factory 对外描述随阶段演变，使命未变：自三年前 Mario 在 newsletter 首次提及以来，始终是为软件工程带来自主性（autonomy）。

最初两年聚焦 Droids——软件研发智能体，不止编码，而是端到端软件开发生命周期；它们是「智能体原生软件开发」世界的基本单元。随采用普及——有组织数万用户——对话重心从「单个 Droid 增量价值」转向客户所建的软件工厂。

视觉类比：Tesla 工厂里机械臂自动装配；未来软件工程是工程师说「这条臂要这样调、那条臂那样调，去装这个/widget 那个/widget」——变成系统设计问题。且不锁定单一模型，产线上同样有资源配置：不能 naive 地凡事用 Opus——费、伤利润率；某臂用开放模型，某臂 Gemini 擅某种语言，某臂为 COBOL 遗留代码微调。

Matan Grinberg 痛陈现状：最聪明工程师花数年成专家，却小时级修持续集成（CI）、写文档、做产品需求文档（PRD）、开二十人会议——不是智力的好用途。Factory 要自动化低杠杆时间，让深度洞察时刻更集中——而工程师本就喜欢做后者。

对谈实录

1

Mario

「Factory 怎么描述？」

2

Matan Grinberg

「使命从第一天没变：为软件工程带来自主性。最初是 Droids——覆盖全生命周期，是智能体原生开发的基本单元。现在企业更谈他们建的软件工厂——像 Tesla 工厂视频里那些机械臂。」

五、产品扩张：瓶颈驱动的 Droid 全生命周期

核心洞察

工厂类比指导产品：Droid Computer、Factory Router、Automated Security 皆由客户产线瓶颈催生。

深度解析

Mario 作为用户提到 Droid Computer、Factory Router、Automated Security 等快速迭代功能。Matan Grinberg 用工厂类比：找出组装线上最大瓶颈，才能更快产出单位产品。

一例：受监管客户要求代码审查不止「代码好不好」，还要是否符合法规/安全标准——催生合规审查能力。Droid Computer：成熟组织大量任务委派给智能体后，本地笔记本承载不了；合上电脑智能体就停——需远程机器/沙箱，生成代码、测试、验证合规，甚至花一周修 messy CI。

Matan Grinberg 自称一直在「droiding」，认为产品节奏 exciting。

对谈实录

1

Mario

「我一直在 droiding。还有 Droid Computer、Factory Router、Automated Security——你怎么决定扩 scope？」

2

Matan Grinberg

「工厂类比很有用：找最大瓶颈。有客户说：很棒，但我们受监管——能否每次 code review 都测合规，不只代码质量？Droid Computer：人人用智能体后，你笔记本扛不住；合上电脑 agent 就完。要能 spin up 远程机，在沙箱里生成、测试、验证，甚至花一周修烂 CI。」

六、模型无关：像用电一样隐藏模型选择

核心洞察

若 Factory 成功，客户对所用模型的感知应像对烤面包机背后电网一样——无感。

深度解析

Mario 问终局是每人用几十种模型，还是前十强分工。Matan Grinberg 答：成功的话，工程师对模型应像对 toaster 背后电力来源——只想面包烤好。

模型越多越好，但工程师跟踪 Opus 4.8 vs GPT-5.5 Ultra High vs Kimi 2.5 在各语言各任务上的表现——不是 Fortune 500 工程师该花的时间（对 Factory 这类公司是核心能力）。还要在 Python、Java、COBOL 间切换——认知 overhead 过大。

Factory 替客户处理路由，按成本、性能、延迟等优化目标配置；工程师专注业务杠杆，而非刷 Twitter 追每个新模型。Factory Router 即此战略的产品化。

对谈实录

1

Mario

「终局是每人用几十种模型，还是前十强就够？」

2

Matan Grinberg

「若我们成功，客户对模型的了解应像对烤面包机背后的电——你只想把面包烤好。模型多很好，但工程师不该追踪每个 Opus 4.8、GPT-5.5、Kimi 2.5 谁擅什么——你在 Fortune 500 做工程，这不是该花的时间。我们替你路由：成本、性能、延迟，你专注业务。」

七、赢得企业客户：因地而异的切入楔子

核心洞察

没有单一万能楔子——有时是抗厂商锁定，有时是三年未完成的迁移，有时是工程师自下而上逼宫。

深度解析

Mario 问开门砖是否为困难迁移。Matan Grinberg 希望如此简单，实则因组织瓶颈而异：

- 厂商锁定焦虑：只绑 OpenAI/Anthropic/Google 之一，需要模型无关的 Factory。 - 迁移僵局：「这个迁移做了三年，求你们帮忙。」 - 自下而上：开发者在 Twitter 发现 Factory，向 CTO 举义旗要求采购。

通常是该组织软件生命周期中不同环节的瓶颈成为切入点。

对谈实录

1

Mario

「你怎么赢客户？困难迁移？」

2

Matan Grinberg

「我希望每次都是一个原因，但其实是各组织的瓶颈不同。有的怕厂商锁定，只要 OpenAI 或 Anthropic——看到我们模型无关就说太好了。有的迁移做了三年：求你们。还有开发者 Twitter 上发现，向 CTO 逼宫。因案例而异。」

八、xAI-Cursor 交易与「一体优化」之辩

核心洞察

xAI 收购 Cursor 对各方有利；但「模型+执行层一体必然更强」与 Factory 实证相反。

深度解析

Mario 提 xAI 拟收购 Cursor 后 Matan Grinberg 在 Twitter 的辩论：一方认为模型公司终应买下执行层/环境层以联合优化；Matan Grinberg 称无数据支持。

Matan Grinberg 分两层：

模型商与编码的关系：Cursor/SpaceX/xAI 交易对 SpaceX/xAI、Cursor 团队与 Factory 都好。xAI 有数据中心、基建、工程师，缺 coding 模型的分发与数据；Cursor 有分发与数据，缺数据中心与现金——互补。Factory 受益：市场需要不绑单一模型的独立软件工厂提供商；此前 Cursor 是主要同类玩家，交易某种程度上为其让路。未来 OpenAI、Anthropic、Google、xAI 都会有强 coding 模型——选择越多越好。

「一体更优」主张：如 Claude Code 配 Claude、Codex 配 Codex 模型。Matan Grinberg 的实证发现相反：Droids 模型无关，反而在 Opus、Codex 等上胜过各自官方执行层——例如用 Opus 的 Droid 胜过 Claude Code 里的 Opus。

类比五年前个人助理：争论应只训你的数据 vs 训全网——后者对你更好。数据之于模型，如同模型之于执行层——向执行层暴露越多模型，越能捕捉细微差别，避免过拟合单一模型怪癖。关键维度包括：上下文压缩（compaction）（上下文超限如何处理，大代码库常见）、代币缓存（token caching）（缓存代币可降至约十分之一成本）、工具调用与环境反馈。模型公司朋友对此很抓狂。Matan Grinberg 称有缓存、压缩相关博文，但缺「为何需要多模型才更强」的完整论述——值得发研究。

对谈实录

1

Mario

「有人说模型公司该买 Cursor 这类执行层公司才能联合优化。你的反对理由？」

2

Matan Grinberg

「Cursor/xAI 交易对他们是好事，对我们也是好事——市场需要不绑单一模型的独立软件工厂；Cursor 曾是主要同类，现在让路。OpenAI、Anthropic、Google、xAI 都会有强 coding 模型，选择越多越好。」

3

Mario

「为什么？直觉上说不通。」
